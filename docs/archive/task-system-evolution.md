# 任务系统演进说明

## 1. 文档目标

本文档只记录一件事：

- 当前 TweetPilot 任务系统在实现上主要支持 Python 脚本
- 但从产品设计上，不应把任务系统永久定义为“只支持 Python”

编写此文档的目的，是在后续继续迭代任务系统时，明确记录当前限制与未来方向，避免后续代码和文档继续把任务系统固化为 Python-only。

---

## 2. 当前状态

### 2.1 当前已稳定支持的执行方式

当前任务系统已经稳定支持的执行方式是：

- Python 脚本执行

现有执行器、调度链路、执行历史和超时控制，也主要围绕 Python 脚本展开。

因此，在“当前实现”层面，把任务系统描述为“Python 脚本任务系统”是成立的。

### 2.2 当前不应过度延伸的结论

虽然当前只支持 Python，但这并不意味着：

- 任务系统的长期形态只能是 Python
- LocalBridge 的调用方式只能通过 Python 实现
- 未来任务类型必须继续围绕 Python 脚本设计

这些都只是**当前实现边界**，不是长期产品边界。

---

## 3. 为什么不能把任务系统永久绑定到 Python

### 3.1 LocalBridge 本身是 REST API 能力源

根据最新产品边界，TweetPilot 的底层能力源是 LocalBridge，而不是 Python 本身。

这意味着从理论上讲，只要某种执行方式能够访问 REST API，就可以成为未来的任务执行方式。

例如：

- shell + curl
- JavaScript / TypeScript
- Go
- Rust
- 通用 HTTP 请求任务

因此，把任务系统永久设计成“只会执行 Python 脚本”，会限制后续演进空间。

### 3.2 Python 只是当前最成熟的一条路径

当前之所以只支持 Python，原因更偏工程现实：

- 已经有现成执行器
- 已经有 ClawBot 代码库
- 当前用户场景与 AI 生成脚本路径大多围绕 Python

这说明：

- **Python 是当前最成熟的任务执行方式**
- 但不是未来唯一合理的任务执行方式

---

## 4. 未来应如何重新定义任务系统

后续任务系统更准确的长期方向应是：

- 从“Python 脚本执行器”演进为“通用任务执行框架”

### 4.1 推荐的长期任务类型视角

未来的任务类型不应只按语言区分，而应按执行模型区分，例如：

- Python script
- Shell script
- HTTP request
- AI workflow
- 其他可扩展执行器

这样比“任务系统 = Python 脚本系统”更合理，也更符合 LocalBridge 作为 REST API 能力源的定位。

### 4.2 当前不需要立刻实现

本文档不是要求现在马上实现多语言执行器。

当前阶段只需要明确记录：

1. 当前支持 Python
2. 未来不应固化为 Python-only
3. 后续在重构任务系统时，应保留扩展到其他执行方式的空间

---

## 5. 当前任务系统文档需要如何表述

后续所有与任务系统相关的文档，都建议采用以下表述方式：

### 推荐表述

- 当前任务系统**已实现的执行方式**主要是 Python
- 当前任务系统**现阶段以 Python 为主**
- 当前执行器**目前只覆盖 Python 脚本**

### 不推荐表述

- 任务系统就是 Python 脚本系统
- 任务永远只支持 Python
- 所有自动化能力必须以 Python 编写

这样可以避免把当前实现误写成产品永久边界。

---

## 6. 对后续工作的提醒

为了防止未来继续遗忘，后续任务系统演进时建议优先记住以下几点：

1. 任务系统的长期目标应是通用执行框架
2. Python 是当前主路径，但不是唯一未来形态
3. LocalBridge 相关任务未来天然可以扩展到更多 REST API 调用方式
4. 任务类型设计、执行器接口、配置模型都应尽量保留扩展空间

---

## 7. 一句话总结

> 当前任务系统只稳定支持 Python，但这只是当前实现边界，不应成为 TweetPilot 任务系统的长期产品边界。
