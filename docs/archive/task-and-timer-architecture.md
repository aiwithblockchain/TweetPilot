# 任务与定时系统架构（当前状态）

> 本文档整合自原有定时器系统与任务模块设计文档。
> 内容已按当前代码实现重新校对，仅保留对现状仍有指导意义的架构说明。
> 对于仅有字段、类型或底层占位而未形成完整产品能力的部分，文中统一标注为“伪实现”。

---

## 1. 文档目标与范围

本文档描述 TweetPilot 当前任务与定时系统的真实架构边界，重点说明：

- 当前已经稳定存在的架构基线
- 尚未实现的能力
- 已有底层占位但未真正落地的伪实现能力
- 已实现但文档表述需要降级的部分

本文档不再保留历史迁移过程、阶段性实施记录或未经代码验证的性能承诺。

---

## 2. 当前架构基线

当前代码库中，任务与定时系统已经具备以下基础能力：

### 2.1 统一定时框架

系统已采用统一定时框架管理系统任务与用户任务，核心组件包括：

- `UnifiedTimerManager`
- `TimerRegistry`
- `EventLoop`

该框架已经替代旧式分裂式设计，成为当前任务调度的主干。

### 2.2 已落地的调度类型

当前已稳定接入的调度类型为：

- `interval`
- `cron`

系统级 LocalBridge 同步任务与用户自定义定时任务都已接入统一调度链路。

### 2.3 已落地的任务生命周期能力

当前任务模块已支持：

- 任务创建、更新、删除
- 任务暂停、恢复
- 任务立即执行
- 执行历史查询
- 启动时重算下次执行时间
- 定时系统状态查询

### 2.4 已落地的脚本执行能力

当前脚本执行链路已支持：

- 通过 `python3` 执行脚本
- 手动执行与定时执行统一按数据库中的 `script_path` 解析并启动脚本
- 不再向脚本注入 `account_id` 或任务参数
- 捕获 `stdout` / `stderr` / `exit code`
- 基于 timeout 的超时控制
- 执行结果落库与基础统计
- 执行前校验脚本是否存在且为文件

补充说明：本轮关于任务脚本路径修复、编辑能力补齐与参数边界收敛的交付结果，已归档于 `docs/archive/task-execution-path-args-implementation-record.md`。

## 3. 未完成的架构能力

### 3.1 OneTime 一次性定时器

**状态**：未实现

虽然 `TimerType::OneTime` 已在底层类型中定义，且 `calculate_next_execution()` 包含对应分支，但当前系统中仍缺少：

- OneTime 任务的持久化模型
- 创建 / 更新入口
- UI 配置入口
- 列表、恢复、编辑等业务分支支持

因此该能力目前仍不能视为已实现。

### 3.2 依赖型定时器（Dependency）

**状态**：伪实现

`TimerType::Dependency` 已存在于底层类型定义中，但当前没有形成完整能力，缺少：

- 依赖关系存储模型
- 依赖链触发逻辑
- 失败传播与延迟执行机制
- 前端配置入口

同时，当前 `calculate_next_execution()` 对 Dependency 直接返回 `None`，说明它仍属于类型级占位，而非可用能力。

### 3.3 并发执行控制

**状态**：未实现

原设计中提出的并发执行池、最大同时执行数与资源隔离能力，当前代码中尚未落地。未见：

- `ExecutorPool`
- `Semaphore`
- `max_concurrent` 配置
- 分类并发策略

当前事件循环仍以逐个出队执行为主。

### 3.4 可配置的外部定时系统配置

**状态**：未实现

原设计中的统一配置文件方案尚未落地。当前系统参数仍主要硬编码在代码中，例如：

- LocalBridge 同步间隔：60 秒
- 任务默认 timeout：300 秒

尚未发现统一的 `timers.toml` 或等价配置加载机制。

## 4. 伪实现能力

以下能力在当前代码库中已经出现字段、类型或底层排序支持，但尚未形成完整产品行为，因此统一定义为“伪实现”。

### 4.1 自动重试机制

**状态**：伪实现

数据库与任务输入模型中已经存在：

- `retry_count`
- `retry_delay`

但当前执行失败后，并不会按这两个字段驱动真实重试流程。现有行为只是回到下一次正常调度时间，而不是进入独立的 retry 生命周期。

因此，自动重试目前仍停留在“配置已存在、行为未实现”的阶段。

### 4.2 脚本内容快照与 Hash 变更检测

**状态**：伪实现

任务模型与数据库表中已经保留：

- `script_content`
- `script_hash`

但当前没有看到：

- 创建或更新任务时保存脚本快照
- 生成并校验 hash
- 执行前检测脚本漂移
- 在界面中提示脚本变更

因此，该能力目前仍只是存储层预留。

### 4.3 执行结果 metadata

**状态**：伪实现

`executions` 表与 Rust 结构体都包含 `metadata` 字段，但当前实际执行链路中基本写入的是 `metadata: None`，也未看到：

- 从脚本输出中解析结构化 metadata
- 将 metadata 传递给前端详情页

因此该能力尚未形成有效结果通路。

### 4.4 优先级调度

**状态**：伪实现 / 部分实现

底层已有以下基础：

- `Timer.priority`
- `BinaryHeap` 中纳入 priority 排序
- 系统任务与用户任务在注册时设置了不同 priority

但当前 priority 只体现为队列排序时的次级比较条件，尚未形成完整的优先级调度体系。缺少：

- 用户可配置优先级
- 并发场景下的优先级资源分配
- 抢占或限流策略

因此，如果文档把它表述为“优先级调度已实现”，会高估当前能力。

## 5. 已实现但需要谨慎表述的部分

### 5.1 事件驱动精确定时

**状态**：部分实现

当前系统已经具备统一事件循环和优先队列，不再是旧式固定双轮询模型；但它仍使用分段 `sleep(...)`，且睡眠时长被限制在 60 秒内，尚未达到原设计中“基于 `sleep_until` 的精确定时”目标。

因此，更准确的表述应为：

- 已实现统一事件循环
- 尚未实现最终形态的精确定时机制

### 5.2 账号在线检查

**状态**：能力边界不完整

旧文档曾将“执行前检查账号在线状态”描述为任务模块的既有能力，但当前代码中尚未看到一套统一、强约束的前置在线校验闭环。

当前更准确的描述应是：

- 任务执行链路会传入 `account_id`
- 账号在线态能力存在于 LocalBridge / 账号管理侧
- 但任务执行前的统一在线阻断校验尚未形成明确闭环

### 5.3 任务统计

**状态**：已实现但存在一致性风险

基础统计字段已经存在，并且执行历史会落库，但当前实现中存在一个值得注意的一致性问题：

- `task_database.save_execution()` 中失败统计判断使用的是 `result.status == "failure"`
- Python 执行器实际写入的失败状态是 `"failed"`

这意味着失败次数统计可能存在偏差。该问题不影响统计框架存在，但会影响其准确性。

## 6. 后续演进建议

### 6.1 优先级建议

**P1：补齐当前伪实现但已对外暴露字段的能力**

- 自动重试机制
- 脚本快照 / Hash 检测
- 执行 metadata

**P2：补齐调度能力类型**

- OneTime 一次性任务
- Dependency 依赖任务
- 定时器配置外置化

**P3：优化调度精度与资源控制**

- 基于 `sleep_until` 的精确定时
- 并发执行控制
- 完整优先级调度体系

### 6.2 文档维护原则

后续该文档应持续遵循以下原则：

- 只描述经代码核对后的当前状态
- 对“字段已存在但行为未落地”的能力统一标注为“伪实现”
- 不保留历史迁移过程与阶段性勾选记录
- 不写未经验证的性能对比承诺

### 6.3 长期产品边界说明

当前任务系统已稳定支持的执行方式主要是 Python 脚本，现有执行器、调度链路、执行历史与超时控制也主要围绕 Python 展开。

但这一定义只能用于描述**当前实现状态**，不应被外推为长期产品边界。

更准确的长期边界应为：

- TweetPilot 任务系统当前以 Python 为主
- Python 是当前最成熟的执行路径，但不是唯一未来形态
- LocalBridge 是底层能力源，因此未来任务执行方式不应永久绑定到 Python
- 后续任务系统应从“Python 脚本执行器”逐步保留向“通用任务执行框架”演进的空间

从长期设计视角看，未来任务类型更适合按执行模型区分，而不是按语言永久固化。例如：

- Python script
- Shell script
- HTTP request
- AI workflow
- 其他可扩展执行器

因此，后续相关文档建议采用以下表述方式：

- 当前任务系统**已实现的执行方式**主要是 Python
- 当前任务系统**现阶段以 Python 为主**
- 当前执行器**目前只覆盖 Python 脚本**

应避免使用以下表述：

- 任务系统就是 Python 脚本系统
- 任务永远只支持 Python
- 所有自动化能力必须以 Python 编写

---

## 7. 结论

TweetPilot 的任务与定时系统主干架构已经落地，统一调度框架、任务 CRUD、Interval/Cron 调度、脚本执行与执行历史记录均已可用。

当前真正需要继续推进的，不再是“是否有任务系统”，而是以下三类剩余工作：

1. 补齐尚未落地的调度类型
2. 将已有字段和类型占位补成真正可用的产品能力
3. 收紧文档表述，避免把底层预埋能力误写成完整实现

本文档即作为当前阶段的正式架构描述基线。
