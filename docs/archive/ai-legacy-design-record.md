# AI 旧设计记录归档

## 1. 文档目的

本文档用于归档以下已不再适合作为当前主设计依据的历史文档：

- `docs/ai-architecture-roadmap.md`
- `docs/ai-session-constraints-implementation.md`
- `docs/ai-thinking-history-implementation-plan.md`

归档原因主要有三类：

1. 其中部分工作已经完成
2. 其中部分设计已被后续实现和新产品边界覆盖
3. 这些文档继续保留在 `docs/` 根目录，会造成“它们仍是当前主方案”的误导

本文档不是新的实施方案，而是保留必要的历史背景，帮助后续追溯为什么曾经这样设计、哪些内容已经落地、哪些内容已经过时。

---

## 2. 被归档文档的历史价值

### 2.1 `ai-architecture-roadmap.md`

这份文档的历史价值主要在于：

- 它整理过交互式 AI 会话系统与自主 AI 工作流系统的总体视角
- 它明确过会话元数据、账号级自治会话、人设与知识库、安全层等方向性的概念
- 它对“哪些能力尚未完成”做过较系统的罗列

但它不再适合作为当前主文档的原因也很明确：

- 它主要围绕旧一轮 AI 架构抽象展开
- 与当前围绕 LocalBridge、开发方式边界、Python / 非 Python 分流的产品方向不完全一致
- 其中不少“未完成项”已经脱离当前最优先的问题域

因此，适合保留其历史记录价值，不再保留其主设计地位。

---

### 2.2 `ai-session-constraints-implementation.md`

这份文档的历史价值主要在于：

- 它明确推动了 `skill.md` 与 ClawBot 资源打包、安装到 `~/.tweetpilot/` 的方案
- 它明确推动了 AI session 创建时注入 `skill.md` 约束文档
- 它帮助当时梳理了工作区约束文件、会话提示词注入、资源安装职责边界等问题

这份文档不再适合作为当前主文档的原因主要有两点：

1. 其中一大部分工作已经完成，例如：
   - `resources/tweetpilot-home/skill.md` 已存在
   - `resources/tweetpilot-home/clawbot/` 已存在
   - 应用已具备将资源复制到 `~/.tweetpilot/` 的逻辑
   - AI session 已改为主要注入 `skill.md`

2. 其中另一部分设计已被新的产品边界改写，例如：
   - 它仍把体系中心过多放在 Python / ClawBot / 特定文档拼装上
   - 它曾围绕旧版约束文件命名和旧阶段 prompt 组织方式展开
   - 当前新的方向已经明确为：LocalBridge 是能力源，最新 REST API 文档统一通过 `GET /api/v1/x/docs` 获取

因此，这份文档的核心历史作用已经完成，不应继续作为 active doc。

---

### 2.3 `ai-thinking-history-implementation-plan.md`

这份文档的历史价值主要在于：

- 它聚焦了历史会话恢复、thinking 持久化、tool call 展示、实时 / 历史消息模型统一等问题
- 它比泛架构文档更接近一次具体的缺陷修复与体验恢复计划
- 它明确拆分了多阶段实施任务，便于当时执行

但它不再适合作为当前主文档的原因是：

- 它本质上是一个阶段性实施计划，而不是长期稳定的顶层设计文档
- 它服务的主要目标是当时那一轮会话恢复与 thinking 展示修复
- 继续摆在 `docs/` 根目录里，会误导为仍然是当前第一优先级主方案

因此更适合归档保存，而不是继续占据主文档位置。

---

## 3. 历史工作中已经沉淀下来的有效结论

虽然以上三份文档不再作为主方案，但其中沉淀下来的若干结论仍然有效。

### 3.1 `skill.md` + 资源安装思路是正确的

已经被实践验证的有效结论包括：

- `skill.md` 作为 AI 会话级约束文档是有价值的
- ClawBot 资源打包进应用资源目录是合理的
- 由应用启动或关键入口进行 `~/.tweetpilot/` 补装，比依赖安装脚本写 Home 目录更合理

这些结论仍应保留，但其内容和边界需要随着新方向继续更新。

### 3.2 “只记录当前有效边界”的文档策略是正确的

无论是旧架构文档还是旧实施计划，随着部分功能完成、部分方向调整，都会快速过时。

因此后续 active docs 应坚持：

- 只记录当前有效边界
- 不把大量历史已完成项继续堆在主文档里
- 历史内容进入 archive，而不是长期混在主设计文档中

### 3.3 任务系统与 AI 系统都不能被过早写死

旧文档中不论是关于 AI session、自治工作流，还是任务执行，都曾反复暴露出一个共同问题：

- 一旦文档过早把当前实现当成最终边界，后续就容易被旧定义束缚

这一点也直接促成了当前新文档需要明确：

- LocalBridge 是能力源
- Python 不是唯一方案
- 任务系统当前只支持 Python，但长期不应固化为 Python-only

---

## 4. 为什么现在要把旧文档移出 docs 根目录

继续把这些旧文档保留在 `docs/` 根目录，会带来三个明显问题：

1. **误导优先级**  
   读者会误以为这些仍是当前主方案，而不是历史记录。

2. **混淆实现边界**  
   文档里既有已完成内容，也有已废弃方向，还夹杂部分仍有参考价值的设计，容易让后续实现者误判。

3. **阻碍新方向落地**  
   当前产品边界已经切换到以 LocalBridge 能力源和开发方式分流为中心，如果旧文档继续占据主位置，会持续拉回旧理解。

因此，最合理的做法是：

- 生成新的 active docs
- 将旧文档内容合并归档
- 从 `docs/` 根目录移除旧 active 身份

---

## 5. 当前新的主文档接替关系

归档这些旧文档后，当前阶段更合适的 active docs 应由新的文档承担：

- `docs/localbridge-and-development-strategy.md`
  - 负责定义 LocalBridge、ClawBot、Python / 非 Python、最新 REST API 文档获取方式等总纲边界

- `docs/task-system-evolution.md`
  - 负责定义任务系统“当前只支持 Python，但长期不应固化为 Python-only”的演进边界

这两份文档更符合当前真实优先级，也更能支持后续继续优化 `skill.md` 与 `clawbot/README.md`。

## 6. 归档结论

本次归档的最终结论是：

1. 旧文档不是没有价值，而是**不再适合作为当前主方案**
2. 其中有效结论已部分落地，部分被新方向继承
3. 当前 active docs 应切换到新的边界文档
4. 后续若还需要回看旧背景，应统一查看本归档文档，而不是继续从旧 active docs 里找答案
