# 后台事件通知系统实施记录

## 状态

**已完成交付。**

本文件用于归档“后台事件通知系统”这一轮工作的背景、交付范围、实现结果、验证结论与当前边界，不再作为待实施方案使用。

---

## 背景

当时提出这份方案的直接目标，是为应用补齐一套稳定的后台事件通知能力，用于解决以下问题：

1. 后端状态变化后，前端缺少统一的通知入口
2. 业务代码容易各自散落实现刷新逻辑，缺少一致约定
3. 任务模块需要在创建、更新、执行、删除等关键节点通知前端完成自动刷新
4. 通知机制本身与任务业务语义曾混在一起，需要重新收敛边界

因此，这一轮工作的目标被拆成两层：

- 建立通用的后台事件通知基础设施
- 让任务模块作为首个接入方完成落地验证

现在回看，文档中的主要开发目标已经完成，这份文档更适合作为实施记录归档。

## 关键实现结论

### 1. 通知机制已经收敛为通用应用事件层

本轮最重要的结果，不是“某个页面能刷新”，而是把通知问题收敛为统一模型：

- 业务模块负责发布事件
- 监听方负责订阅事件
- 中间层负责按事件 ID 广播

这意味着后续新增接入方时，不需要再为某个页面单独设计通知协议。

### 2. 事件通道已按稳定业务语义命名

当前事件体系已经从“页面动作驱动”转向“业务事件驱动”。

也就是说，事件表达的是类似：

- 某个任务被创建
- 某个任务被更新
- 某个任务被执行
- 某个任务被删除

而不是：

- 某个面板去刷新
- 某个列表去重载

这样可以保证：

- 发布方不需要知道有哪些页面在监听
- 同一个事件可以被多个前端位置复用
- 后续扩展到其他模块时不需要推翻命名体系

### 3. 第一版底层实现已经落在现有事件系统上

当前实现并没有额外引入新的本地消息总线，而是基于项目现有能力完成落地。

这带来两个直接结果：

- 实现成本低，能够快速接入现有业务
- 上层业务语义已经独立，未来如果需要替换底层广播实现，改动面会更可控

也就是说，第一版重点已经不是“做一个复杂框架”，而是先把统一事件层真正落到代码里并被业务稳定使用。

### 4. 任务模块已完成首轮接入验证

任务模块作为第一批接入方，已经证明这套机制是可用的。

当前可确认的实现结论是：

- 任务相关状态变化能够通过统一事件层向前端发出通知
- 前端任务相关界面能够按需订阅这些事件
- 刷新逻辑保留在前端消费端，不反向污染事件层设计

因此，这份文档原本强调的“任务模块接入方案”，其核心价值已经转变为历史实施背景，而不是未来待做事项。

## 涉及范围

本轮工作覆盖的范围，主要可以概括为两部分。

### 后端侧

- 统一应用事件定义与发布入口
- 业务模块在关键节点发布事件
- 保持事件 ID 与 payload 约定一致

### 前端侧

- 在任务相关页面、组件或 hook 中监听目标事件
- 收到事件后触发必要的数据刷新或状态同步
- 保持订阅与取消订阅边界清晰，避免监听泄漏

## 当前边界

虽然该轮目标已经完成，但它的边界也应明确记录：

- 当前归档重点是“本地应用内事件通知”，不是跨进程或远程消息系统
- 当前主要验证了任务模块接入，其他模块是否继续复用可按需要逐步扩展
- 当前事件机制的职责仍是“通知”，而不是替代业务数据查询本身
- 前端在收到事件后，仍应根据页面需要决定是局部更新还是重新拉取数据

这些属于有意保持的边界，不代表遗漏。
