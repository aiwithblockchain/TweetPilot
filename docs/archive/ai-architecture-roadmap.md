## 2. AI 系统整体架构

TweetPilot 当前 AI 能力应统一理解为两条主线：

### 2.1 交互式 AI 会话系统

用于用户在应用内直接与 AI 对话。

**核心职责：**
- 在当前工作区上下文中与 AI 对话
- 维护会话历史
- 支持会话切换、新建、删除
- 支持工具调用（文件读写、命令执行、搜索）
- 提供流式消息与工具执行反馈

**核心模块：**
- 前端聊天界面：`src/components/ChatInterface.tsx`
- 会话 UI：`src/components/ChatInterface/SessionPanel.tsx`、`SessionList.tsx`、`SessionListItem.tsx`
- 前端调用层：`src/services/ai/tauri.ts`
- Tauri 命令层：`src-tauri/src/commands/ai.rs`
- 会话封装：`src-tauri/src/claurst_session.rs`
- 会话存储：`src-tauri/src/services/conversation_storage.rs`

### 2.2 自主 AI 工作流系统

用于定时任务或自动化任务触发 AI，自主完成多步操作。

**目标形态：**
- 由定时器或任务系统触发
- AI 根据自然语言任务描述自主决定步骤
- AI 自动调用 Bash / Read / Write / Grep / Glob 等工具
- 支持按账号绑定独立 AI 上下文
- 支持结合账号人设、知识库、内容策略执行自动发文/分析任务

**目标执行链路：**

```text
UnifiedTimerManager / Task Scheduler
  → AI Workflow Executor
  → Account-scoped AI Session
  → Tools (Bash / Read / Write / Grep / Glob)
  → 外部脚本、知识库、内容生成、发送动作
```

## 3. 会话体系架构

### 3.1 会话类型划分

建议统一分为两类：

#### A. 用户交互会话
- 用于 ChatInterface
- 生命周期由前端显式控制
- 强调可见性、可切换性、可管理性

#### B. 账号级自治会话
- 用于自动化 AI 工作流
- Session 与账号绑定
- 强调上下文连续性、人设一致性、内容避免重复

建议命名规范：

```text
session-<uuid>           // 用户交互会话
account-<account_id>     // 账号级自治会话
```

### 3.2 会话存储原则

统一基于本地持久化存储，但应进一步从“仅消息日志”演进为“消息 + 元数据”。

**消息层：**
- 保存 user / assistant 消息
- 保存时间戳

**元数据层：**
- 会话标题
- 所属工作区
- 会话类型（用户 / 账号）
- 关联账号 ID（若为账号级会话）
- 创建时间 / 更新时间
- 消息数
- 可选标签、状态、摘要

### 3.3 会话恢复原则

任何会话恢复都不应依赖“当前已经存在一个活动 session 才能恢复”。

恢复会话时，系统应能仅凭持久化信息完成：
- 找回工作区
- 找回历史消息
- 找回会话关联信息
- 重新构建运行时 AI Session

## 4. 自主工作流架构

### 4.1 核心设计

AI 自主工作流应建立在现有任务系统和定时器系统之上，但保持独立执行器。

**建议核心组件：**

#### AiAutonomousExecutor
负责：
- 接收任务调度触发
- 构建完整执行上下文
- 获取或创建账号级 AI Session
- 注入人设、知识库、内容策略
- 启动一次完整的 AI 自主执行
- 收集结果、日志、工具调用记录

#### Account-scoped Session Manager
负责：
- 管理 `account-{account_id}` 级别的 AI Session
- 在多次任务执行之间复用上下文
- 控制 session 生命周期与清理策略

#### Workflow Context Builder
负责：
- 装配任务 prompt
- 装配账号人设
- 装配知识库路径
- 装配全局策略与安全规则

#### Content Safety Layer
负责：
- 在生成内容前后执行安全校验
- 敏感词检测
- 内容策略校验
- 高风险任务的人工审核入口

### 4.2 推荐执行流程

```text
任务触发
  → 读取任务配置
  → 解析关联账号 / 工作区 / 知识库
  → 获取账号级 Session
  → 构建系统提示词与执行提示词
  → AI 自主调用工具完成任务
  → 内容安全检查
  → 执行动作（如发送推文）
  → 记录执行结果与历史
```

## 5. 知识库与人设架构

### 5.1 知识源分层

建议保留以下分层模型：

```text
knowledge/
├── global/
│   ├── sensitive-words.md
│   └── content-policy.md
├── personas/
│   └── account-<id>.md
└── products/
    └── <product-name>/...
```

### 5.2 加载优先级

加载顺序建议固定为：

1. 全局内容规则
2. 全局敏感词规则
3. 账号人设
4. 任务级知识库
5. 用户输入或任务 prompt

### 5.3 人设模型

账号的人设信息应允许来自两个来源：
- 数据库字段
- 文件化知识库（`knowledge/personas/account-<id>.md`）

当两者并存时，应定义明确优先级，避免运行时行为不一致。

## 6. 安全与内容控制架构

### 6.1 多层内容安全模型

对于自动发文类任务，至少应具备两层保护：

#### 第一层：Prompt 级约束
- 将禁止内容、敏感词、平台规则注入系统提示词
- 让模型在生成阶段主动规避

#### 第二层：发送前规则校验
- 在真正发送前进行本地校验
- 检查敏感词、变体、PII、黑名单规则
- 不通过则拦截发送

可选第三层：
- 高风险任务启用人工审核

### 6.2 输入安全

需要对以下输入做严格校验：
- `session_id`
- 工作区路径
- 知识库路径
- 任务执行参数
- AI 工作流配置 JSON

### 6.3 工具边界

AI 虽然具备 Bash / Read / Write 等工具能力，但自动化工作流必须引入更明确的边界：
- 限定工作目录
- 限定可执行脚本范围
- 限定允许写入的位置
- 限定发送类动作的触发条件

## 7. 数据模型方向

### 7.1 会话元数据模型

后续需要将会话从“纯 JSONL 消息文件”扩展为可管理的数据对象，至少包含：

```ts
interface SessionMetadata {
  id: string
  type: 'user_chat' | 'account_workflow'
  title: string
  workspace: string
  accountId?: number
  createdAt: number
  updatedAt: number
  messageCount: number
}
```

### 7.2 AI 工作流配置模型

建议统一为结构化配置，而不是散落在不同任务字段中：

```ts
interface AiWorkflowConfig {
  accountId: number
  initialPrompt: string
  workingDir: string
  knowledgeBase?: string
  timeoutSeconds?: number
  maxRetries?: number
  requireHumanReview?: boolean
}
```

### 7.3 执行结果模型

自主工作流执行后应沉淀标准结果结构：
- success / failure
- final output
- tool call trace
- token usage
- duration
- blocked reason（如安全拦截）
