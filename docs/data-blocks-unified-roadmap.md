## 三、总体方向

数据积木后续分成两条能力线推进：

### 3.1 内置数据积木补齐
继续完善当前已经存在的官方数据卡能力。

### 3.2 自定义数据积木建设
建设一种受控开放机制，让用户可以通过：

- Python
- 本地数据库
- 自定义模板

接入自己的数据卡。

最终目标不是继续零散增加固定卡片，而是把数据积木从固定模块逐步升级为可扩展的数据卡平台。

## 四、自定义数据积木核心结论

### 4.1 是否可行

结论：**可行。**

可行链路为：

- 用户写 Python
- Python 创建本地数据库表并写入数据
- Python 注册一张自定义数据卡定义
- 前端读取这张卡定义
- 前端读取数据库数据
- 前端按模板渲染成数据卡

### 4.2 采用受控开放，不做无限制执行

不能直接做成：

- 前端直接执行任意 HTML
- 后端直接执行任意 SQL
- 页面打开时直接运行任意 Python

推荐方式是：

- Python 负责产出数据
- Python 负责注册卡定义
- 系统负责受控读取
- 前端负责受控模板渲染

### 4.3 自定义数据积木的推荐边界

#### Python 负责
- 创建或维护业务表
- 写入业务数据
- 注册卡定义

#### 系统负责
- 读取卡定义
- 读取数据
- 安全校验
- 模板渲染
- 统一外壳和错误态

#### 用户模板允许
- 受控 HTML 模板
- 白名单标签
- 变量插值，如 `{{data.xxx}}`

#### 用户模板不允许
- 任意 JavaScript
- 外部资源
- 事件绑定
- 破坏宿主页面结构

## 六、统一优先级

以下是当前统一后的任务优先级。

---

## P0：最高优先级

### 1. 自定义数据积木 MVP

目标：先跑通最小闭环。

> Python 写数据 → Python 注册卡定义 → 前端看到卡 → 前端读取数据 → 前端受控模板渲染

#### 要做的内容

- 新增自定义卡定义表
- 明确卡定义结构
- 新增读取自定义卡定义列表接口
- 新增读取自定义卡数据接口
- 前端支持展示自定义卡目录
- 前端支持新增自定义卡实例
- 新增 `CustomDataCard` 渲染器
- 支持受控 HTML 模板渲染
- 做一张完整示例卡验证链路

#### 暂时不做

- 模板编辑器
- 复杂模板语法
- 图表脚本模板
- 在线执行 Python
- 模板市场
- 高级配置器

#### 为什么是最高优先级

因为这是平台化的起点。只要这条链路打通，后续很多积木能力都可以通过统一机制接入，而不是继续写死在前后端里。

## 八、MVP 成功标准

自定义数据积木 MVP 完成标准如下。

### 必须做到

- 用户可通过 Python 向本地数据库写一张业务表
- 用户可通过 Python 注册一张自定义数据卡定义
- 前端“添加积木”弹窗中能看到这张卡
- 用户可把这张卡添加到页面
- 页面能读取数据库数据并渲染卡片
- 模板渲染是受控的，不允许任意脚本执行
- 出错时有统一错误态

### 暂时不要求做到

- 在线执行 Python
- 自定义复杂图表
- 模板可视化编辑
- 完整模板语言
- 模板共享机制

## 十一、最终结论

数据积木模块后续统一按下面这条路线推进：

- 一边补齐现有内置积木能力
- 一边优先建设自定义数据积木 MVP
- 再把卡片实例和定义逐步统一到数据库
- 最终把数据积木从固定模块升级为可扩展的数据卡平台

当前最重要的事情不是继续零散加功能，而是先把 **自定义数据积木 MVP** 做出来，并把它放在全局最高优先级。
