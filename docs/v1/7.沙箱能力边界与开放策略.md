# 7.沙箱能力边界与开放策略

- 版本：`v0.1.0`
- 更新日期：`2026-04-09`
- 文档状态：`首版策略文档`
- 文档定位：本文档用于定义 `TweetPilot` 中沙箱的能力边界、开放策略与控制关系。本文档回答“沙箱应该开放什么能力、限制什么能力、禁止什么能力”，不讨论底层实现细节、字段设计、接口参数和开发排期。

## 1. 文档目的

本文档用于回答四个问题：

- 在 `TweetPilot` 中，沙箱到底承担什么职责
- 完整版沙箱应该开放哪些能力
- 哪些能力只能受限开放
- 哪些能力原则上不应向沙箱开放

这份文档的价值，不是为了把沙箱做成“更强大的执行环境”，而是为了确保平台在支持 Agent 生成扩展能力的同时，仍然保持长期稳定、边界清晰、权限可控和结果可审计。

## 2. 基本判断

`TweetPilot` 中的沙箱，不应被定义为：

- 一个无边界的本地代码执行平台
- 一个允许 Agent 直接改写平台核心的运行环境
- 一个默认拥有系统级权限的通用容器

`TweetPilot` 中的沙箱，应被定义为：

**一个用于承载 Agent 生成扩展能力的受控运行边界。**

它的目标不是“给 Agent 更多自由”，而是：

- 让 Agent 可以长出新能力
- 让新能力不污染平台核心
- 让新能力可以被挂载、启停、替换和接管
- 让平台对开放范围、触发方式和长期保留拥有最终控制权

## 3. 沙箱的职责边界

从产品角度看，沙箱只应承担三类职责：

### 3.1 承载扩展能力

沙箱负责承载 Agent 生成的能力单元，例如：

- 报表扩展
- 分析扩展
- 策略扩展
- 定时任务扩展
- 功能入口扩展

### 3.2 承载派生数据

沙箱可以拥有自己的派生数据空间，用于保存：

- 扩展能力生成的分析结果
- 扩展能力生成的临时或长期派生表
- 扩展能力自身的运行结果和刷新结果

这部分数据属于扩展能力的运行结果，不属于平台核心事实层和状态层。

### 3.3 承载受控展示

沙箱生成的能力，最终需要以受控方式进入平台展示层，例如：

- 菜单入口
- 报表页面
- 结果刷新入口
- 特定客户空间中的功能页

沙箱不负责决定这些能力是否长期出现在用户界面中，平台负责决定。

## 4. 完整版沙箱应开放的能力

这里说的“开放”，指的是平台在完整形态下可以正式授权给沙箱使用的能力范围。

### 4.1 只读访问平台事实层和状态层

沙箱应允许在授权范围内只读访问：

- 历史推文
- 历史互动
- 账号状态
- 任务状态
- 通道状态
- 内容表现快照
- 风险与主题聚类结果
- 角色与 Agent 表现数据

这是沙箱生成高价值分析和报表的基础。

### 4.2 读取客户知识层

沙箱应允许在授权范围内读取：

- 客户知识库
- 产品资料
- FAQ
- 品牌边界
- 宣传素材说明
- 指定 MCP 服务输出

这是保证扩展能力专业性和客户特异性的基础。

### 4.3 使用平台统一管理的模型能力

沙箱应允许通过平台统一网关使用：

- 本地模型
- 云端模型
- 第三方 REST API 型 AI

但沙箱不应自行管理模型密钥、模型路由和模型安全边界。

### 4.4 生成和维护沙箱派生数据

沙箱应允许在自己的边界内：

- 创建和维护派生数据
- 保存中间分析结果
- 保存扩展能力的刷新结果
- 形成面向报表和策略的客户定制数据空间

这类写入只能发生在沙箱自己的派生层，不能直接改写平台核心层。

### 4.5 生成受控展示结果

沙箱应允许生成：

- 报表页面
- 分析页面
- 自定义视图
- 结果刷新入口
- 特定扩展能力的功能入口

但这些展示结果是否进入主界面、客户空间菜单或长期导航，必须由平台决定。

### 4.6 使用平台开放的调度能力

沙箱应允许在授权边界内为扩展能力配置：

- 手动触发
- 定时触发
- 周期刷新
- 条件触发

但平台必须控制：

- 频率上限
- 资源上限
- 自动运行权限
- 是否允许长期后台执行

### 4.7 使用平台开放的只读 Twitter / X 能力

完整版沙箱可以在平台授权范围内调用：

- 只读账号信息
- 只读互动信息
- 只读推文信息
- 只读趋势和关系信息

这些只读能力对报表、分析、策略判断和客户定制能力非常重要。

### 4.8 使用平台开放的受控写能力

完整版沙箱在严格约束下，可以通过平台开放的受控写能力触发：

- 回复
- 发帖
- 点赞
- 关注
- 特定任务型动作

但前提是：

- 动作本身已被平台定义为允许开放
- 对应客户空间、角色、账号和通道具备权限
- 满足审核、限流、配额、风控等前置条件

这类能力不是“默认开放”，而是“平台有条件开放”。

## 5. 只能受限开放的能力

这类能力不是绝对禁止，但不应默认开放，也不应在没有平台治理的前提下开放。

### 5.1 跨账号聚合分析

在同一客户空间内，允许围绕多个账号做聚合分析是合理的。  
但跨账号能力必须受客户空间边界和权限边界控制。

### 5.2 跨客户对比能力

这类能力只有在平台显式允许并完成脱敏、授权和审计设计后才可开放。  
默认情况下不应让沙箱获得跨客户对比能力。

### 5.3 自动化写动作

自动发帖、自动回复、自动互动可以成为完整版能力的一部分，  
但必须受到：

- 审核策略
- 配额策略
- 冷却策略
- 风险策略
- 通道策略

的共同约束。

### 5.4 长周期后台任务

允许长期运行的定时任务是有价值的，  
但它们必须接受平台在运行频率、资源消耗、失败处理和人工接管上的治理。

### 5.5 外部能力调用

如果将来需要调用额外外部能力，也应通过平台能力网关受控开放。  
沙箱不应自行决定访问哪些外部系统。

## 6. 原则上不应开放的能力

以下能力不应作为完整版沙箱的默认开放项。

### 6.1 直接改写平台核心数据

沙箱不应直接写入：

- 平台事实层
- 平台状态层
- 平台核心配置
- 平台核心报表定义

否则平台边界会失效。

### 6.2 直接改写平台核心代码和核心界面

沙箱可以生成扩展能力，但不应直接修改：

- 平台主程序逻辑
- 平台核心菜单结构
- 平台核心导航体系
- 平台核心权限体系

这些内容属于平台核心，不属于扩展运行边界。

### 6.3 直接持有或读取原始密钥

沙箱不应直接读取或持有：

- X API 原始密钥
- 模型原始 API Key
- MCP 原始接入密钥
- 其他平台级凭证

沙箱只能通过平台开放的能力网关使用这些能力，不能直接接触原始 Secrets。

### 6.4 任意系统命令执行

沙箱不应被定义为可随意运行系统命令的环境。  
否则它会迅速从“受控扩展区”退化成“高风险本地代码执行平台”。

### 6.5 任意文件系统访问

沙箱不应默认拥有：

- 任意目录读写
- 任意文件修改
- 任意扫描本地系统文件

如果确有文件读写需求，也应通过平台开放的受控文件能力完成，而不是直接开放系统文件权限。

### 6.6 任意网络访问

沙箱不应自行决定访问哪些外部地址。  
外部访问如果有必要，应由平台能力网关、允许列表或代理边界统一控制。

### 6.7 任意安装依赖或构建运行时

沙箱不应以“任意安装依赖、任意扩展运行环境”为前提。  
这会迅速带来不可控的安全风险、兼容性风险和维护风险。

### 6.8 绕过平台审计与审批

任何沙箱能力都不应绕过：

- 审计
- 审批
- 权限判断
- 限流与配额
- 通道治理

否则平台就失去治理意义。

## 7. 平台与沙箱的控制关系

平台与沙箱的关系必须明确分工：

### 平台负责决定

- 开放哪些数据
- 开放哪些知识
- 开放哪些模型
- 开放哪些执行通道
- 开放哪些展示容器
- 哪些扩展能力可以被挂载
- 哪些扩展能力可以自动运行
- 哪些扩展能力只能人工触发
- 哪些扩展能力可以跨账号或跨边界工作

### 沙箱负责决定

- 在开放边界内如何组合能力
- 如何利用开放数据生成分析和报表
- 如何在派生数据空间中组织结果
- 如何把结果转化成可供展示和交付的扩展能力

也就是说：

**平台决定开放什么，沙箱决定如何组合什么。**

## 8. Agent、沙箱与平台核心的责任边界

三者的边界应清楚分开：

### 平台核心

负责：

- 稳定
- 控制
- 权限
- 审计
- 宿主
- 挂载
- 交付

### Agent

负责：

- 理解
- 组合
- 判断
- 生成

### 沙箱

负责：

- 承载
- 隔离
- 运行
- 派生
- 刷新

如果这三者混在一起，平台会失去长期可维护性。

## 9. 完整版沙箱的最终边界

从产品视角看，完整版沙箱应该做到：

- 能基于平台事实层、状态层和客户知识生成扩展能力
- 能拥有自己的派生数据空间
- 能使用平台统一开放的模型、知识、调度和执行能力
- 能把结果转化为菜单入口、报表入口和刷新入口
- 能在平台治理下运行定时能力和受控写能力

但它不应该做到：

- 任意接触平台核心
- 任意改写平台核心
- 任意持有密钥
- 任意访问系统资源
- 任意跨客户访问数据
- 任意绕过平台控制

## 10. 当前结论

在 `TweetPilot` 的场景中，沙箱是**可实现**的，但必须被定义为：

**受控扩展运行边界，而不是通用本地代码执行平台。**

这意味着：

- 做“受控扩展能力沙箱”是合理且有价值的
- 做“无边界本地代码执行平台”既不必要，也不适合作为产品目标

对 `TweetPilot` 来说，真正正确的目标不是让 Agent 获得无限自由，而是：

**让 Agent 在平台授权边界内长出高价值能力，同时让平台始终保有最终控制权。**
