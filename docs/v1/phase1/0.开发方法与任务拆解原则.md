## 1. 文档目的

本文档用于回答一个关键问题：

**`TweetPilot` 第一阶段应该如何从需求文档走向可验证的开发任务。**

本文档不讨论具体业务功能细节，也不替代各工作包文档。

它只定义三件事：

- 第一阶段采用什么开发方法
- 第一阶段的文档体系应该如何组织
- 每一轮开发任务应该如何拆解、如何验收、如何测试

## 2. 基本判断

第一阶段不应采用“按模块整体做完”的开发方式。

也就是说，不应按下面这种顺序推进：

- 先把 A 全做完
- 再把 B 全做完
- 再把 C 全做完

这种方式的问题是：

- 周期长
- 验证慢
- 很容易做到一半才发现设计偏了
- 很容易做出很多“还不能被用户真实使用”的中间层能力

第一阶段更适合采用：

**架构分工 + 纵向切片验证**

也就是：

- `A-H` 用来定义平台结构和职责边界
- 每一轮开发用“一个可验证的业务切片”来推进

## 3. 第一阶段推荐开发方法

第一阶段应采用一种受控的敏捷开发方式：

### 3.1 用 `A-H` 做结构分工

`docs/phase1/A-H` 的价值是：

- 定义职责边界
- 定义依赖关系
- 定义平台长期结构
- 防止边界混乱

它们不应直接等于迭代任务本身。

### 3.2 用“纵向切片”做实际开发

每一轮真正开发的单位，不应是“模块”，而应是：

- 一个用户可感知的结果
- 一个可以跑通的最小链路
- 一个可以被代码和人工共同验证的交付单元

例如：

- 评论进入平台并被本地沉淀
- Reply Agent 生成候选回复
- 高风险回复进入审核队列
- 审核通过后走本地扩展执行通道
- 结果进入报表

这些才是适合迭代的真实开发单元。

### 3.3 每个切片都必须跨工作包

第一阶段的大多数有效切片，都不是单工作包任务。

例如“Reply Agent 生成候选回复”通常会同时涉及：

- `B` 本地数据与状态沉淀
- `C` AI 与知识基础层
- `D` 数字员工与 Reply Agent

这正是敏捷切片的价值：

- 每次迭代都交付一个真实结果
- 每次迭代都验证架构是否成立
- 每次迭代都能发现设计问题，而不是等到所有模块都做完

## 6. 每个迭代切片必须包含什么

当前和后续每一个切片文档，都应至少包含以下部分：

### 6.1 切片目标

说明这一轮到底交付什么。

### 6.2 切片范围

说明：

- 涉及哪些工作包
- 不涉及哪些工作包
- 这一轮明确不做什么

### 6.3 代码验收标准

必须明确：

- 什么行为成立才算完成
- 什么结果出现才算通过
- 哪些异常情况也必须被覆盖

### 6.4 自动测试要求

至少应写明：

- 单元测试覆盖什么
- 集成测试覆盖什么
- 契约测试覆盖什么
- 端到端测试覆盖什么

### 6.5 人工测试手册

至少应写明：

- 测试前准备
- 测试步骤
- 预期结果
- 异常判定标准
- 通过标准

### 6.6 设计守卫清单

必须检查：

- 是否破坏客户空间边界
- 是否破坏事实层 / 状态层边界
- 是否把 AI、知识、执行通道写死
- 是否让 Agent 越权决定平台控制逻辑
- 是否绕过审核和权限边界
- 是否把扩展能力写进平台核心

### 6.7 Definition of Done

只有以下同时成立，切片才算完成：

- 功能可运行
- 自动测试通过
- 人工测试通过
- 日志 / 审计可观察
- 边界不被破坏
- 文档已同步

## 7. 自动测试策略建议

第一阶段建议采用分层测试，而不是只写单元测试。

### 7.1 单元测试

适合覆盖：

- 风险判断逻辑
- 角色选择逻辑
- 状态转换逻辑
- 通道路由规则

### 7.2 集成测试

适合覆盖：

- 本地数据落库
- 任务状态流转
- AI 调用封装
- 审核 / 接管回流

### 7.3 契约测试

适合覆盖：

- `LocalBridge`
- 本地模型网关
- 后续 `X API / XMCP`

这类测试的目标不是验证外部服务本身，而是验证平台适配层没有偏离契约。

### 7.4 端到端测试

适合覆盖：

- Electron 主界面启动
- 用户操作主路径
- 从评论进入到候选回复、审核、执行、报表的最小闭环

## 8. 人工测试策略建议

第一阶段必须保留人工测试手册。

原因很简单：

- 桌面软件很多问题自动化不一定先发现
- 本地执行通道和真实账号上下文天然更适合人工验证
- 审核、接管、报表可读性这些内容，也需要人工判断

每个切片的人工测试手册至少应包含：

- 测试目标
- 环境准备
- 测试数据
- 操作步骤
- 预期结果
- 异常结果判断
- 测试通过标准

## 9. 当前推荐流程

在真正拆第一批切片之前，建议按这个顺序推进：

1. 先保留当前 `A-H` 工作包文档作为结构边界
2. 在 `docs/phase1` 下补一份第一阶段架构执行文档
3. 再开始拆第一批纵向切片
4. 每个切片都带代码验收、自动测试和人工测试手册

## 10. 当前结论

第一阶段正确的方法不是：

- 把 `A-H` 一个个做完

而是：

- 用 `A-H` 守住结构边界
- 用“纵向切片”推动真实迭代
- 每轮都带验收、自动测试和人工测试

这样才能做到：

- 每做完一块就能验证
- 每次迭代都不偏离初始设计
- 平台能在快速验证中持续成长
