## 1. 文档目的

本文档定义第一阶段第四个纵向切片的正式开发边界。

这份文档要解决的问题不是“平台是否已经能真实发出回复”，而是：

- 候选回复如何进入平台任务体系
- 平台如何根据最小风险规则把结果分流
- 高风险结果如何进入审核队列
- 团队成员如何接手 Agent 输出并形成责任归属

## 2. 切片目标

让 Reply Agent 生成的候选回复进入任务体系，并根据最小风险规则进入审核队列或等待后续执行路径，从而证明平台已经具备最小治理链路。

对第一阶段而言，这个切片优先证明：

- Reply Agent 输出不会直接越过人工边界
- 高风险任务已经能被平台接住
- 团队成员已经能对 Agent 输出进行审核和接管
- 平台已经开始形成任务、治理和审计的统一链路

## 3. 主要涉及的工作包

`D + E + F`

补充关系：

- `D` 提供候选回复、角色上下文和最小风险建议
- `E` 负责把候选回复包装成任务对象，并保留后续执行路径
- `F` 负责审核队列、责任归属、人工接管和审计入口

## 4. 前置依赖

依赖 Slice 3 已提供：

- 候选回复已经生成并可被再次读取
- 候选回复已经关联评论输入、账号和角色
- Reply Agent 已能输出最小风险建议或风险标记
- 模型来源与知识命中追踪已可见

同时依赖平台已具备：

- 最小客户空间边界
- 最小账号与角色绑定关系
- 最小任务对象容器或任务入口位

## 5. 本切片的治理链路定义

## 5.1 第一阶段优先支持的治理对象

第一阶段本切片应优先支持：

- 针对单条评论输入生成的候选回复任务

当前不要求：

- 大规模批量审核
- 多层级审批流
- 多团队并行治理流
- 自动化风险升级编排

因此本切片的原则是：

- 先把“候选回复 -> 任务 -> 风险分流 -> 审核入口”打稳
- 先让平台能把高风险结果接住
- 不把第一阶段治理链路做成复杂审批系统

## 5.2 第一阶段最小任务化要求

本切片中，候选回复不能继续停留为单纯结果列表。

平台必须至少把候选回复包装成最小任务对象，任务对象至少能表达：

- 来源评论输入
- 归属客户空间
- 归属账号
- 关联角色
- 候选回复结果
- 最小风险标记
- 当前处理状态

原则：

- 治理动作必须围绕任务对象发生
- 不允许审核和接管绕过任务体系

## 5.3 第一阶段允许的最小风险分流

本切片允许：

- 低风险任务进入“可继续处理”状态
- 高风险任务进入审核队列
- 风险输出作为任务状态或任务标记的一部分被记录

当前不要求：

- 复杂风险评分模型
- 多维风险解释系统
- 风险自动升级到复杂组织流

原则：

- 风险分流要足够简单，但必须可执行
- 重点是把“高风险不能直接越过人工边界”做成立

## 5.4 第一阶段最小审核队列

审核队列在本切片中至少需要支持：

- 让团队看到待处理高风险任务
- 让团队查看候选回复及其上下文
- 让团队对任务执行最小审核动作
- 让任务具备明确责任归属

当前不要求：

- 复杂筛选器
- 复杂审批流编排
- 复杂 SLA 管理

## 5.5 第一阶段最小接管与责任归属

本切片中至少应形成：

- 一个任务可以被指派给某个处理人
- 一个任务可以被人工接管
- 接管和处理结果可以再次回写到任务链路中

原则：

- 第一阶段先解决“任务不能无人负责”
- 不提前做大而全协作系统

## 5.6 本切片不承担的执行深度

本切片当前不承担：

- 真实本地扩展执行通道发送
- 官方执行通道发送
- 最终发送成功或失败的真实回流

这些内容属于后续切片。

因此本切片的边界是：

- 把任务准备好
- 把治理入口接好
- 把人工边界接好
- 但不在这里打通真实发送动作

## 6. 当前轮廓交付

- 高风险任务进入审核队列
- 任务存在明确责任归属
- 人工可指派、审核或接管任务
- 治理结果可回流任务体系

## 7. 当前明确不做

- 不在这一切片中打通真实执行通道
- 不在这一切片中扩展完整权限体系
- 不在这一切片中形成完整客户交付视图
- 不在这一切片中实现复杂审批系统
- 不在这一切片中实现企业级治理编排

## 8. 代码验收标准

本切片完成时，至少必须满足以下条件：

### 8.1 候选回复任务化成立

- 候选回复能够进入任务体系
- 任务能够关联评论输入、账号、角色和候选回复
- 任务不是页面瞬时状态，而是平台正式对象

### 8.2 最小风险分流成立

- 平台能够根据最小风险规则区分不同任务去向
- 高风险任务能够进入审核队列
- 低风险任务至少可以进入“待后续处理”状态

### 8.3 审核队列成立

- 平台中存在审核队列入口
- 团队成员能看到待处理高风险任务
- 团队成员能查看候选回复和最小上下文

### 8.4 责任归属和接管成立

- 一个任务能被指派给处理人
- 一个任务能被人工接管
- 接管或处理结果能回写任务对象

### 8.5 为下个切片提供最小执行准备

- 任务对象已经具备后续进入执行通道的最小结构
- 平台至少可以从任务中读取：候选回复、风险状态、责任归属、当前处理状态

## 9. 自动测试要求

本切片进入开发前，至少应补齐以下自动测试：

## 9.1 单元测试

至少覆盖：

- 候选回复到任务对象的转换逻辑
- 最小风险分流规则
- 审核状态变更逻辑
- 责任归属与接管逻辑
- 任务回写逻辑

## 9.2 集成测试

至少覆盖：

- 候选回复进入任务体系
- 高风险任务进入审核队列
- 任务能够被指派或接管
- 审核或接管结果能够回写任务状态

## 9.3 契约测试

至少覆盖：

- 任务对象与候选回复对象之间的转换契约
- 审核队列读取接口或读取层契约
- 任务状态流转与审计入口之间的契约

说明：

- 本切片契约测试的重点不是外部接口
- 而是平台内部治理链路的结构稳定性

## 9.4 端到端测试

至少覆盖一条最小链路：

1. 平台存在一条候选回复结果
2. 该候选回复进入任务体系
3. 高风险任务进入审核队列
4. 团队成员可在界面中看到并接管该任务

## 10. 人工测试手册

## 10.1 测试目的

验证平台已经具备最小治理链路，能够把高风险候选回复接入审核和接管流程。

## 10.2 测试前准备

- 平台应用可启动
- 已存在一个客户空间和账号
- 平台中已存在至少一条评论输入
- 平台中已存在至少一组候选回复结果
- 候选回复结果中至少有一条带有最小风险标记

## 10.3 测试步骤

1. 打开平台
2. 进入目标客户空间和账号上下文
3. 找到一条已生成候选回复的评论任务
4. 观察该任务是否已经进入任务体系
5. 观察高风险任务是否出现在审核队列中
6. 选择一条高风险任务进行指派、审核或接管
7. 观察处理结果是否回写到任务对象中

## 10.4 预期结果

- 候选回复能够进入任务体系
- 高风险任务能够进入审核队列
- 团队成员可以接手任务
- 治理动作可以回写任务状态
- 平台已经形成最小人工边界

## 10.5 异常判定

以下任一情况都视为未通过：

- 候选回复仍然停留在结果列表，无法进入任务体系
- 高风险任务没有进入审核队列
- 审核队列无法看到任务上下文
- 任务没有责任归属，或无法接管
- 审核和接管结果不能回写任务对象
- 平台在这一切片中直接尝试真实发送动作

## 11. 设计守卫清单

在开发本切片时，必须反复检查：

1. 是否让治理动作围绕任务对象发生，而不是围绕页面状态发生
2. 是否把高风险边界先接住，而不是让高风险结果继续停留在生成侧
3. 是否把责任归属和接管做成了最小可用能力，而不是复杂协作系统
4. 是否提前把真实执行通道打通塞进了治理切片
5. 是否为后续执行切片保留了清晰的任务输入边界
6. 是否让治理结果能够稳定回流到平台任务链路中

## 12. Definition of Done

满足以下条件才算本切片完成：

- 候选回复能够进入任务体系
- 最小风险分流成立
- 高风险任务能够进入审核队列
- 责任归属和人工接管成立
- 治理结果能够回写任务对象
- 自动测试覆盖到单元、集成、契约和最小端到端层
- 人工测试按手册执行后通过
- 没有把真实执行通道、复杂权限体系和复杂审批系统错误塞入本切片
