## 1. 文档目的

本文档定义第一阶段第三个纵向切片的正式开发边界。

这份文档要解决的问题不是“平台是否已经能自动运营”，而是：

- Reply Agent 如何读取平台已有输入
- 账号角色如何真实影响输出结果
- 客户知识与模型能力如何进入生成主链路
- 系统如何稳定产出可采用的候选回复

## 2. 切片目标

让 Reply Agent 基于角色、知识、历史和状态生成候选回复，并让这些候选回复成为平台可追踪、可被后续治理和执行链路消费的正式结果。

对第一阶段而言，这个切片优先证明：

- Reply Agent 不是简单聊天框包装
- 数字员工角色已经真实影响输出
- 平台的本地数据与知识已经进入生成主流程
- 候选回复已经成为平台结果对象，而不是页面文本

## 3. 主要涉及的工作包

`B + C + D`

补充关系：

- `B` 提供评论输入、历史互动、历史推文和最小状态输入
- `C` 提供统一模型入口、知识入口和 AI 调用追踪
- `D` 负责账号角色绑定、上下文组装和候选回复生成

## 4. 前置依赖

依赖 Slice 2 已提供：

- 评论输入已作为平台事实进入本地层
- 评论输入已归属到明确客户空间和账号
- 至少存在一条可重复触发的输入链路
- 平台可从本地层再次读取评论输入

同时依赖平台已具备最小模型和知识前提：

- 至少存在一个可用模型来源
- 至少存在一组可供 Reply Agent 使用的最小知识输入

## 5. 本切片的生成链路定义

## 5.1 第一阶段优先支持的生成对象

第一阶段本切片应优先支持：

- 针对单条评论输入生成候选回复

当前不要求：

- 面向整条会话线程的复杂长期对话生成
- 面向一批评论的批量自动生成
- 面向多账号的批量并行生成

因此本切片的原则是：

- 先把“单条评论 -> 多个候选回复”打稳
- 先让上下文来源和角色影响可解释
- 不把第一阶段生成链路做成过宽的自动化系统

## 5.2 第一阶段最小上下文来源

Reply Agent 在这一切片中至少可以读取：

- 当前评论输入
- 当前评论归属的账号
- 当前账号绑定的默认角色
- 当前账号关联的最小历史互动
- 当前账号关联的最小历史推文
- 当前客户空间下的最小知识输入
- 当前任务使用的模型来源标识

原则：

- Reply Agent 不直接从浏览器原始环境读取上下文
- Reply Agent 优先消费平台本地层已经准备好的输入
- 角色、知识、历史和状态必须通过平台入口被组合，而不是在页面中临时拼接

## 5.3 第一阶段允许的模型与知识使用方式

第一阶段允许：

### A. 统一模型入口

- Reply Agent 通过平台统一模型入口调用模型
- 模型来源可以是本地模型、云端模型或第三方 REST AI
- 但在本切片中，不要求同时支持多模型协同

### B. 统一知识入口

- Reply Agent 通过平台统一知识入口读取客户知识
- 知识可以来自 FAQ、产品资料、品牌边界或 MCP 输出
- 但在本切片中，不要求复杂知识路由和复杂知识治理

原则：

- 模型调用是平台能力，不是 Reply Agent 自行管理的能力
- 知识接入是平台能力，不是页面级临时上下文

## 5.4 第一阶段的最小输出结构

本切片在产品上必须至少形成以下逻辑链：

1. 平台读取一条评论输入
2. 平台识别该评论归属的账号与角色
3. Reply Agent 读取最小历史、知识和状态输入
4. Reply Agent 调用模型生成候选回复
5. 平台将候选回复保存为正式结果对象
6. 平台能够再次读取并展示这些候选回复

这意味着候选回复必须具备以下产品属性：

- 它属于某条评论输入
- 它属于某个账号和角色上下文
- 它来自某次明确的生成动作
- 它可以被后续切片继续消费

## 5.5 第一阶段允许的风险输出形态

本切片允许 Reply Agent 产出最小风险建议或最小风险标记。

但本切片不承担：

- 审核队列
- 审核动作
- 风险升级路径
- 最终执行放行

原则：

- 风险在本切片中只允许形成“生成侧输出”
- 真正的治理流仍然属于后续切片

## 6. 当前轮廓交付

- 账号可绑定默认角色
- Reply Agent 可读取最小历史、状态和知识输入
- 系统可生成多个候选回复版本
- AI 调用与知识命中可被追踪

## 7. 当前明确不做

- 不在这一切片中实现最终执行动作
- 不在这一切片中实现完整审核队列
- 不在这一切片中实现复杂多 Agent 协作
- 不在这一切片中实现复杂模型编排策略
- 不在这一切片中实现批量生成工作流

## 8. 代码验收标准

本切片完成时，至少必须满足以下条件：

### 8.1 角色绑定成立

- 一个账号能够绑定一个默认角色
- Reply Agent 在生成时会读取该角色
- 角色不是展示字段，而会真实影响生成结果

### 8.2 最小上下文组装成立

- Reply Agent 可读取评论输入
- Reply Agent 可读取最小历史互动和历史推文
- Reply Agent 可读取最小客户知识输入
- Reply Agent 可读取当前模型来源标识

### 8.3 候选回复生成成立

- 系统至少可以生成两个候选回复版本
- 候选回复不是页面瞬时文本，而是平台正式结果对象
- 候选回复可以被再次读取和展示

### 8.4 生成追踪成立

- 平台可追踪本次生成使用了哪个模型来源
- 平台可追踪本次生成是否命中了知识输入
- 平台可追踪候选回复属于哪条评论输入和哪个账号

### 8.5 为下个切片提供最小治理准备

- 候选回复已具备后续进入风险分流和审核队列的基础条件
- 至少可以从平台读取：候选回复、来源评论、归属账号、关联角色、生成元信息

## 9. 自动测试要求

本切片进入开发前，至少应补齐以下自动测试：

## 9.1 单元测试

至少覆盖：

- 角色选择与绑定逻辑
- Reply Agent 上下文组装逻辑
- 候选回复结果归一化逻辑
- 生成结果最小校验逻辑
- 风险建议或风险标记的最小输出逻辑

## 9.2 集成测试

至少覆盖：

- 从本地评论输入读取上下文并触发生成
- 候选回复结果写入平台结果层
- 再次读取候选回复结果并建立与评论输入、账号、角色的关联
- 模型来源与知识命中结果可被追踪

## 9.3 契约测试

至少覆盖：

- 平台模型网关适配层
- 平台知识/MCP 输入适配层
- Reply Agent 与统一模型入口之间的调用契约

说明：

- 契约测试的目标不是验证具体大模型质量
- 而是验证平台统一模型与知识入口的行为稳定性

## 9.4 端到端测试

至少覆盖一条最小链路：

1. 平台存在客户空间、账号和角色
2. 平台存在一条评论输入
3. 触发 Reply Agent 生成
4. 看到多个候选回复结果出现在界面中

## 10. 人工测试手册

## 10.1 测试目的

验证 Reply Agent 已经能够站在平台输入层之上，生成可采用的候选回复结果。

## 10.2 测试前准备

- 平台应用可启动
- 已存在一个客户空间
- 已存在一个账号
- 该账号已绑定默认角色
- 平台中已存在至少一条评论输入
- 平台中已存在至少一个可用模型来源
- 平台中已准备最小知识输入，真实或模拟均可

## 10.3 测试步骤

1. 打开平台
2. 进入目标客户空间和账号上下文
3. 选择一条已进入平台的评论输入
4. 触发 Reply Agent 生成流程
5. 观察系统是否生成多个候选回复版本
6. 检查候选回复是否带有正确的归属关系
7. 检查生成过程是否能看到模型来源和知识命中结果

## 10.4 预期结果

- 候选回复能够生成
- 候选回复不止一个版本
- 角色差异能够真实反映在输出中
- 候选回复属于正确评论输入和账号
- 生成追踪信息存在

## 10.5 异常判定

以下任一情况都视为未通过：

- Reply Agent 只能生成单一固定文本
- 角色切换对输出没有任何影响
- 候选回复无法再次读取
- 候选回复无法关联到评论输入或账号
- 模型来源与知识命中结果完全不可见
- 生成流程直接带出了执行或审核动作

## 11. 设计守卫清单

在开发本切片时，必须反复检查：

1. 是否让 Reply Agent 站在平台输入层之上，而不是直接读浏览器原始世界
2. 是否让角色真实影响输出，而不是停留在 UI 配置层
3. 是否让知识输入和模型调用都经过平台统一入口
4. 是否把候选回复沉淀为平台正式结果，而不是页面临时文本
5. 是否提前把执行、审核和复杂治理逻辑塞进了生成切片
6. 是否为后续 Slice 4 保留了最小风险与治理输入

## 12. Definition of Done

满足以下条件才算本切片完成：

- 账号默认角色绑定成立
- Reply Agent 最小上下文组装成立
- 候选回复可被稳定生成、保存和读取
- 生成结果与评论输入、账号、角色之间的归属关系成立
- 模型来源与知识命中追踪成立
- 自动测试覆盖到单元、集成、契约和最小端到端层
- 人工测试按手册执行后通过
- 没有把执行、审核和复杂多 Agent 协作错误塞入本切片
