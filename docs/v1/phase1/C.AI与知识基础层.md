## 1. 文档目的

本文档用于拆解第一阶段工作包 C：AI 与知识基础层。

本文档只定义：

- 第一阶段最小 AI 编排能力需要做到什么程度
- 第一阶段最小知识接入能力需要做到什么程度
- AI、知识、执行通道之间在第一阶段应如何保持解耦
- 本工作包如何支撑 Reply Agent，而不是提前展开完整 Agent 平台

本文档不展开模型接口、提示词细节和具体知识库实现。

## 2. 工作包定位

工作包 C 负责建立第一阶段的智能底座。

它的目标不是“接上一个模型就结束”，而是让平台从第一阶段开始具备以下条件：

- 模型来源可以被统一纳管
- 客户知识可以被统一接入
- Agent 不是死提示词工具
- AI、知识、执行通道不是强耦合流程

如果工作包 C 做得过薄，第一阶段就会变成一次性提示词工程；如果做得过重，又会把第一阶段拖进复杂 AI 平台建设。

## 2A. 第一阶段智能形态

第一阶段的 AI 与知识基础层，不应被定义为“把模型接上去”。

它更准确的产品形态应是：

- 平台统一管理的模型入口
- 平台统一管理的知识入口
- Reply Agent 可调用但不可越权控制的智能底座

第一阶段真正要证明的，不是“模型很多”，而是：

- 平台能够统一纳管不同模型来源
- 平台能够把客户知识引入 Reply Agent 主流程
- AI 调用过程、知识命中和采用结果可以被平台观察和追踪

## 3. 本工作包要解决的问题

### 3.1 统一管理模型来源

第一阶段就要明确平台不是只支持一个模型源，而是至少要形成最小统一接入能力：

- 本地模型
- 云端模型
- 第三方 REST API 型 AI

### 3.2 建立最小模型切换能力

第一阶段不要求复杂策略路由，但必须让平台具备：

- 至少可切换不同模型来源
- 至少可为后续角色和任务绑定模型能力预留入口

### 3.3 建立最小知识接入能力

第一阶段 Reply Agent 不能只靠通用模型。

它至少应能读取：

- 客户 FAQ
- 产品资料
- 品牌边界
- 指定 MCP 输出结果

### 3.4 建立最小解耦框架

第一阶段必须避免写死成：

- 某一个模型 + 某一个知识来源 + 某一个执行通道 = 唯一流程

应从一开始就预留：

- 模型可替换
- 知识可替换
- 执行通道可替换

### 3.5 让 AI 过程可被平台观察

第一阶段不能只关心最终输出，还必须开始建立最小可观测能力：

- 当前用了哪个模型来源
- 是否使用了客户知识或 MCP 输出
- 生成过程是否成功
- 输出结果是否被采用

## 4. 第一阶段交付目标

工作包 C 在第一阶段至少应交付以下结果：

### 4.1 最小 AI 编排中心

至少需要支持：

- 统一登记不同模型来源
- 查看当前可用模型来源
- 为后续任务和角色绑定模型能力预留入口

### 4.2 本地模型与云端模型的最小统一接入

至少需要支持：

- 接入本地模型
- 接入至少一种云端模型
- 保持两者在平台内有统一调用入口

### 4.3 最小模型切换能力

至少需要支持：

- 在第一阶段业务闭环内切换模型来源
- 不因为切换模型来源而重写主流程

### 4.4 最小 MCP / 知识接入

至少需要支持：

- 引入客户知识
- 引入指定 MCP 输出
- 让 Reply Agent 在生成回复时可消费这些知识

### 4.5 AI 调用过程与知识命中追踪

至少需要支持：

- 查看任务实际使用的模型来源
- 查看任务是否命中了客户知识或 MCP 输出
- 查看 AI 调用是否成功
- 为后续审计和报表保留基础追踪结果

### 4.6 AI、知识、执行通道的最小解耦框架

至少需要做到：

- Reply Agent 的生成不被某一个执行通道写死
- 客户知识接入不依赖某一个模型源
- 模型调用不与某个具体账号或实例强耦合

## 5. 范围边界

### 5.1 本工作包负责什么

工作包 C 负责：

- 模型来源纳管
- 模型切换基础能力
- 客户知识与 MCP 的最小接入
- AI 调用过程与知识命中的最小追踪
- AI、知识、执行通道的基础解耦

### 5.2 本工作包不负责什么

工作包 C 不负责：

- 复杂模型编排策略
- 多模型协同推理
- 长链路 Agent 规划能力
- 完整知识治理体系
- 客户知识的复杂版本控制
- 完整沙箱内模型执行策略
- 复杂 AI 评估体系

### 5.3 本工作包与其他工作包的边界

- 与工作包 B 的边界：B 提供数据与状态输入，C 提供模型与知识输入
- 与工作包 D 的边界：C 提供 Reply Agent 的智能底座，D 负责角色、回复生成与风险分流逻辑
- 与工作包 E 的边界：C 负责生成侧解耦，E 负责执行侧解耦
- 与工作包 F 的边界：C 提供 AI 调用和知识命中追踪基础，F 负责把这些结果纳入治理与审计链路

## 6. 核心设计约束

### 6.1 模型调用是基础能力，模型管理才是本工作包重点

第一阶段不需要反复强调“Agent 会调用模型”，但必须明确平台如何管理模型来源和切换能力。

### 6.2 客户知识必须从第一阶段进入闭环

第一阶段 Reply Agent 必须能消费客户知识，否则生成质量和专业度无法体现平台差异。

### 6.3 AI、知识、执行通道必须解耦

第一阶段即使做最小系统，也不能把 AI、知识和执行通道写成一条不可拆的流程。

### 6.4 平台负责模型管理，Agent 负责调用与组合

模型来源和访问权限应由平台统一控制，Agent 只在平台开放的边界内调用和组合。

### 6.5 知识接入必须是平台能力，而不是临时拼接

第一阶段客户知识和 MCP 接入应作为平台能力存在，而不是在某个 Reply Agent 会话里临时拼接。

### 6.6 第一阶段要证明“可管理的智能”，不是“更多模型”

第一阶段的重点不是接入尽可能多的模型，而是让模型、知识和追踪结果进入平台长期能力边界。

## 7. 依赖关系

### 7.1 本工作包的前置依赖

工作包 C 依赖工作包 A 提供：

- 平台宿主
- 配置入口
- 客户空间归属边界

### 7.2 本工作包的输出依赖

工作包 C 的输出会被下列工作包直接依赖：

- 工作包 D：Reply Agent 读取模型与知识能力
- 工作包 E：执行任务在生成前使用模型与知识编排结果
- 工作包 G：报表与交付可以消费 AI 洞察结果
- 工作包 H：后续扩展能力通过平台统一模型网关调用模型

## 8. 完成标准

工作包 C 完成至少应满足：

- 平台至少可统一接入本地模型和一种云端模型
- 平台具备最小模型切换能力
- 平台具备最小客户知识 / MCP 接入能力
- 平台具备最小 AI 调用过程与知识命中追踪能力
- Reply Agent 主流程可消费客户知识和模型能力
- AI、知识、执行通道之间未被写成不可替换的一条流程

## 9. 后续补充方向

后续补充应优先围绕：

- 模型来源管理方式
- 模型切换与路由策略
- 客户知识和 MCP 的最小治理边界
- AI 调用进展与知识命中追踪方式
- 角色与模型绑定关系
- Reply Agent 生成上下文组装原则
