## 1. 文档目的

本文档用于拆解第一阶段工作包 D：数字员工与 Reply Agent。

本文档只定义：

- 第一阶段数字员工体系最少要做到什么程度
- 第一阶段 Reply Agent 的职责边界是什么
- Reply Agent 与平台、数据、AI、执行通道之间如何配合
- 第一阶段完成时，什么样的回复闭环才算成立

本文档不展开提示词细节、评分规则、实现算法和接口参数。

## 2. 工作包定位

工作包 D 是第一阶段最核心、最值钱的工作包。

它承接的是 `TweetPilot` 第一阶段对外最容易被客户感知的价值：

- 平台不是冷冰冰的回复工具
- 平台可以让不同账号拥有不同数字员工角色
- Agent 可以基于历史数据、状态和客户知识生成可用回复
- 系统可以把回复这件事从人工逐条处理升级为 AI 协同处理

如果工作包 D 不成立，第一阶段就没有真正的“产品亮点”。

## 2A. 第一阶段产品形态

第一阶段的 Reply Agent 不应被定义为一个“自动回复机器人”。

它更准确的产品形态应是：

- 一个围绕评论与提及任务工作的数字员工
- 一个建立在平台历史数据、客户知识和角色设定之上的回复生成器
- 一个会把结果送入平台治理与执行流程的 Agent

第一阶段用户真正感知到的，不应只是“系统会生成回复”，而应是：

- 某个账号拥有自己的默认数字员工角色
- 新评论进入平台后可以被 Agent 处理
- Agent 会给出多个候选回复
- 高风险任务不会直接失控执行
- 最终结果会进入任务、审计和报表体系

## 3. 本工作包要解决的问题

### 3.1 让账号拥有角色，而不是只有统一语气

第一阶段必须支持：

- 不同账号绑定不同默认角色
- 相同任务在不同角色下生成不同风格回复
- 平台开始形成“数字员工”而不是“统一模型输出”的产品感知

### 3.2 让 Reply Agent 基于真实上下文工作

第一阶段 Reply Agent 不能只看一条评论文本。

它至少应组合：

- 当前评论或提及的上下文
- 历史互动与历史推文
- 本地状态信息
- 客户知识与品牌边界
- 当前账号绑定的角色设定

### 3.3 让生成结果进入受控流程

第一阶段 Reply Agent 不只是“给一个答案”，而是要把结果送入平台受控流程：

- 输出候选回复
- 区分高风险和低风险
- 交给执行通道或审核流程
- 记录采用结果和最终执行结果

### 3.4 让数字员工价值可被观察

第一阶段至少要让平台能看到：

- 某个角色是否被使用
- 某个回复版本是否被采用
- 某次回复最终是自动执行还是人工审核

### 3.5 让 Reply Agent 成为可持续工作单元

第一阶段不要求 Reply Agent 变成复杂多 Agent 系统，但必须让它具备最小“持续工作单元”的特征：

- 它有明确输入来源
- 它有明确上下文组装方式
- 它有明确输出结果
- 它有明确后续流转路径
- 它的结果可以被后续团队成员和后续系统接管

## 4. 第一阶段交付目标

工作包 D 在第一阶段至少应交付以下结果：

### 4.1 最小角色系统

至少需要支持：

- 定义角色
- 让账号绑定默认角色
- 在回复生成时带入角色设定

### 4.2 账号绑定默认角色

至少需要支持：

- 一个账号存在一个明确默认角色
- 角色配置不依赖临时会话存在
- 后续任务默认继承账号角色上下文

### 4.3 Reply Agent 的最小任务入口

至少需要支持：

- 承接评论或提及任务
- 让任务进入 Reply Agent 处理路径
- 让任务与账号、角色和客户空间产生明确归属关系

### 4.4 Reply Agent 的最小上下文组装

至少需要支持：

- 读取当前互动上下文
- 读取历史互动和历史推文
- 读取客户知识和品牌边界
- 把账号角色作为生成上下文的一部分带入

### 4.5 Reply Agent 的最小生成逻辑

至少需要支持：

- 生成可供采用的候选回复

### 4.6 多版本候选回复

至少需要支持：

- 输出不止一个候选回复版本
- 让用户能看到角色差异或表达差异
- 为审核和人工接管提供选择空间

### 4.7 高风险与低风险基础分流

至少需要支持：

- 对回复任务做最小风险判断
- 区分可直接进入执行路径的低风险任务
- 区分应进入人工审核或升级路径的高风险任务

### 4.8 执行前的人机协同落点

至少需要支持：

- 候选回复可被人工选择
- 高风险回复可进入审核
- Reply Agent 输出可被送入执行通道或人工接管路径

### 4.9 回复结果回流

至少需要支持：

- 记录最终采用了哪个候选版本
- 记录最终是否发送成功
- 让结果能回流到本地数据、报表和审计体系中

## 5. 范围边界

### 5.1 本工作包负责什么

工作包 D 负责：

- 最小角色系统
- 账号与角色绑定
- Reply Agent 的任务入口
- Reply Agent 的上下文组装与回复生成
- 候选回复输出
- 风险基础分流
- 人机协同落点
- 结果回流入口

### 5.2 本工作包不负责什么

工作包 D 不负责：

- Content Agent 全量能力
- Growth Agent 全量能力
- Risk Agent 全量能力
- 自动化高风险写动作
- 完整多 Agent 协同
- 沙箱中的 Agent 生成式扩展能力
- 执行通道路由策略本体
- 审核队列和权限治理本体

### 5.3 本工作包与其他工作包的边界

- 与工作包 B 的边界：B 提供历史数据与状态输入，D 负责利用这些输入做回复生成
- 与工作包 C 的边界：C 提供模型与知识输入，D 负责把这些输入与角色和任务结合
- 与工作包 E 的边界：D 生成候选回复和风险分流结果，E 负责决定结果如何以任务形式进入执行通道和任务系统
- 与工作包 F 的边界：D 输出高风险任务，F 承接审核、接管和审计
- 与工作包 G 的边界：D 提供角色使用结果和回复结果，G 负责汇总展示

## 6. 核心设计约束

### 6.1 Reply Agent 必须建立在平台资产上

第一阶段 Reply Agent 必须优先消费平台的本地历史数据、状态快照和客户知识，而不是只依赖临时上下文。

### 6.2 角色系统必须真实影响输出

角色不能只是展示字段，必须真实影响回复风格、表达方式和输出结果。

### 6.3 高风险动作不能被默认自动化

第一阶段 Reply Agent 可以做风险基础分流，但不应默认自动化放行高风险写动作。

### 6.4 Reply Agent 的结果必须可追踪

第一阶段必须能追踪：

- 生成了哪些候选回复
- 最终采用了哪个版本
- 最终如何被执行或审核

### 6.5 Reply Agent 不应直接拥有执行控制权

第一阶段 Reply Agent 可以给出候选结果和风险分流判断，但不应自己决定最终执行通道、最终审核路径和最终权限放行。

### 6.6 第一阶段应优先证明“可控协同”，而不是“完全自动化”

第一阶段的目标是做出一个可控的 Reply Agent 闭环，而不是证明平台已经具备全面自动回复能力。

## 7. 依赖关系

### 7.1 本工作包的前置依赖

工作包 D 依赖：

- 工作包 B 提供历史数据与状态快照
- 工作包 C 提供模型与知识能力

### 7.2 本工作包的输出依赖

工作包 D 的输出会被下列工作包直接依赖：

- 工作包 E：把候选回复与风险分流结果接入执行流程
- 工作包 F：承接高风险审核和人工接管
- 工作包 G：形成报表、复盘和客户可读结果

## 8. 完成标准

工作包 D 完成至少应满足：

- 至少一个账号可以绑定默认角色
- 至少一个 Reply Agent 流程可以承接评论或提及任务
- 至少一个 Reply Agent 流程可以基于历史数据、状态和客户知识生成候选回复
- 系统可以输出多个候选回复版本
- 系统可以对高风险与低风险任务做基础分流
- 系统可以让候选结果进入执行或审核路径
- 最终采用结果和执行结果可被回写与追踪

## 9. 后续补充方向

后续补充应优先围绕：

- 角色体系的结构化方式
- Reply Agent 任务入口与任务归属方式
- Reply Agent 的上下文组装方式
- 风险分流原则
- 候选回复的比较与采用逻辑
- 角色效果和回复质量的观测方式
