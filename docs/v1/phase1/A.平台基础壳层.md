## 1. 文档目的

本文档用于拆解第一阶段工作包 A：平台基础壳层。

本文档只定义：

- 这个工作包在第一阶段的定位
- 这个工作包必须交付的最小范围
- 这个工作包与其他工作包的边界
- 这个工作包完成的判断标准

本文档不展开技术实现细节、字段设计、接口参数和数据库结构。

## 2. 工作包定位

工作包 A 负责建立 `TweetPilot` 第一阶段运行所需的最小平台宿主。

它解决的不是“业务是否智能”，而是更底层的问题：

- 平台是否有稳定的桌面壳层
- 核心对象是否有统一的归属关系
- 运行状态是否有统一入口
- 后续数据、Agent、执行通道、协作和报表是否有稳定承载面

如果工作包 A 没有成立，后续工作包就只能变成零散功能堆叠，而不是一个平台。

## 2A. 第一阶段平台形态

第一阶段的平台基础壳层，不应被理解为“把几个页面放进桌面窗口里”。

它更准确的产品形态应是：

- 平台所有能力的稳定宿主
- 平台统一控制面的入口
- 客户空间、账号、实例、执行通道和运行状态的统一承载层

第一阶段真正要证明的，不是“界面已经很丰富”，而是：

- 平台已经有一个稳定的主壳层
- 核心对象已经有明确归属和入口
- 后续 Agent、任务、报表、扩展能力都能进入同一宿主，而不是各自长成独立页面

## 3. 本工作包要解决的问题

### 3.1 建立平台宿主

第一阶段必须先有一个可持续承载后续能力的桌面宿主，而不是直接围绕某个 Agent 页面做单点功能。

### 3.2 建立核心对象入口

第一阶段的核心对象至少包括：

- 客户空间
- 账号资产
- 扩展实例
- 执行通道

这些对象必须有统一入口，否则后续数据、任务、审计和报表都无法稳定挂靠。

### 3.3 建立最小总控视图

第一阶段即使不做复杂总控台，也必须让用户能看到：

- 当前有哪些客户空间
- 当前有哪些账号
- 当前有哪些实例
- 当前有哪些执行通道
- 当前哪些对象在线、异常或不可用

### 3.4 建立后续能力的挂载面

第一阶段不要求把所有业务面板做完，但必须预留：

- 页面导航结构
- 模块挂载位置
- 状态展示容器
- 后续扩展能力和受控展示的入口位

### 3.5 建立平台控制面

第一阶段的平台壳层还必须承担一个更上位的作用：

- 成为平台控制面的统一入口
- 让用户能够切换客户空间、查看状态、进入任务和进入结果视图
- 让后续所有核心能力都围绕同一控制面组织，而不是围绕单个业务流零散展开

## 4. 第一阶段交付目标

工作包 A 在第一阶段至少应交付以下结果：

### 4.1 最小桌面宿主

- 应用主窗口与基础布局
- 全局导航与基础页面切换能力
- 稳定的状态栏或总览入口

### 4.2 平台控制面入口

至少需要支持：

- 进入总控台
- 进入客户空间视图
- 进入账号、实例、执行通道等核心对象入口
- 进入后续任务、报表和扩展能力的预留挂载位

### 4.3 最小总控台

至少能够展示：

- 客户空间总数
- 账号总数与基础状态
- 实例总数与基础状态
- 执行通道总数与基础状态
- 关键异常提示入口

### 4.4 最小运行状态总览

至少需要支持：

- 平台级运行状态统一入口
- 账号、实例和执行通道状态的统一可见性
- 关键异常的统一提示位置

### 4.5 最小客户空间边界

至少需要支持：

- 创建或识别客户空间
- 在客户空间维度组织账号、知识、报表和扩展能力归属
- 让后续所有第一阶段能力都默认落在客户空间边界内

### 4.6 最小账号资产管理入口

至少需要支持：

- 查看账号列表
- 查看账号归属的客户空间
- 查看账号基础状态
- 查看账号与默认角色、实例、执行通道之间的挂接关系

### 4.7 最小实例管理入口

至少需要支持：

- 查看扩展实例列表
- 查看实例在线状态
- 查看实例关联账号
- 查看实例基础健康状态

### 4.8 最小执行通道管理入口

至少需要支持：

- 查看可用执行通道
- 区分本地扩展执行通道与官方执行通道
- 查看通道基础状态
- 为后续任务执行、限流和回退提供挂载入口

### 4.9 后续工作包挂载位

至少需要支持：

- 角色与 Agent 的挂载位
- 任务与审核的挂载位
- 报表与交付的挂载位
- 扩展能力与受控展示的挂载位

## 5. 范围边界

### 5.1 本工作包负责什么

工作包 A 负责：

- 平台桌面宿主
- 平台控制面入口
- 页面与导航骨架
- 核心对象的管理入口
- 基础运行状态的统一展示入口
- 后续工作包的挂载面

### 5.2 本工作包不负责什么

工作包 A 不负责：

- 具体数据沉淀逻辑
- 具体 AI 编排逻辑
- 具体 Reply Agent 生成逻辑
- 具体执行动作流程
- 具体审核与审计流程
- 具体报表生成逻辑
- 完整沙箱能力
- 平台业务规则本体

### 5.3 本工作包与其他工作包的边界

- 与工作包 B 的边界：A 提供对象宿主和展示入口，B 负责本地数据与状态沉淀
- 与工作包 C 的边界：A 提供配置和管理入口，C 负责 AI 与知识能力本体
- 与工作包 D 的边界：A 提供角色与 Agent 的挂载位，D 负责数字员工和 Reply Agent 行为本体
- 与工作包 E 的边界：A 提供执行通道入口和平台控制面，E 负责执行与任务本体
- 与工作包 F 的边界：A 提供治理入口和展示位，F 负责协作、审核和审计本体
- 与工作包 G 的边界：A 提供总控与结果展示挂载位，G 负责报表与交付本体
- 与工作包 H 的边界：A 提供主壳层与展示宿主，H 负责扩展能力边界预留

## 6. 核心设计约束

### 6.1 平台壳层必须稳定

平台基础壳层应作为稳定宿主存在，不应与某个单独 Agent 流程强耦合。

### 6.2 客户空间必须前置出现

虽然第一阶段不会做完整多租户治理，但客户空间边界必须从一开始就进入平台壳层，否则后续所有归属关系都会漂移。

### 6.3 状态展示必须是平台级的

运行状态不应分散到各个功能页里，而应在平台层有统一入口。

### 6.4 扩展能力入口必须预留

第一阶段即使不落完整沙箱，也必须保证后续扩展能力可以被挂载到平台，而不是被迫重构主壳层。

### 6.5 平台壳层必须先于业务闭环形成

第一阶段不应先围绕 Reply Agent 做孤立页面，再反向拼出平台。正确顺序应是先有平台宿主，再让业务闭环挂载进来。

### 6.6 控制面必须属于平台，而不是属于某个单独工作包

总控入口、状态入口和核心对象入口都应属于平台壳层统一承载，而不是分散在数据、Agent、任务或报表页面中各自维护。

## 7. 依赖关系

### 7.1 本工作包的前置依赖

工作包 A 没有业务层前置依赖，它是第一阶段的起点工作包。

### 7.2 本工作包的输出依赖

工作包 A 的输出会被下列工作包直接依赖：

- 工作包 B：依赖客户空间、账号、实例和状态展示挂载面
- 工作包 C：依赖 AI 与知识配置入口
- 工作包 D：依赖角色、Agent 和结果展示挂载位
- 工作包 E：依赖执行通道入口
- 工作包 F：依赖平台权限和任务入口
- 工作包 G：依赖总控台与报表挂载位
- 工作包 H：依赖基础展示容器与边界预留

## 8. 完成标准

工作包 A 完成至少应满足：

- 平台有稳定可运行的桌面宿主
- 平台有统一控制面入口
- 平台有最小总控入口
- 客户空间、账号、实例、执行通道有统一管理入口
- 基础运行状态可见
- 后续工作包有明确挂载位置
- 第一阶段后续能力不需要为了进入主界面而重构壳层

## 9. 后续补充方向

后续补充应优先围绕：

- 页面结构与导航拆分
- 平台控制面结构
- 总控台信息层级
- 客户空间切换与归属规则
- 账号、实例、执行通道页面之间的关系
- 平台级状态展示方式
