## 1. 任务目标

建立 Slice 5 的执行通道对象模型和执行请求基础，让平台能够明确表达本地扩展执行通道和官方执行通道，并为任务到执行请求的转换提供统一抽象。

本任务卡是 Slice 5 的基础，后续所有执行相关任务都将建立在这些对象之上。

### 2.2 执行请求对象

**文件**: `src/domain/executionRequest.ts`

```typescript
export type ExecutionRequestId = string;
export type ExecutionRequestStatus = 
  | "pending"
  | "in_progress"
  | "completed"
  | "failed"
  | "cancelled";

export type ExecutionActionType = "reply"; // 第一阶段只支持回复动作

export interface ExecutionRequest {
  id: ExecutionRequestId;
  taskId: string;
  channelId: string;
  channelType: ExecutionChannelType;
  actionType: ExecutionActionType;
  status: ExecutionRequestStatus;
  payload: ExecutionPayload;
  result?: ExecutionResult;
  error?: ExecutionError;
  createdAt: Date;
  updatedAt: Date;
  executedAt?: Date;
}

export interface ExecutionPayload {
  commentInputId: string;        // 来源：CommentInput ID
  candidateReplyId: string;      // 来源：CandidateReply ID
  targetTweetId: string;         // 来源：CommentInput.targetTweetId
  replyContent: string;          // 来源：CandidateReply.content
  accountId: string;             // 来源：ReplyTask.accountId
  workspaceId: string;           // 来源：ReplyTask.workspaceId
  roleId?: string;               // 来源：ReplyTask.roleId（可选）
}

export interface ExecutionResult {
  success: boolean;
  tweetId?: string;
  platformResponse?: Record<string, unknown>;
  executedAt: Date;
}

export interface ExecutionError {
  code: string;
  message: string;
  retryable: boolean;
  details?: Record<string, unknown>;
}
```

**说明**:
- `ExecutionRequest` 是任务到真实执行的桥梁
- 第一阶段优先支持 `actionType: "reply"`
- `payload` 包含执行所需的全部信息
- `result` 和 `error` 互斥，分别表达成功和失败结果

### 2.3 执行请求工厂

**文件**: `src/domain/executionRequest.ts`

```typescript
export interface CreateExecutionRequestParams {
  taskId: string;
  channelId: string;
  channelType: ExecutionChannelType;
  actionType: ExecutionActionType;
  payload: ExecutionPayload;
}

export function createExecutionRequest(
  params: CreateExecutionRequestParams
): ExecutionRequest {
  return {
    id: generateExecutionRequestId(),
    taskId: params.taskId,
    channelId: params.channelId,
    channelType: params.channelType,
    actionType: params.actionType,
    status: "pending",
    payload: params.payload,
    createdAt: new Date(),
    updatedAt: new Date(),
  };
}

export function generateExecutionRequestId(): ExecutionRequestId {
  return `exec-req-${Date.now()}-${Math.random().toString(36).substring(2, 9)}`;
}
```

### 2.4 任务到执行请求的转换逻辑

**文件**: `src/services/executionRequestBuilder.ts`

```typescript
import type { ReplyTask } from "../domain/replyTask";
import type { CandidateReply } from "../domain/candidateReply";
import type { CommentInput } from "../domain/commentInput";
import type { ExecutionChannel } from "../domain/executionChannel";
import type { ExecutionRequest, CreateExecutionRequestParams } from "../domain/executionRequest";
import { createExecutionRequest } from "../domain/executionRequest";
import type { ICommentInputRepository } from "../data/commentInputRepository";

export interface BuildExecutionRequestParams {
  task: ReplyTask;
  candidateReply: CandidateReply;
  channel: ExecutionChannel;
}

export class ExecutionRequestBuilder {
  constructor(
    private readonly commentInputRepository: ICommentInputRepository
  ) {}

  async build(params: BuildExecutionRequestParams): Promise<ExecutionRequest> {
    const { task, candidateReply, channel } = params;

    // 验证通道类型与任务匹配
    if (channel.accountId !== task.accountId) {
      throw new Error(
        `Channel accountId mismatch: channel=${channel.accountId}, task=${task.accountId}`
      );
    }

    // 验证通道支持回复能力
    if (!channel.capabilities.includes("reply")) {
      throw new Error(
        `Channel ${channel.id} does not support reply capability`
      );
    }

    // 读取 CommentInput 获取 targetTweetId
    // 字段来源链：ReplyTask -> CandidateReply -> CommentInput -> ExecutionRequest
    const commentInput = await this.commentInputRepository.findById(
      candidateReply.commentInputId
    );
    
    if (!commentInput) {
      throw new Error(
        `CommentInput ${candidateReply.commentInputId} not found`
      );
    }

    if (!commentInput.targetTweetId) {
      throw new Error(
        `CommentInput ${commentInput.id} has no targetTweetId`
      );
    }

    const requestParams: CreateExecutionRequestParams = {
      taskId: task.id,
      channelId: channel.id,
      channelType: channel.type,
      actionType: "reply",
      payload: {
        commentInputId: commentInput.id,
        candidateReplyId: candidateReply.id,
        targetTweetId: commentInput.targetTweetId,
        replyContent: candidateReply.content,
        accountId: task.accountId,
        workspaceId: task.workspaceId,
        roleId: task.roleId,
      },
    };

    return createExecutionRequest(requestParams);
  }
}
```

**说明**:
- `ExecutionRequestBuilder` 负责从任务、候选回复和执行通道构造执行请求
- **字段来源链**：`ReplyTask -> CandidateReply -> CommentInput -> ExecutionRequest`
  - `targetTweetId` 来自 `CommentInput.targetTweetId`（不是 `CandidateReply`）
  - `replyContent` 来自 `CandidateReply.content`
  - `accountId`, `workspaceId`, `roleId` 来自 `ReplyTask`
- 包含基础验证逻辑（账号匹配、能力检查、字段完整性）
- 第一阶段只支持 `actionType: "reply"`

### 2.5 执行请求仓储接口

**文件**: `src/data/executionRequestRepository.ts`

```typescript
import type { ExecutionRequest, ExecutionRequestId } from "../domain/executionRequest";

export interface IExecutionRequestRepository {
  save(request: ExecutionRequest): Promise<void>;
  findById(id: ExecutionRequestId): Promise<ExecutionRequest | null>;
  findByTaskId(taskId: string): Promise<ExecutionRequest[]>;
  findByAccountId(accountId: string, limit?: number): Promise<ExecutionRequest[]>;
  update(request: ExecutionRequest): Promise<void>;
}

export class InMemoryExecutionRequestRepository implements IExecutionRequestRepository {
  private requests: Map<ExecutionRequestId, ExecutionRequest> = new Map();

  async save(request: ExecutionRequest): Promise<void> {
    this.requests.set(request.id, { ...request });
  }

  async findById(id: ExecutionRequestId): Promise<ExecutionRequest | null> {
    const request = this.requests.get(id);
    return request ? { ...request } : null;
  }

  async findByTaskId(taskId: string): Promise<ExecutionRequest[]> {
    return Array.from(this.requests.values())
      .filter((req) => req.taskId === taskId)
      .map((req) => ({ ...req }));
  }

  async findByAccountId(accountId: string, limit?: number): Promise<ExecutionRequest[]> {
    const results = Array.from(this.requests.values())
      .filter((req) => req.payload.accountId === accountId)
      .sort((a, b) => b.createdAt.getTime() - a.createdAt.getTime())
      .map((req) => ({ ...req }));
    
    return limit ? results.slice(0, limit) : results;
  }

  async update(request: ExecutionRequest): Promise<void> {
    if (!this.requests.has(request.id)) {
      throw new Error(`ExecutionRequest ${request.id} not found`);
    }
    this.requests.set(request.id, { ...request, updatedAt: new Date() });
  }
}
```

## 3. 架构约束

### 3.1 执行通道属于平台，不属于 Agent

- 执行通道是平台对象，由平台统一管理
- Agent 不直接持有或选择执行通道
- 通道路由由平台策略决定

### 3.1.5 与现有平台对象的关系

- 本任务卡对 `src/domain/types.ts` 中的 `ExecutionChannel` 进行增量扩展
- 不是定义第二套通道模型
- 新增字段（`capabilities`）需要在平台状态初始化时提供
- 后续任务卡（T5-02, T5-04）将使用统一的通道对象

### 3.1.6 执行通道对象与执行器实现的边界

**重要边界说明**：

- `ExecutionChannel` 是**平台路由与治理对象**，不等于具体 Twitter 管理实现
- 它表达"平台选择走哪条通道"，不表达"这条通道由谁提供能力"
- 当前 `local-bridge` 通道由 `LocalBridgeReplyExecutor` 提供能力（见 T5-01）
- 后续 `x-mcp` 通道可由新的执行器实现提供能力（见 Slice 7）
- 平台编排层依赖 `ExecutionChannel` 进行路由决策
- 执行服务层依赖统一执行器接口进行真实发送（见 T5-01 的 `ITwitterReplyExecutor`）

这种分层设计确保：
- 通道路由逻辑与具体实现解耦
- 后续接入新执行器时不需要重构路由层
- 平台可以在不改变通道模型的情况下切换底层实现

### 3.2 任务先于执行请求

- 必须先有 `ReplyTask`，再有 `ExecutionRequest`
- 执行请求不能绕过任务体系直接创建

### 3.3 官方执行通道预留

- 第一阶段主要使用本地扩展执行通道
- 但 `ExecutionChannelType` 必须包含 `"x-api"` 和 `"x-mcp"`
- 为后续官方通道接入预留结构

### 3.4 执行请求与任务解耦

- `ExecutionRequest` 是独立对象，不嵌入 `ReplyTask`
- 一个任务可能产生多个执行请求（重试、多通道尝试）
- 执行请求通过 `taskId` 关联任务
