## 1. 任务目标

建立执行保护边界和失败处理机制，让平台能够避免重复执行、处理执行失败、记录失败原因，并为人工接管提供入口。

本任务卡确保执行动作在受控边界内运行，失败后不会失控，为 Slice 5 提供最小安全保障。

## 2. 核心交付物

**说明**：本任务卡的所有保护和失败处理逻辑都是"执行器无关"的，只依赖平台统一的执行请求和错误结构，不依赖 LocalBridge 或 Twitter MCP 的具体实现。

### 2.1 重复执行保护

**文件**: `src/domain/executionProtection.ts`

```typescript
import type { ReplyTask } from "./replyTask";
import type { ExecutionRequest } from "./executionRequest";

export interface DuplicateExecutionCheckResult {
  allowed: boolean;
  reason?: string;
  existingRequestId?: string;
}

export class DuplicateExecutionProtector {
  /**
   * 检查任务是否已有执行请求
   * 防止同一任务被重复执行
   */
  check(
    task: ReplyTask,
    existingRequests: ExecutionRequest[]
  ): DuplicateExecutionCheckResult {
    // 查找该任务的所有执行请求
    const taskRequests = existingRequests.filter(
      (req) => req.taskId === task.id
    );

    if (taskRequests.length === 0) {
      // 没有执行请求，允许执行
      return { allowed: true };
    }

    // 检查是否有进行中或已完成的请求
    const activeRequest = taskRequests.find(
      (req) =>
        req.status === "pending" ||
        req.status === "in_progress" ||
        req.status === "completed"
    );

    if (activeRequest) {
      return {
        allowed: false,
        reason: `Task already has an active execution request (status: ${activeRequest.status})`,
        existingRequestId: activeRequest.id,
      };
    }

    // 所有请求都是 failed 或 cancelled，允许重试
    return { allowed: true };
  }
}
```

**说明**:
- `DuplicateExecutionProtector` 防止同一任务被重复执行
- 如果任务已有 pending/in_progress/completed 请求，拒绝新请求
- 如果所有请求都是 failed/cancelled，允许重试

### 2.2 执行失败处理器

**文件**: `src/services/executionFailureHandler.ts`

```typescript
import type { ReplyTask } from "../domain/replyTask";
import { appendReplyTaskEvent, markReplyTaskStatus } from "../domain/replyTask";
import type { ExecutionRequest, ExecutionError } from "../domain/executionRequest";
import type { IReplyTaskRepository } from "../data/replyTaskRepository";

export interface HandleExecutionFailureParams {
  task: ReplyTask;
  request: ExecutionRequest;
  error: ExecutionError;
}

export interface HandleExecutionFailureResult {
  taskUpdated: boolean;
  requiresManualIntervention: boolean;
  retryable: boolean;
}

export class ExecutionFailureHandler {
  constructor(private readonly taskRepository: IReplyTaskRepository) {}

  async handle(
    params: HandleExecutionFailureParams
  ): Promise<HandleExecutionFailureResult> {
    const { task, request, error } = params;

    // 判断是否需要人工介入
    const requiresManualIntervention = this.shouldRequireManualIntervention(error);

    let updatedTask = task;

    // 根据失败状态矩阵决定任务后续状态（见 2.1.5 节）
    if (requiresManualIntervention) {
      // 需要人工介入，任务进入 in_takeover 状态
      updatedTask = markReplyTaskStatus(updatedTask, "in_takeover");
      updatedTask = appendReplyTaskEvent(updatedTask, {
        type: "task_failed",
        actorId: "system",
        payload: {
          executionRequestId: request.id,
          errorCode: error.code,
          errorMessage: error.message,
          retryable: error.retryable,
          requiresManualIntervention: true,
        },
      });
    } else if (error.retryable) {
      // 可重试错误，任务回到 ready_for_execution 状态
      updatedTask = markReplyTaskStatus(updatedTask, "ready_for_execution");
      updatedTask = appendReplyTaskEvent(updatedTask, {
        type: "task_failed",
        actorId: "system",
        payload: {
          executionRequestId: request.id,
          errorCode: error.code,
          errorMessage: error.message,
          retryable: true,
          requiresManualIntervention: false,
        },
      });
    } else {
      // 不可重试错误，任务进入 rejected 状态
      updatedTask = markReplyTaskStatus(updatedTask, "rejected");
      updatedTask = appendReplyTaskEvent(updatedTask, {
        type: "task_failed",
        actorId: "system",
        payload: {
          executionRequestId: request.id,
          errorCode: error.code,
          errorMessage: error.message,
          retryable: false,
          requiresManualIntervention: false,
        },
      });
    }

    await this.taskRepository.update(updatedTask);

    return {
      taskUpdated: true,
      requiresManualIntervention,
      retryable: error.retryable,
    };
  }

  private shouldRequireManualIntervention(error: ExecutionError): boolean {
    // 定义需要人工介入的错误类型（与 T5-01 的 2.0.4 节对齐）
    const manualInterventionCodes = [
      "NOT_LOGGED_IN",
      "RATE_LIMITED",
      "CONTENT_VIOLATION",
      "ACCOUNT_SUSPENDED",
      "PERMISSION_DENIED",
    ];

    return manualInterventionCodes.includes(error.code);
  }
}
```

**说明**:
- `ExecutionFailureHandler` 处理执行失败场景
- 使用 `markReplyTaskStatus()` 和 `appendReplyTaskEvent()` 聚合根方法
- 根据失败状态矩阵（见 2.1.5 节）决定任务后续状态
- 与 T4-04 的接管服务对齐：进入 `in_takeover` 后等待人工处理

### 2.1.5 失败后任务状态决策矩阵

**说明**：本节定义执行失败后任务状态的统一决策规则，与 T5-01 的错误码分类对齐。

| 错误类型 | 任务状态 | 说明 |
|---------|---------|------|
| 需要人工介入（见下表） | `in_takeover` | 进入人工接管流程，等待 T4-04 的接管服务处理 |
| 可重试错误 | `ready_for_execution` | 回到待执行状态，等待重试 |
| 不可重试错误 | `rejected` | 任务失败，不再重试 |

**需要人工介入的错误码**（与 T5-01 的 2.0.4 节对齐）：
- `NOT_LOGGED_IN` - 账号未登录
- `RATE_LIMITED` - 触发速率限制
- `CONTENT_VIOLATION` - 内容违规
- `ACCOUNT_SUSPENDED` - 账号被封禁
- `PERMISSION_DENIED` - 权限不足

**与治理层的衔接**：
- 任务进入 `in_takeover` 后，应通知治理层或记录到接管队列
- 后续由 T4-04 的 `taskOwnershipService.takeOverTask()` 处理
- 接管完成后，可将任务重新置为 `ready_for_execution`、`rejected` 或 `completed`

### 2.3 执行冷却机制

**文件**: `src/domain/executionCooldown.ts`

```typescript
import type { ExecutionRequest } from "./executionRequest";

export interface CooldownCheckResult {
  allowed: boolean;
  reason?: string;
  cooldownUntil?: Date;
}

export interface CooldownConfig {
  minIntervalMs: number; // 最小执行间隔（毫秒）
}

export class ExecutionCooldownChecker {
  constructor(
    private readonly config: CooldownConfig = { minIntervalMs: 5000 }
  ) {}

  /**
   * 检查是否在冷却期内
   * 防止短时间内频繁执行
   * 注意：冷却检查按账号或通道维度，而非单个任务
   */
  check(
    accountId: string,
    channelId: string,
    recentRequests: ExecutionRequest[]
  ): CooldownCheckResult {
    // 找到该账号或通道最近的执行请求
    // 注意：这里需要按 accountId 或 channelId 过滤，而非 taskId
    const relevantRequests = recentRequests
      .filter((req) => {
        // 从 payload 中提取 accountId
        const reqAccountId = req.payload.accountId;
        return reqAccountId === accountId || req.channelId === channelId;
      })
      .sort((a, b) => b.createdAt.getTime() - a.createdAt.getTime());

    if (relevantRequests.length === 0) {
      // 没有历史请求，允许执行
      return { allowed: true };
    }

    const lastRequest = relevantRequests[0];
    const timeSinceLastExecution = Date.now() - lastRequest.createdAt.getTime();

    if (timeSinceLastExecution < this.config.minIntervalMs) {
      const cooldownUntil = new Date(
        lastRequest.createdAt.getTime() + this.config.minIntervalMs
      );

      return {
        allowed: false,
        reason: `Account ${accountId} or channel ${channelId} is in cooldown period (${this.config.minIntervalMs}ms)`,
        cooldownUntil,
      };
    }

    return { allowed: true };
  }
}
```

**说明**:
- `ExecutionCooldownChecker` 实现最小执行间隔
- 防止短时间内频繁执行（默认 5 秒间隔）
- 第一阶段使用简单的时间间隔策略

### 2.3.5 执行请求仓储扩展

**说明**：冷却保护需要按账号或通道维度查询最近的执行请求，因此需要扩展仓储接口。

**文件**: `src/data/executionRequestRepository.ts`（扩展）

```typescript
import type { ExecutionRequest, ExecutionRequestId } from "../domain/executionRequest";

export interface IExecutionRequestRepository {
  save(request: ExecutionRequest): Promise<void>;
  findById(id: ExecutionRequestId): Promise<ExecutionRequest | null>;
  findByTaskId(taskId: string): Promise<ExecutionRequest[]>;
  update(request: ExecutionRequest): Promise<void>;
  
  /**
   * 查询指定账号最近的执行请求
   * 用于冷却期保护（跨任务维度）
   */
  findRecentByAccountId(accountId: string, withinMs: number): Promise<ExecutionRequest[]>;
  
  /**
   * 查询指定通道最近的执行请求
   * 用于冷却期保护（跨账号维度）
   */
  findRecentByChannelId(channelId: string, withinMs: number): Promise<ExecutionRequest[]>;
}

export class InMemoryExecutionRequestRepository implements IExecutionRequestRepository {
  private requests: Map<ExecutionRequestId, ExecutionRequest> = new Map();

  async save(request: ExecutionRequest): Promise<void> {
    this.requests.set(request.id, { ...request });
  }

  async findById(id: ExecutionRequestId): Promise<ExecutionRequest | null> {
    const request = this.requests.get(id);
    return request ? { ...request } : null;
  }

  async findByTaskId(taskId: string): Promise<ExecutionRequest[]> {
    return Array.from(this.requests.values())
      .filter((req) => req.taskId === taskId)
      .map((req) => ({ ...req }));
  }

  async update(request: ExecutionRequest): Promise<void> {
    if (!this.requests.has(request.id)) {
      throw new Error(`ExecutionRequest ${request.id} not found`);
    }
    this.requests.set(request.id, { ...request, updatedAt: new Date() });
  }

  async findRecentByAccountId(accountId: string, withinMs: number): Promise<ExecutionRequest[]> {
    const cutoffTime = Date.now() - withinMs;
    return Array.from(this.requests.values())
      .filter((req) => {
        const reqAccountId = req.payload.accountId;
        return reqAccountId === accountId && req.createdAt.getTime() >= cutoffTime;
      })
      .map((req) => ({ ...req }));
  }

  async findRecentByChannelId(channelId: string, withinMs: number): Promise<ExecutionRequest[]> {
    const cutoffTime = Date.now() - withinMs;
    return Array.from(this.requests.values())
      .filter((req) => {
        return req.channelId === channelId && req.createdAt.getTime() >= cutoffTime;
      })
      .map((req) => ({ ...req }));
  }
}
```

**说明**:
- 新增 `findRecentByAccountId()` 方法：按账号查询最近的执行请求
- 新增 `findRecentByChannelId()` 方法：按通道查询最近的执行请求
- 这两个方法是冷却保护的正式仓储契约，不再是 TODO
- 内存实现已包含完整逻辑，可直接用于测试

```typescript
import type { ReplyTask } from "../domain/replyTask";
import type { ExecutionRequest } from "../domain/executionRequest";
import type { IExecutionRequestRepository } from "../data/executionRequestRepository";
import { DuplicateExecutionProtector } from "../domain/executionProtection";
import { ExecutionCooldownChecker } from "../domain/executionCooldown";

export interface CheckExecutionProtectionParams {
  task: ReplyTask;
  channelId: string;  // 新增：用于冷却期检查
}

export interface CheckExecutionProtectionResult {
  allowed: boolean;
  reason?: string;
  violations: string[];
}

export class ExecutionProtectionService {
  constructor(
    private readonly requestRepository: IExecutionRequestRepository,
    private readonly duplicateProtector: DuplicateExecutionProtector = new DuplicateExecutionProtector(),
    private readonly cooldownChecker: ExecutionCooldownChecker = new ExecutionCooldownChecker()
  ) {}

  async checkProtection(
    params: CheckExecutionProtectionParams
  ): Promise<CheckExecutionProtectionResult> {
    const { task, channelId } = params;
    const violations: string[] = [];

    // 读取该任务的所有执行请求
    const taskRequests = await this.requestRepository.findByTaskId(task.id);

    // 检查 1: 重复执行保护
    const duplicateCheck = this.duplicateProtector.check(task, taskRequests);
    if (!duplicateCheck.allowed) {
      violations.push(duplicateCheck.reason!);
    }

    // 检查 2: 冷却期保护
    // 按账号查询最近的执行请求（跨任务维度）
    const recentRequests = await this.requestRepository.findRecentByAccountId(
      task.accountId,
      60000 // 最近 1 分钟
    );
    
    const cooldownCheck = this.cooldownChecker.check(
      task.accountId,
      channelId,
      recentRequests
    );
    if (!cooldownCheck.allowed) {
      violations.push(cooldownCheck.reason!);
    }

    // 汇总结果
    if (violations.length > 0) {
      return {
        allowed: false,
        reason: violations.join("; "),
        violations,
      };
    }

    return {
      allowed: true,
      violations: [],
    };
  }
}
```

**说明**:
- `ExecutionProtectionService` 整合所有保护检查
- 检查重复执行和冷却期
- 返回所有违规项

### 2.5 带保护的执行编排服务

**文件**: `src/services/protectedTaskExecutionOrchestrator.ts`

```typescript
import type { ReplyTask } from "../domain/replyTask";
import { TaskExecutionOrchestrator, type ExecuteTaskParams, type ExecuteTaskResult } from "./taskExecutionOrchestrator";
import { ExecutionProtectionService } from "./executionProtectionService";
import { ExecutionFailureHandler } from "./executionFailureHandler";
import type { IReplyTaskRepository } from "../data/replyTaskRepository";
import type { IExecutionRequestRepository } from "../data/executionRequestRepository";

/**
 * 受保护执行编排服务的参数
 * 扩展基础参数，增加权限检查所需的 actorId 和 actorRoles
 */
export interface ProtectedExecuteTaskParams extends ExecuteTaskParams {
  actorId: string;        // 执行者 ID（必填）
  actorRoles?: string[];  // 执行者角色（可选）
}

export class ProtectedTaskExecutionOrchestrator {
  constructor(
    private readonly taskRepository: IReplyTaskRepository,
    private readonly requestRepository: IExecutionRequestRepository,
    private readonly baseOrchestrator: TaskExecutionOrchestrator,
    private readonly protectionService: ExecutionProtectionService,
    private readonly failureHandler: ExecutionFailureHandler
  ) {}

  async executeTask(params: ProtectedExecuteTaskParams): Promise<ExecuteTaskResult> {
    const { taskId, channelId, actorId, actorRoles } = params;

    // 步骤 0: 权限检查
    if (!this.hasExecutionPermission(actorId, actorRoles)) {
      return {
        success: false,
        executionRequestId: "",
        error: {
          code: "PERMISSION_DENIED",
          message: `Actor ${actorId} does not have execution permission`,
          retryable: false,
        },
      };
    }

    // 读取任务
    const task = await this.taskRepository.findById(taskId);
    if (!task) {
      throw new Error(`ReplyTask ${taskId} not found`);
    }

    // 步骤 1: 执行保护检查
    // channelId 必须来自 T5-02 的路由服务，不允许为空
    const protectionCheck = await this.protectionService.checkProtection({ 
      task,
      channelId  // 来自上游路由服务的必填参数
    });
    if (!protectionCheck.allowed) {
      return {
        success: false,
        executionRequestId: "",
        error: {
          code: "PROTECTION_VIOLATION",
          message: protectionCheck.reason!,
          retryable: false,
        },
      };
    }

    // 步骤 2: 执行任务
    const result = await this.baseOrchestrator.executeTask({
      taskId,
      channelId,
      actorId,  // 必须传递执行者 ID
    });

    // 步骤 3: 如果执行失败，处理失败
    if (!result.success && result.error) {
      const request = await this.requestRepository.findById(result.executionRequestId);
      if (request && request.error) {
        await this.failureHandler.handle({
          task,
          request,
          error: request.error,
        });
      }
    }

    return result;
  }

  private hasExecutionPermission(actorId: string, actorRoles?: string[]): boolean {
    // 第一阶段简单实现：检查角色
    // 后续可扩展为完整的 RBAC 模型
    return actorRoles?.includes("executor") || actorRoles?.includes("admin") || false;
  }
}
```

**说明**:
- `ProtectedTaskExecutionOrchestrator` 是带保护的执行编排服务
- 在执行前检查保护边界
- 在执行失败后调用失败处理器
- 对外提供统一的执行入口

**channelId 来源说明**:
- `channelId` 必须来自 T5-02 的路由服务（`ExecutionReadinessService` 或 `ChannelRoutingStrategy`）
- 不允许在 UI 层或编排层自行推断 channelId
- 调用链路: UI → 路由服务（获取 channelId）→ 受保护执行编排服务
- 这确保通道选择权始终属于平台策略，不属于 UI 或编排层

## 3. 架构约束

### 3.1 保护边界必须先于执行出现

- 所有执行动作必须通过保护检查
- 不允许绕过保护直接执行
- 保护逻辑与执行逻辑解耦

### 3.2 失败处理必须是统一链路

- 所有执行失败必须通过 `ExecutionFailureHandler` 处理
- 不允许在执行器实现层或 UI 层直接处理失败
- 失败处理只依赖统一的 `ExecutionError` 结构，不依赖具体执行器实现
- 失败事件必须记录到任务事件流

**执行器无关设计**：
- 失败处理器接收的是平台统一的 `ExecutionError`，不是 LocalBridge 或 Twitter MCP 的原始错误
- 同一套失败治理逻辑可以兼容所有执行器实现
- 错误码分类（可重试、需要人工介入）由执行器实现层统一映射

### 3.3 人工接管入口必须保留

- 特定错误类型自动触发人工接管
- 任务进入 `in_takeover` 状态后，等待人工处理
- 人工接管与自动重试互斥

### 3.4 保护策略可配置

- 冷却时间、重试策略等应该可配置
- 第一阶段使用简单默认值
- 为后续企业级治理预留扩展点
