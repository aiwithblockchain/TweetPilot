- 任务类型：`基础设施 + 宿主`

## 1. 任务目标

建立稳定的 Electron 宿主和第一阶段基础测试底座，让后续任务可以在同一宿主和同一测试体系上继续推进。

## 2. 建议修改范围

优先修改：

- `/Users/wesley/aiwithblockchain/TweetPilot/package.json`
- `/Users/wesley/aiwithblockchain/TweetPilot/vite.config.ts`
- `/Users/wesley/aiwithblockchain/TweetPilot/tsconfig.json`
- `/Users/wesley/aiwithblockchain/TweetPilot/electron/main.ts`
- `/Users/wesley/aiwithblockchain/TweetPilot/electron/preload.ts`

允许新增：

- `/Users/wesley/aiwithblockchain/TweetPilot/src/test/**`
- `/Users/wesley/aiwithblockchain/TweetPilot/tests/**`
- `/Users/wesley/aiwithblockchain/TweetPilot/playwright.config.*`
- `/Users/wesley/aiwithblockchain/TweetPilot/vitest.config.*`

## 3. 详细工作内容

1. 规范 Electron 主窗口启动逻辑，保证开发和构建后启动都稳定。
2. 为渲染层和 Electron 侧建立最小测试基础设施。
3. 为 preload 暴露的最小平台 API 建立稳定类型边界。
4. 在不引入业务逻辑的前提下，让后续 UI 测试和集成测试可以落地。

## 4. 明确不做

- 不实现业务页面
- 不实现数据模型
- 不实现评论输入链路
- 不实现真实 AI 调用

## 5. 交付结果

完成后至少应存在：

- 可稳定运行的桌面宿主
- 最小 preload API 类型定义
- `unit/integration/e2e` 测试命令骨架
- 一条最小启动级测试用例

## 6. AI 自检清单

1. 开发模式和构建后启动都能工作。
2. preload 暴露 API 不依赖未实现的业务对象。
3. 新增测试工具链不会破坏现有 `build/start`。
4. 没有把业务数据或业务流程提前塞进宿主层。

## 7. 测试代码要求

必须新增：

- 至少 1 个单元测试：验证 preload 暴露对象或工具函数。
- 至少 1 个集成测试：验证应用主入口可被挂载。
- 至少 1 个端到端测试：验证应用启动后能进入主界面。

## 8. 人工验收说明

1. 运行开发命令，确认窗口正常打开。
2. 运行构建并通过 `npm start` 启动，确认窗口正常打开。
3. 执行测试命令，确认最小测试链路可运行。
4. 确认此任务未引入任何业务页面或假业务逻辑。

## 9. 完成定义

- 宿主稳定
- 测试底座可用
- 基础类型边界成立
- 测试代码已提交
