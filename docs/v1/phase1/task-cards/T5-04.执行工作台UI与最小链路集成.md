## 3. 架构约束

### 3.1 UI 层不直接调用执行器

- UI 层只能调用服务层（如 `ProtectedTaskExecutionOrchestrator`）
- 不允许 UI 层直接调用执行器实现（如 `LocalBridgeReplyExecutor`）
- 保持分层清晰

### 3.2 执行状态必须实时反馈

- 执行过程中显示进度状态
- 执行完成后立即刷新任务列表
- 失败时显示明确的错误信息

### 3.3 执行通道由平台策略决定，UI 只展示

- UI 不允许用户自由选择通道，通道由平台路由策略决定
- UI 应展示平台已选择的通道类型和路由原因
- 通道选择权属于平台策略，不属于 UI 层
- 如需手动覆盖通道，应作为受控管理动作，而非默认 UI 交互

### 3.3.5 UI 展示内容说明

UI 应分开展示两个概念：
- **通道类型**：`local-bridge`、`x-api`、`x-mcp`（平台路由层概念）
- **当前执行器**：`LocalBridge`、`Twitter MCP`（具体实现层概念）

示例展示：
```
通道类型: local-bridge
当前执行器: LocalBridge (http://127.0.0.1:10088)
路由原因: Local bridge channel is available (Slice 5 primary path)
```

这种展示方式确保后续切换执行器时，UI 层不需要重构概念模型。

### 3.4 与审核工作台保持一致

- 执行工作台的 UI 风格应与审核工作台（T4-05）保持一致
- 复用相同的组件和样式
- 保持用户体验的连贯性

### 3.5 执行权限边界

**UI 层展示**：
- 查看权限：所有团队成员可以查看任务列表和执行历史
- 执行权限：只有具备执行权限的成员可以看到"执行"按钮
- 接管权限：只有具备接管权限的成员可以看到"接管"按钮

**服务层契约**（必须下沉到服务层）：
- `ProtectedTaskExecutionOrchestrator.executeTask()` 必须接收 `actorId` 参数
- 执行前必须验证 `actorId` 是否具备执行权限
- 无执行权限时返回 `PERMISSION_DENIED` 错误码
- 第一阶段使用简单的权限检查（如角色判断），后续可扩展为完整的 RBAC 模型

**修订后的服务接口**：
```typescript
export interface ExecuteTaskParams {
  taskId: string;
  channelId: string;
  actorId: string;        // 新增：执行者 ID
  actorRoles?: string[];  // 新增：执行者角色（可选）
}

export class ProtectedTaskExecutionOrchestrator {
  async executeTask(params: ExecuteTaskParams): Promise<ExecuteTaskResult> {
    const { taskId, channelId, actorId, actorRoles } = params;
    
    // 步骤 0: 权限检查
    if (!this.hasExecutionPermission(actorId, actorRoles)) {
      return {
        success: false,
        executionRequestId: "",
        error: {
          code: "PERMISSION_DENIED",
          message: `Actor ${actorId} does not have execution permission`,
          retryable: false,
        },
      };
    }
    
    // ... 后续执行逻辑
  }
  
  private hasExecutionPermission(actorId: string, actorRoles?: string[]): boolean {
    // 第一阶段简单实现：检查角色
    // 后续可扩展为完整的 RBAC 模型
    return actorRoles?.includes("executor") || actorRoles?.includes("admin") || false;
  }
}
```

**说明**：
- 权限边界不能只停留在 UI 文本，必须在服务层强制执行
- UI 层根据权限决定是否显示按钮，但服务层必须再次验证
- 这种设计确保即使 UI 被绕过，服务层仍然能够拒绝无权限的执行请求

### 8.1 测试目的

### 8.2 测试前准备

- 平台应用可启动
- 存在至少一个 `ready_for_execution` 状态的任务
- 本地扩展执行通道可用（或使用 Mock）
- 测试账号和测试环境已准备

### 8.3 测试步骤

1. 启动平台应用
2. 导航到 "Execution" 视图
3. 验证可执行任务列表显示
4. 选择一个任务
5. 验证任务详情显示
6. 验证平台选择的执行通道信息显示（通道类型、执行器、路由原因）
7. 验证"执行"按钮根据权限显示或隐藏
8. 点击 "Execute Task" 按钮
9. 观察执行状态变化
10. 等待执行完成
11. 验证任务状态更新
12. 查看执行历史

### 8.4 预期结果

- 可执行任务正确显示
- 任务详情完整
- 平台选择的执行通道信息清晰展示（通道类型 + 执行器）
- 执行按钮根据权限正确显示或隐藏
- 执行过程有状态反馈
- 执行完成后任务状态更新
- 执行历史正确记录

以下情况视为未通过：
- UI 无法显示可执行任务
- 执行按钮无响应
- 执行过程无状态反馈
- 执行完成后任务状态未更新
- 执行失败时无错误提示
- 执行历史未记录
