## 1. 任务目标

建立第一阶段最小审核队列，让团队成员能读取待审核任务、查看候选回复上下文，并对任务执行基础审核动作。

## 2. 核心交付物

### 2.1 领域对象

- `src/domain/reviewQueue.ts`
- `ReviewQueueItem`
- `ReviewDecision`

```typescript
export interface ReviewQueueItem {
  taskId: string;
  candidateReplyId: string;
  commentInputId: string;
  accountId: string;
  workspaceId: string;
  riskLevel: "medium" | "high";
  assigneeId?: string;
  createdAt: Date;
}

export interface ReviewQueueQuery {
  limit?: number;
  offset?: number;
  riskLevel?: "medium" | "high";
  sortBy?: "createdAt" | "riskLevel";
  sortOrder?: "asc" | "desc";
}

export interface ReviewDecision {
  taskId: string;
  action: "approve" | "reject" | "return_to_queue";
  actorId: string;
  note?: string;
}
```

### 2.2 仓储接口

- 可复用 `IReplyTaskRepository.findPendingReview()` 作为审核队列底层数据源。
- 如读取逻辑变复杂，可新增 `IReviewQueueQueryService`，但不再引入独立持久化层。

### 2.3 服务层

- `src/services/reviewQueueService.ts`

```typescript
export interface ReviewQueueService {
  listPending(
    workspaceId: string,
    query?: ReviewQueueQuery,
  ): Promise<{ items: ReviewQueueItem[]; total: number }>;
  decide(decision: ReviewDecision): Promise<ReplyTask>;
}
```

服务职责：

- 读取待审核任务并组装最小上下文。
- 审批通过时，将任务置为 `ready_for_execution`。
- 审批拒绝时，将任务置为 `rejected`。
- 审核动作前必须检查最小权限，第一阶段至少要求 `assigneeId`、`takenOverBy` 或具备 `reviewer` / `admin` 能力的操作者才能审核。
- 所有审核动作写入 `review_decided` 事件。

### 2.4 错误处理策略

- `REVIEW_PERMISSION_DENIED`
- `REVIEW_TASK_NOT_FOUND`
- `REVIEW_QUEUE_INVALID_QUERY`

### 2.5 UI 组件

- 本任务定义数据模型，不强制交付完整 UI。
- 只要求后续 UI 能消费 `ReviewQueueItem`。

## 3. 实施步骤

1. 定义 `ReviewQueueItem`、`ReviewQueueQuery` 和 `ReviewDecision`。
2. 实现审核队列读取服务，支持分页、过滤和排序，并拼接任务、候选回复、评论输入的最小上下文。
3. 实现审核动作 `approve`、`reject`、`return_to_queue`。
4. 约束只有待审核任务且权限通过的操作者才能执行审核动作。
5. 补齐读取与审核动作的单元测试、集成测试。

## 4. 验收标准

- [ ] 平台可分页读取待审核任务列表。
- [ ] 列表结果至少包含候选回复、评论输入、账号、风险等级和责任归属信息。
- [ ] 审核动作只能作用于待审核任务。
- [ ] 审核动作包含最小权限检查，未经授权的操作者无法审批。
- [ ] 审核结果会回写到任务状态并记录事件。

## 5. 测试要求

### 5.1 单元测试

- 验证待审核任务会被组装为 `ReviewQueueItem`。
- 验证 `listPending()` 的 `limit`、`offset`、`riskLevel`、`sortBy`、`sortOrder` 生效。
- 验证 `approve` 会把任务从 `pending_review` 推进到 `ready_for_execution`。
- 验证 `reject` 会把任务置为 `rejected`。
- 验证非待审核任务调用审核动作时会失败。
- 验证未被指派且无审核权限的操作者会收到 `REVIEW_PERMISSION_DENIED`。

### 5.2 集成测试

- 验证高风险任务进入审核队列后可被分页读取，并返回 `total`。
- 验证审批通过后任务不再出现在待审核列表。
- 验证审批拒绝后任务状态可在详情读取中看到。
- 验证带过滤条件时只返回匹配风险等级的任务。
