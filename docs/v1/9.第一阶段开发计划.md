# 9.第一阶段开发计划

- 版本：`v0.3.0`
- 更新日期：`2026-04-09`
- 文档状态：`轮廓版`
- 文档定位：本文档用于定义 `TweetPilot` 第一阶段的开发轮廓、工作包划分、交付顺序与阶段门槛。本文档不展开字段、接口参数、数据库结构与详细开发步骤，后续逐步补充。

## 1. 文档目的

本文档只回答四个问题：

- 第一阶段到底要交付什么
- 第一阶段按什么工作包推进
- 各工作包之间的依赖关系是什么
- 第一阶段做到什么程度才算完成

本文档的目标不是“把第一阶段写细”，而是先把第一阶段的整体骨架定住，避免后续一边开发一边改目标。

## 2. 第一阶段定位

第一阶段不是做一个大而全的平台，也不是把所有 Agent 体系都一次性做完。

第一阶段的核心目标只有一个：

**交付一个最小但完整可用、可演示、可销售、可继续演进的 Reply Agent 闭环。**

这个闭环必须同时证明七件事：

- 平台能管理账号、实例和执行通道
- 平台能沉淀本地运营数据和状态
- Agent 能结合角色与客户知识生成高质量回复
- 高风险动作可以被审核、接管和审计
- 任务与结果可以被持续追踪
- Agent 的关键运行状态可以被观察
- 结果能进入报表和基础交付视图

## 3. 第一阶段成功标准

第一阶段完成，不以“功能数量”判断，而以以下结果判断：

### 3.1 能跑通最小业务闭环

围绕某个客户空间下的某个账号，系统可以完成：

1. 发现评论或提及
2. 读取上下文
3. 读取本地历史数据、状态快照和客户知识
4. 调用 AI 生成候选回复
5. 按角色生成不同风格版本
6. 根据策略选择本地扩展执行通道或官方执行通道
7. 发送、审核或人工接管
8. 回写结果、记录日志、进入报表

### 3.2 能体现平台与 Agent 的边界

第一阶段必须能证明：

- 平台核心负责控制、数据、权限、审计和展示
- Agent 负责理解、判断、生成和执行
- 扩展能力边界已预留，但不侵入平台核心

### 3.3 能体现产品价值

第一阶段必须让客户看见：

- 回复效率提升
- 回复质量和一致性提升
- 风险识别能力
- 报表与审计能力
- 角色化数字员工价值

## 4. 第一阶段交付范围

第一阶段交付范围分成八个工作包。

## 4.1 工作包 A：平台基础壳层

目标：

- 建立第一阶段运行所需的桌面宿主和基础控制层

本阶段需要承接：

- 最小总控台
- 最小运行状态总览
- 最小客户空间边界
- 最小账号资产管理
- 最小实例管理
- 最小执行通道管理
- 平台控制面入口
- 后续工作包挂载位

这一工作包解决的是“平台有没有壳”和“核心对象是否有归属与管理入口”。

## 4.2 工作包 B：本地数据与状态沉淀

目标：

- 建立 Reply Agent 闭环所需的最小数据底座

本阶段需要承接：

- 历史互动落库
- 历史推文落库
- 任务结果落库
- 最小事实层与状态层分层
- 最小状态快照
- 基础运行状态输出
- 关键结果回流

这一工作包解决的是“Agent 和报表是不是建立在本地长期资产上”，而不是建立在一次性抓取上。

## 4.3 工作包 C：AI 与知识基础层

目标：

- 建立最小 AI 编排能力和最小知识接入能力

本阶段需要承接：

- 最小 AI 编排中心
- 本地模型 / 云端模型的最小统一接入
- 最小模型切换能力
- 最小 MCP / 知识接入
- AI 调用过程与知识命中追踪
- AI、知识、执行通道的最小解耦框架

这一工作包解决的是“Agent 会不会只是一个死提示词工具”。

## 4.4 工作包 D：数字员工与 Reply Agent

目标：

- 建立第一阶段真正对外可展示的智能能力

本阶段需要承接：

- 最小角色系统
- 账号绑定默认角色
- Reply Agent 的任务入口
- Reply Agent 的上下文组装与生成逻辑
- 多版本候选回复
- 高风险与低风险基础分流
- 执行前的人机协同落点

这是第一阶段的核心工作包。

## 4.5 工作包 E：执行通道与任务编排

目标：

- 建立受控执行能力，而不是只做生成

本阶段需要承接：

- 本地扩展执行通道接入
- 官方执行通道预留
- 基础凭证与密钥治理
- 基础任务编排模型
- 基础任务与结果追踪
- 基础限流、冷却、失败保护

这一工作包解决的是“系统是否真的能把 AI 结果变成运营动作”。

## 4.6 工作包 F：协作、审核与审计

目标：

- 建立团队使用所需的最小治理能力

本阶段需要承接：

- 基础审核队列
- 基础处理责任归属
- 基础指派与人工接管
- 高风险任务升级
- 最小权限分层边界
- 基础审计记录
- Agent 基础可观测性
- 治理结果回流

这一工作包决定产品是不是“团队可用”，而不只是“个人 demo 可用”。

## 4.7 工作包 G：报表与交付

目标：

- 建立第一阶段的业务结果输出

本阶段需要承接：

- 基础运营报表
- 基础风险摘要
- 基础任务结果视图
- 基础客户可读输出

这一工作包决定第一阶段能不能真正形成演示、销售和交付闭环。

## 4.8 工作包 H：扩展能力与沙箱边界预留

目标：

- 不在第一阶段完整开放沙箱，但必须把后续扩展路线预留出来

本阶段需要承接：

- 平台能力分层边界
- 扩展能力边界预留
- 最小受控展示容器预留
- 最小扩展能力挂载位预留
- 最小派生数据边界预留
- 平台最终控制权边界预留

说明：

- 第一阶段不以完整沙箱落地为目标
- 第一阶段不以自动化写 Twitter / X 动作为目标
- 第一阶段只要求后续扩展路线不被架构堵死

沙箱第一版开放范围以后续文档为准：

- [8.第一版沙箱开放能力清单.md](/Users/wesley/aiwithblockchain/TweetPilot/docs/8.第一版沙箱开放能力清单.md)

## 5. 第一阶段的交付顺序

第一阶段建议按下面顺序推进：

1. 平台基础壳层  
2. 本地数据与状态沉淀  
3. AI 与知识基础层  
4. 数字员工与 Reply Agent  
5. 执行通道与任务编排  
6. 协作、审核与审计  
7. 报表与交付  
8. 扩展能力与沙箱边界预留

这个顺序的原因是：

- 先有壳，再有数据
- 先有数据，再做 Agent
- 先有生成能力，再接执行能力
- 先有执行能力，再补治理和交付
- 最后预留扩展能力边界

## 6. 第一阶段明确不做什么

为了保证范围收敛，第一阶段明确不做：

- 大而全多 Agent 协同平台
- 完整 Content Agent / Growth Agent / Risk Agent 体系
- 完整客户交付门户
- 完整企业权限体系
- 完整多租户治理体系
- 完整沙箱运行时
- 自动化高风险写动作
- 通用代码执行平台

这些能力不是不要，而是不属于第一阶段的交付目标。

## 7. 第一阶段依赖关系

各工作包之间的关键依赖如下：

- `工作包 B` 依赖 `工作包 A`
- `工作包 C` 依赖 `工作包 A`
- `工作包 D` 依赖 `工作包 B` 和 `工作包 C`
- `工作包 E` 依赖 `工作包 A` 和 `工作包 D`
- `工作包 F` 依赖 `工作包 D` 和 `工作包 E`
- `工作包 G` 依赖 `工作包 B`、`工作包 D`、`工作包 E`、`工作包 F`
- `工作包 H` 依赖 `工作包 A` 的平台边界和 `工作包 G` 的展示基础

## 8. 第一阶段验收门槛

第一阶段完成至少应满足：

### 8.1 业务门槛

- 至少一个客户空间内的 Reply Agent 闭环可稳定运行
- 至少一个账号可完成从评论发现到回复执行的完整流程
- 至少一种角色化回复生成能力可被稳定使用

### 8.2 管理门槛

- 运行状态可见
- 任务结果可追踪
- 高风险任务可审核
- 基础审计可用
- 最小权限分层边界可用
- Agent 基础可观测性可用

### 8.3 数据门槛

- 历史互动、历史推文与任务结果可沉淀
- 最小状态快照可用
- 基础报表可生成

### 8.4 架构门槛

- AI、知识、执行通道没有被写死成一套组合
- 客户空间边界清楚
- 平台能力分层边界清楚
- 官方执行通道预留已存在
- 基础凭证与密钥治理边界已存在
- 基础限流、冷却与失败保护已存在
- 后续扩展能力路线已预留

## 10. 当前结论

`TweetPilot` 第一阶段的正确目标不是“尽量多做功能”，而是：

**围绕 Reply Agent，做出一个最小但完整的平台闭环。**

这个闭环必须同时证明：

- 平台能控制
- 数据能沉淀
- AI 能理解
- 任务能执行
- 风险能治理
- 结果能交付

只要这六件事成立，第一阶段就是成功的。
