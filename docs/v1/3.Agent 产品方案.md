# 3.Agent 产品方案

- 版本：`v1.1.0`
- 更新日期：`2026-04-09`
- 文档状态：`扩展能力纳入版`

## 1. 文档目的

本文档用于定义 `TweetPilot` 的 Agent 产品方向。

重点回答以下问题：

1. `TweetPilot` 为什么适合做 Agent
2. `TweetPilot` 的 Agent 最终形态应该是什么
3. 平台与 Agent 的关系是什么
4. Agent 应该拆成哪些子能力和子模块
5. 哪些内容必须第一期就做
6. 哪些内容可以分期推进

本文档不是技术实现细节文档，而是产品方案文档。

## 2. 一句话结论

`TweetPilot` 不应该只是一个“可调用 Twitter 扩展的桌面管理平台”，而应该逐步演化为：

**一个面向 Twitter 代运营团队的垂直领域 Agent Operating Platform。**

也就是说：

- 平台负责管理、配置、权限、报表、审计、调度
- Agent 负责观察、判断、生成、建议、执行、复盘

最终卖点不是“支持接大模型”，而是：

**帮助客户拥有一组可被管理、可被监督、可被复盘，并且能够持续长出新能力的 Twitter 数字员工与运营 Agent。**

这组客户不只包括代运营团队，也包括一人公司和极小创业团队。

对这类客户来说，`TweetPilot` 的意义不是“再多一个工具”，而是：

- 用 token 成本拥有一支能持续工作的 Twitter 运营 AI 团队
- 把宣传、互动、客服和基础增长工作从创始人手里接走
- 让创始人把精力重新集中到产品和业务本身

对大集团和大客户来说，`TweetPilot` 还应提供另一层价值：

- 在保留本地扩展能力的同时，支持走 X 官方 API 与官方 MCP 通道
- 用更稳定、更官方、更可审计的方式承载关键运营动作
- 让企业客户在稳定性优先的前提下使用 Agent 能力

## 3. 为什么 `TweetPilot` 适合做 Agent

一个真正可商用的 Agent，不只是“会聊天的模型”，而需要：

- 明确目标
- 可调用的工具
- 可持续的状态
- 可观测的执行过程
- 可审计的结果

`TweetPilot` 已经天然具备了这些条件：

### 3.1 已有真实工具层

当前产品已经有明确的工具能力基础：

- 浏览器扩展实例
- LocalBridge API
- Twitter 读写动作
- 本地模型、云端模型和外部 REST API 型 AI 的统一接入与管理能力
- X 官方 API 与官方 Agent 资源能力

这意味着 Agent 不是空转推理，而是能真正执行动作。

### 3.1A 官方 Agent 通道已具备现实基础

根据 X 官方 `Agent Resources` 页面，X 已提供面向 Agent 的资源：

- `XMCP`：把 `200+` 个 X API endpoint 暴露为可调用 MCP tools
- `Docs MCP`：让 Agent 能检索和读取 X 官方文档
- `OpenAPI Spec`：可用于代码生成和 Agent 工具接入

这意味着 `TweetPilot` 的 Agent 体系，不必只依赖自研本地执行链路，也可以接入官方 Agent 友好的能力层。

### 3.2 已有真实执行环境

我们不是停留在“生成文案”，而是可以：

- 读取评论
- 回复评论
- 发推文
- 点赞、转发、关注
- 切换和管理 Twitter tab
- 路由到不同扩展实例
- 调用 X 官方 API 或 XMCP 执行标准化动作

这让 Agent 天然具备“手和脚”。

### 3.3 已有真实业务闭环

代运营场景本身就适合 Agent：

- 评论需要持续监控
- 内容需要持续生产
- 任务需要持续执行
- 风险需要持续预警
- 报表需要持续生成

这些都不是一次性动作，而是长期运行型工作流。

### 3.4 已有明确客户价值

Agent 的价值必须能转化成客户愿意付费的结果。

在 `TweetPilot` 里，这个结果非常明确：

- 更快回复
- 更少人力
- 更高互动效率
- 更稳定的交付
- 更强的客户可见性
- 更好的风险识别与产品反馈洞察

### 3.5 具备接入客户专属知识的条件

一个可卖高价的垂直 Agent，不能只依赖通用模型。

`TweetPilot` 后续必须把 MCP 与客户专属知识源接进来，让 Agent 能访问：

- 产品资料
- 功能说明
- FAQ
- 售前话术
- 品牌边界
- 禁止承诺事项

这样形成三层壁垒：

- 基础模型是通用的
- Agent 工作流和执行策略是平台独有的
- 客户产品知识和业务事实是客户独有的

三者结合之后，Agent 的回复和宣传内容才会真正专业、准确且难以被复制。

### 3.6 具备承载 Agent 生成式能力的条件

`TweetPilot` 还有一个传统 Agent 产品通常不具备的条件：

- 平台拥有自己的本地数据层
- 平台拥有自己的状态层
- 平台拥有自己的任务系统
- 平台拥有自己的展示容器
- 平台拥有自己的执行能力网关

这意味着 Agent 不只可以“使用平台能力”，还可以在平台提供的受控边界中：

- 生成新的报表能力
- 生成新的分析视角
- 生成新的策略逻辑
- 生成新的定时任务能力
- 生成新的客户定制功能入口

这会让 `TweetPilot` 从“会执行的 Agent 平台”升级为“可持续生长的 Agent 平台”。

## 4. Agent 的最终产品形态

从长期看，`TweetPilot` 的最终形态不应是一个单一总 Agent，而应是：

**一个由多个垂直 Agent + 数字员工角色系统 + 任务编排系统 + 风险与报表系统 + 受控扩展能力层组成的平台。**

### 4.1 平台层、Agent 层与扩展层的关系

#### 平台层负责

- 账号管理
- 扩展与实例管理
- 官方 API / 官方 MCP 通道管理
- 标签页状态管理
- 数据与状态沉淀
- 任务创建与调度
- AI 提供者配置
- 报表
- 风控
- 审计
- 团队权限
- 展示宿主
- 扩展能力挂载控制

#### Agent 层负责

- 持续观察数据与状态
- 识别机会或异常
- 生成候选内容
- 提出执行建议
- 在授权范围内执行动作
- 产出总结和复盘

#### 扩展层负责

- 承载 Agent 生成的新报表
- 承载 Agent 生成的新分析能力
- 承载 Agent 生成的新策略逻辑
- 承载 Agent 生成的新定时任务
- 承载 Agent 生成的新功能入口

这一层必须运行在受控边界中，不能直接侵入平台核心。

它与平台核心之间的关系应保持清晰：

- 平台核心是稳定宿主
- 扩展层是受控扩展区
- 扩展层对平台核心数据应以只读访问为主
- 是否允许扩展能力被长期挂载、自动运行或进入主界面，最终由平台决定

### 4.2 最终不应该只有一个 Agent

我们不建议做“一个万能 Agent”。

正确方向是做多个垂直 Agent，每个 Agent 解决一类清晰、高频、可量化的问题。

建议的长期 Agent 体系如下。

#### 1. Reply Agent

职责：

- 监控评论和提及
- 识别待回复内容
- 生成候选回复
- 自动或半自动发送
- 记录结果与质量
- 结合客户知识源生成更专业的互动内容

#### 2. Content Agent

职责：

- 根据素材和账号人设生成内容
- 生成发帖草稿
- 协同生成文案、图片、视频等发布素材
- 给出多个版本建议
- 组织、复用和管理待发布素材版本
- 参与排期和发布流程
- 结合知识库输出更精准的产品宣传内容

#### 3. Growth Agent

职责：

- 发现值得互动的人群和账号
- 跟踪关注、被关注、互动对象
- 生成增长任务建议
- 驱动互动和关注计划

#### 4. Risk Agent

职责：

- 识别负面评论与高风险互动
- 聚类投诉主题
- 输出风险预警
- 生成 AI 产品反馈与改进建议报告

#### 5. Account Health Agent

职责：

- 检查账号健康状态
- 检查实例在线状态
- 检查标签页状态
- 检查任务失败和异常
- 输出巡检与修复建议

#### 6. Delivery Agent

职责：

- 自动整理客户周报 / 月报
- 汇总任务执行结果
- 输出客户可读成果说明

### 4.3 数字员工与 Agent 的关系

数字员工不是 Agent 的替代品，而是 Agent 的人格层和执行风格层。

可以这样理解：

- Agent 是“工作职责”
- 数字员工是“表达方式和行为风格”

例如：

- `Reply Agent` 负责回复任务
- 但它可以调用不同数字员工角色生成不同风格的回复：
  - 品牌客服型
  - 创始人口吻型
  - 社群互动型
  - 危机公关型

所以最终系统不是：

- 一个大模型

而是：

- 多个 Agent
- 多个角色
- 多种 AI 提供者
- 多种执行策略
- 多种可持续扩展能力

### 4.4 Agent 的执行不应绑定单一通道

`TweetPilot` 的 Agent 不应被设计成“只能通过浏览器扩展工作”。

长期看，Agent 应至少支持两类执行通道：

#### 本地执行通道

- `LocalBridge`
- 浏览器扩展实例
- Twitter tab 上下文

适合：

- 强依赖浏览器上下文的动作
- 已有自研扩展能力可覆盖的动作
- 对本地可控性要求更高的场景

#### 官方执行通道

- X 官方 REST API
- X 官方 `XMCP`

适合：

- 企业级稳定性优先场景
- 大客户和大集团
- 需要更标准化、更可审计、更官方的执行路径

正确产品形态不是二选一，而是：

**统一 Agent 编排层 + 多执行通道。**

### 4.5 Agent 不只执行任务，也应生成扩展能力

传统 Agent 平台通常到“完成任务”就结束了。

`TweetPilot` 的长期方向不应止步于此。

在我们的体系里，Agent 除了可以执行具体运营任务，还应在平台授权范围内生成可持续能力，例如：

- 面向客户特定需求的报表
- 面向客户特定业务的分析视角
- 围绕特定账号或特定人群的策略逻辑
- 围绕特定主题的定时任务
- 围绕特定目标的功能入口

这些能力不是一次性回答，而是平台内可保留、可挂载、可复用、可接管的长期资产。

### 4.6 扩展能力必须运行在受控沙箱中

如果 Agent 生成的新能力直接改写平台核心，那么平台会失去稳定性、边界和可审计性。

因此正确形态不是“Agent 直接改主程序”，而是：

- 平台核心保持稳定
- Agent 生成新能力
- 新能力运行在受控沙箱中
- 平台决定是否挂载、启停、替换和长期保留
- 沙箱优先读取平台核心事实和状态，而不是直接改写它们

这不是实现细节，而是产品边界本身。

## 5. Agent 的核心组成

为了真正把 Agent 做成产品，而不是宣传概念，Agent 至少要由以下能力组成。

### 5.1 观察层

负责持续感知：

- 账号状态
- 实例状态
- 官方 API / XMCP 通道状态
- tab 状态
- 评论 / 提及 / 时间线
- 任务状态
- AI 调用结果
- 风险信号

### 5.2 记忆层

负责沉淀长期上下文：

- 账号历史
- 品牌语气
- 角色设定
- 过去互动
- 高价值用户
- 已处理记录
- 风险主题

### 5.3 知识层

负责把客户专属知识接入到 Agent 执行链路中：

- MCP 服务接入
- FAQ 与产品资料
- 品牌规范
- 宣传素材说明
- 敏感话题边界
- 知识命中与引用追踪

这层能力决定 Agent 是“通用回复器”还是“真正懂客户业务的数字员工”。

### 5.4 数据与状态层

负责把平台长期沉淀下来的运营资产组织成可被 Agent 使用的输入层：

- 历史推文
- 历史互动
- 周期性表现快照
- 风险主题
- 内容表现结果
- 执行通道表现
- 角色与 Agent 表现结果

这层能力决定 Agent 是否每次都从原始数据重新推理，还是能够站在已有运营状态之上快速分析。

### 5.5 决策层

负责决定：

- 是否需要回复
- 是否需要人工审核
- 使用哪个 AI 提供者
- 使用哪个数字员工角色
- 使用哪个知识源或 MCP 范围
- 使用哪个执行通道
- 路由到哪个实例和账号

### 5.6 扩展生成层

负责在授权范围内把 Agent 的高级产出沉淀成可持续能力：

- 生成新报表
- 生成新分析视角
- 生成新策略逻辑
- 生成新定时任务
- 生成新功能入口

这层能力的意义在于让 Agent 的结果不只停留在一次性会话里，而能被平台长期承载。

### 5.7 执行层

负责调用实际工具：

- LocalBridge API
- 浏览器扩展实例
- X 官方 REST API
- X 官方 XMCP
- 模型 API
- 外部 REST API

### 5.8 沙箱承载层

负责为 Agent 生成的新能力提供受控运行边界：

- 扩展能力运行
- 扩展能力隔离
- 与平台核心的边界控制
- 对平台核心数据的受控读取
- 扩展能力的启停与替换

这层能力的存在，是为了保证 Agent 可以生成新能力，但不会破坏平台核心。

### 5.9 审核层

负责高风险动作的人机协同：

- 哪些动作必须人工审核
- 哪些动作可自动执行
- 哪些账号启用更严格策略

### 5.10 观测与复盘层

负责输出：

- 执行日志
- 采用率
- 成功率
- 风险命中情况
- 报表和复盘总结
- MCP / 知识命中记录
- 执行通道命中记录
- 扩展能力生成、挂载和运行结果

## 6. 第一性原则：第一期必须做什么

虽然最终形态可以很大，但第一期必须极度克制。

第一期必须满足三个标准：

1. 最有价值
2. 最不可缺失
3. 最容易形成闭环并卖出去

按照这个标准，**第一期不应该做“大而全 Agent 平台”**。

第一期应该只做：

## 6.1 第一优先级：Reply Agent

这是整个 Agent 方向中最重要、最不可缺失、最有商业价值的部分。

原因：

- 评论回复是高频刚需
- 对团队人效提升最直接
- 最容易形成“AI + 扩展 + 审核 + 报表”的闭环
- 最容易被客户感知到价值
- 最容易切入收费和交付

### 第一阶段 Reply Agent 的最小闭环

1. 监控某账号的评论或提及
2. 发现新的待处理回复
3. 结合账号角色和客户知识源调用 AI 生成候选回复
4. 按数字员工角色输出不同版本
5. 根据账号策略选择本地扩展通道或官方执行通道
6. 人工审核或低风险自动发送
7. 记录结果
8. 汇总到报表

这条链路跑通，产品就已经从工具升级到了“最小可售卖 Agent”。

## 6.2 第一优先级：数字员工角色基础能力

如果只有 Reply Agent，而没有角色系统，那么客户感知到的仍然只是“自动回复工具”。

所以第一期必须包含最小角色系统。

最小角色系统只需要支持：

- 账号绑定一个默认角色
- 角色包含：
  - 名称
  - 语气
  - 人设
  - 禁止事项
- 生成回复时可切换角色

第一期不需要做非常复杂的角色管理市场，但必须把“数字员工”这个价值点立起来。

## 6.3 第一优先级：最小 MCP 知识能力

如果第一期完全不接入客户知识，Agent 很容易停留在“会说话但不懂业务”。

第一期不需要做复杂知识工程，但至少要支持：

- 为客户绑定一个可用知识源或 MCP 入口
- 为账号或角色绑定默认知识范围
- 在回复生成时把知识源纳入上下文
- 对知识命中结果做基础记录

这样第一期交付的就不是“泛化回复能力”，而是“带客户业务理解的回复能力”。

## 6.4 第一优先级：企业级执行通道预留

第一期即使不完整接入全部官方能力，也必须从产品结构上预留：

- 官方 API 通道
- 官方 XMCP 通道
- 通道路由位
- 通道状态观测位
- 通道审计字段

否则后面接企业客户时，架构会被迫重做。

## 6.5 第一优先级：扩展能力边界预留

第一期即使不完整开放 Agent 生成新功能，也必须从产品结构上预留：

- 平台核心与扩展能力的边界
- 扩展能力的受控承载位置
- 扩展能力的挂载与启停控制
- 扩展能力的可审计位

否则后面平台会被迫在核心逻辑中硬塞客户定制能力，失去稳定性。

## 6.6 第一优先级：风险识别基础能力

如果第一期完全不做风险识别，产品就还是停留在“执行工具”。

第一期至少应该做最小版本的风险能力：

- 标记高风险评论
- 标记负面反馈
- 把高风险评论提升到人工审核
- 对评论主题做简单聚类
- 输出基础风险摘要

这能直接把产品价值抬高到“运营 + 风险感知”。

## 6.7 第一优先级：Agent 可观测性

第一期即使功能少，也必须有可观测性。

至少要让用户知道：

- Agent 看到了什么
- Agent 为何这样判断
- Agent 调用了哪个 AI
- Agent 是否调用了扩展还是官方通道
- Agent 最终执行结果是什么
- Agent 是否生成了新的扩展能力，以及该能力的状态是什么

没有这个能力，Agent 就不可控，也很难卖给团队客户。

## 7. 第一阶段明确不做什么

为了保证第一期能打穿，不建议一开始就做：

- 通用大而全总 Agent
- 多 Agent 协同编排
- 自动完成所有运营动作
- 无边界自由扩展平台
- 高复杂度长期记忆系统
- 全量风控中心
- 完整交付门户
- 复杂多租户权限

这些都很有想象力，但不适合第一期。

## 8. 分期路线图

## 8.1 第一阶段：Reply Agent MVP

目标：

- 做出第一个可卖的 Agent 闭环

包含：

- Reply Agent
- 最小数字员工角色
- 最小 MCP 知识能力
- 企业级执行通道预留
- 扩展能力边界预留
- 人工审核流
- 风险识别基础能力
- AI 调用进展展示
- 回复结果记录
- 基础报表

第一阶段的销售话术应是：

**“让一个运营团队拥有一位可以 24/7 监控评论、生成回复并辅助互动的 Twitter 数字员工。”**

对一人公司的表达应是：

**“让创始人用 token 成本拥有一位 24/7 在线的 Twitter 运营与客服 AI 员工。”**

## 8.2 第二阶段：多 Agent 扩展

目标：

- 从单点回复，扩展到内容与增长

包含：

- Content Agent
- Growth Agent
- 受控报表扩展能力
- 受控分析扩展能力
- MCP 知识接入增强
- 官方 API / XMCP 通道增强
- 风险预警增强
- 产品反馈洞察增强
- 定时任务结合 Agent
- 多角色体系增强

第二阶段的价值会从“评论回复”扩展到“内容生产 + 增长互动”。

## 8.3 第三阶段：Agent Operating Platform

目标：

- 从“若干 Agent 功能”升级为“完整 Agent 平台”

包含：

- 多 Agent 协同
- Agent 生成式扩展能力体系
- Agent 策略中心
- Agent 权限中心
- Agent 审计中心
- Agent 评估体系
- 企业级风控与报表
- 客户交付与汇报自动化

## 9. 商业价值与价格提升逻辑

Agent 方向最大的意义，不只是技术先进，而是能把价格带抬高。

### 9.1 如果只是平台

客户理解为：

- 账号管理工具
- 自动化执行工具
- AI 接口接入工具

这类产品通常会被拿去和各种普通 SaaS 工具比较价格。

### 9.2 如果是垂直 Agent 平台

客户理解为：

- 我拥有了一组可工作的数字员工
- 我不仅能执行动作，还能持续感知和优化
- 平台会帮我减少人力、缩短响应时间、识别风险、产出报告
- 平台还能结合我的专属知识输出更专业、更可信的内容和回复
- 平台还能按客户需求选择本地扩展链路或 X 官方执行链路
- 平台还能在受控边界内承接客户高级定制需求，而不必每次都回到研发排期

这时定价逻辑就可以从“工具订阅”升级成：

- 软件订阅
- 数字员工席位
- Agent 模块包
- 运营增强服务
- 企业交付服务

### 9.3 长期最强卖点

长期最强的外部价值表达不是：

- 我们支持很多模型
- 我们支持很多 API

而是：

**“TweetPilot 让代运营团队、一人公司和企业客户拥有一组可管理、可审计、结合私有知识并支持官方执行通道的 AI 数字员工和垂直运营 Agent。”**

## 10. 当前结论

结论很明确：

1. `TweetPilot` 非常适合做 Agent 化升级
2. 但不应抛弃平台，而应做成“平台 + Agent 系统”
3. 第一阶段最重要、最不可缺失、最有价值的部分是：
   - `Reply Agent`
   - `数字员工角色基础能力`
   - `基础风险识别能力`
   - `Agent 可观测性`
4. 只有先打穿这个最小闭环，后续的 Content Agent、Growth Agent、Risk Agent、Delivery Agent 才有意义

因此，产品执行策略应该是：

**先做最小可售卖 Agent，再逐步做成完整的垂直 Agent Operating Platform。**
