# 8.第一版沙箱开放能力清单

- 版本：`v0.1.0`
- 更新日期：`2026-04-09`
- 文档状态：`首版落地清单`
- 文档定位：本文档用于定义 `TweetPilot` 第一版沙箱的实际开放能力范围。本文档回答“第一版沙箱现在应该开放什么，不应该开放什么”，不讨论底层实现细节和具体开发步骤。

## 1. 文档目的

`7.沙箱能力边界与开放策略.md` 已经定义了完整版沙箱的能力边界。  
本文件只解决一个更现实的问题：

**第一版沙箱，到底开放哪些能力才既有价值，又不过度冒险。**

第一版的目标不是证明“沙箱什么都能做”，而是证明：

- Agent 生成的扩展能力可以真正落地
- 平台核心不会因此失控
- 客户可以从中获得实际价值

## 2. 第一版沙箱的目标

第一版沙箱应只聚焦一类能力：

**报表扩展和分析扩展。**

也就是说，第一版的沙箱不是为了自动发帖、自动回复、自动互动而设计，  
而是为了让 Agent 能在平台已有数据和知识之上，生成：

- 客户专属报表
- 客户专属分析视角
- 客户专属风险洞察页
- 客户专属内容效果页

第一版沙箱的价值在于：

- 低风险
- 高业务价值
- 易于演示
- 易于交付
- 易于积累方法论

## 3. 第一版沙箱的三条原则

### 3.1 只读平台核心

第一版沙箱默认只读平台事实层和状态层，不直接写入平台核心数据。

### 3.2 只写沙箱派生层

第一版沙箱允许写入自己的派生数据空间，用来保存分析结果、报表结果和刷新结果。

### 3.3 只做受控展示

第一版沙箱可以生成展示结果，但是否挂载到菜单、是否长期保留、是否对客户可见，全部由平台决定。

## 4. 第一版默认开放的能力

以下能力应作为第一版沙箱的默认开放项。

### 4.1 客户空间内的只读数据访问

允许沙箱在所属客户空间内只读访问：

- 历史推文
- 历史互动
- 账号状态
- 任务状态
- 通道状态
- 内容表现快照
- 风险主题结果
- 角色与 Agent 表现结果

这类能力是第一版沙箱最基础的输入来源。

### 4.2 客户知识与 MCP 只读访问

允许沙箱在授权范围内只读访问：

- 客户知识库
- FAQ
- 产品资料
- 品牌边界
- 宣传素材说明
- 指定 MCP 输出结果

第一版就应该支持这一点，否则报表和分析不会具备客户特异性。

### 4.3 通过平台网关调用模型

允许沙箱通过平台统一网关调用：

- 本地模型
- 云端模型
- 第三方 REST API 型 AI

但第一版沙箱不直接持有模型密钥，也不自行管理模型路由。

### 4.4 写入沙箱派生数据空间

允许沙箱在自己的边界内：

- 保存分析结果
- 保存报表刷新结果
- 保存扩展能力需要的派生数据
- 保存扩展能力自己的运行状态

第一版可以把这部分能力理解为：

**可以写自己的结果，不可以写平台核心。**

### 4.5 生成报表和分析页面

允许沙箱生成：

- 报表页面
- 分析页面
- 自定义视图
- 风险洞察页
- 内容表现页

这些页面应运行在平台受控展示容器中。

### 4.6 手动刷新和受控刷新入口

允许第一版扩展能力具备：

- 手动触发
- 手动刷新
- 平台控制下的刷新入口

这类能力足以支撑第一版报表和分析能力，不需要一开始就开放复杂自动调度。

### 4.7 扩展能力挂载与说明

允许第一版扩展能力具备：

- 菜单入口
- 页面入口
- 说明信息
- 上下文摘要
- 可接管说明

这是保证扩展能力能够长期保留和被后续 AI 接管的基础。

## 5. 第一版受限开放的能力

以下能力第一版可以预留，但不应默认开放。

### 5.1 同客户空间内的多账号聚合分析

这是有价值的，但第一版应在客户空间边界、权限边界和资源边界明确后再开放。

### 5.2 定时刷新

定时刷新对报表类扩展有价值，  
但第一版建议只做“平台可控的有限刷新”，而不是开放通用长期后台任务。

### 5.3 平台开放的只读 Twitter / X 实时读取能力

第一版应优先基于平台已经沉淀的本地数据工作。  
如果需要实时读取 Twitter / X 数据，也应作为受限能力逐步开放，而不是默认开放。

### 5.4 条件触发型刷新

例如：

- 某类数据更新后自动刷新
- 某类风险出现后自动刷新

这类能力很有价值，但第一版不应作为默认能力。

## 6. 第一版默认不开放的能力

以下能力第一版不应开放。

### 6.1 任意写 Twitter / X 动作

第一版沙箱不应直接开放：

- 自动发帖
- 自动回复
- 自动点赞
- 自动关注

因为这些能力一旦和 Agent 生成逻辑直接结合，风险会明显上升。

### 6.2 平台核心数据写入

第一版沙箱不应写入：

- 平台事实层
- 平台状态层
- 平台核心配置
- 平台核心报表定义

### 6.3 原始凭证读取

第一版沙箱不应读取：

- X API 原始密钥
- 模型原始密钥
- MCP 原始密钥
- 其他平台级 Secrets

### 6.4 任意系统能力

第一版沙箱不应开放：

- 任意系统命令执行
- 任意文件系统读写
- 任意网络访问
- 任意安装依赖

### 6.5 跨客户空间访问

第一版沙箱不应默认获得跨客户空间的数据、知识、报表或扩展能力访问权。

### 6.6 自动运行型高风险扩展

第一版不应允许扩展能力默认：

- 自动长期运行
- 自动跨账号动作
- 自动触发高风险任务

## 7. 第一版推荐承接的扩展能力类型

第一版最适合承接的扩展能力类型有四类：

### 7.1 客户专属运营报表

例如：

- 某客户的周报页面
- 某客户的账号对比页
- 某客户的内容表现汇总页

### 7.2 趋势分析扩展

例如：

- 涨粉趋势变化
- 内容表现变化
- 互动效率变化

### 7.3 风险洞察扩展

例如：

- 投诉主题页
- 品牌风险页
- 高风险评论监控页

### 7.4 内容复盘扩展

例如：

- 不同主题推文表现比较
- 不同角色文案采用率分析
- 素材类型表现分析

## 8. 第一版的平台控制关系

第一版必须明确：

### 平台决定

- 哪些客户空间允许启用沙箱
- 哪些数据和知识对沙箱可见
- 哪些扩展能力可挂载
- 哪些扩展能力可刷新
- 哪些扩展能力可长期保留

### 沙箱决定

- 如何组合已开放的数据、知识和模型能力
- 如何组织自己的派生数据
- 如何生成报表和分析结果

因此，第一版仍然遵守同一条原则：

**平台决定开放什么，沙箱决定如何组合什么。**

## 9. 当前结论

`TweetPilot` 第一版沙箱最合理的落地方式不是：

- 写任意代码
- 做任意自动化
- 直接控制 Twitter / X 账号

而是：

**先把沙箱做成一个面向报表扩展和分析扩展的受控运行边界。**

这条路径的优点是：

- 风险最低
- 价值直接
- 最容易交付
- 最容易形成客户定制能力
- 最容易验证“Agent 生成扩展能力”这条路线本身成立

如果第一版这条路径打通，后续再逐步开放：

- 更强的调度能力
- 更强的只读外部能力
- 更强的受控写能力

才是合理顺序。
