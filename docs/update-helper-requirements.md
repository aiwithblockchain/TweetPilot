# update-helper 需求说明（草案）

## 1. 背景

当前项目处于开发早期阶段，本地数据库数据不是关键资产。

在 AI scheduled task session 这次改造中，我们已经明确：

- 当前阶段**不要求为旧数据库设计迁移兼容逻辑**
- 如果数据库结构变化，可以接受直接删除当前工作区数据库并重建
- 这样可以避免为了早期过渡而引入大量只使用一次、后续会失效的迁移代码

也就是说：

- **当前 AI task session 功能实现，不以 migration 体系为前提**
- **当前数据库可以按“空库重建”思路处理**
- 现阶段重点是把任务 AI session 主能力做对，而不是先做数据库升级框架

但是在讨论过程中，也暴露出一个后续必然会面对的问题：

随着功能继续增加，系统未来不只会有数据库结构变化，还可能出现其它升级场景，例如：

- 数据库 schema 升级
- 本地文件位置迁移
- 工作区内部目录结构调整
- 某些缓存、配置或持久化数据的重建
- 第三方库升级后引发的数据修复或转换

因此，后续有必要单独设计一个更专业的升级模块，暂定名为 **update-helper**。

---

## 2. 当前结论

本轮对话已经明确以下结论：

### 2.1 对当前 AI task session 开发的结论

- 不把“数据库迁移系统”纳入当前这轮实现范围
- 不要求为了这次 task session 改造专门补数据库版本升级逻辑
- 当前文档中如果存在“为了兼容旧数据库而设计迁移”的倾向，应移除
- 当前默认前提为：**开发阶段数据库可删除、可重建、可重新初始化**

### 2.2 对后续系统演进的结论

虽然这次不做 migration，但未来应该补一个**独立、专业、可扩展的 update-helper 模块**，用于统一处理升级类需求，而不是把升级逻辑零散塞进各个业务模块中。

---

## 3. update-helper 的目标

update-helper 的职责不是只处理数据库。

它未来应该是一个更通用的“本地数据与运行环境升级管理模块”，统一处理版本升级过程中可能涉及的各种变更。

目标包括：

1. 统一管理升级入口
2. 根据版本号决定需要执行哪些升级动作
3. 允许数据库升级与非数据库升级并存
4. 避免把升级逻辑散落在 task、AI、workspace 等业务代码里
5. 为后续正式发布版本提供稳定的升级基础

---

## 4. 未来可能纳入 update-helper 的升级类型

以下内容是需求方向，不是这次要实现的方案。

### 4.1 数据库升级

包括但不限于：

- 新增表
- 修改表结构
- 数据修复
- 数据重建
- 索引调整
- 不同工作区数据库的版本升级

### 4.2 文件与目录升级

包括但不限于：

- 文件移动
- 目录重命名
- 工作区内部存储结构调整
- 历史文件兼容清理
- 临时文件或缓存重建

### 4.3 配置与本地状态升级

包括但不限于：

- 配置文件字段升级
- 默认配置重写
- 旧配置映射到新配置
- 本地状态恢复策略调整

### 4.4 依赖或内部模块升级后的辅助处理

包括但不限于：

- 因库升级导致的数据格式调整
- 因内部存储方式变化导致的本地资源修复
- 因运行时能力调整导致的辅助初始化逻辑

---

## 5. 为什么现在先写需求、不写方案

当前讨论的重点仍然是 AI scheduled task session 本身。

如果现在同时展开 update-helper 的完整架构设计，会把对话目标拉散，影响当前主线开发。

因此当前阶段只记录：

- 为什么需要 update-helper
- 它未来解决什么问题
- 它不是当前 task session 功能的前置条件
- 后续应单独开一个话题来设计它

也就是说，这份文档是**需求记录**，不是实现方案，也不是技术设计稿。

---

这份文档与 `ai-scheduled-task-session-refactor-brief.md` 的关系如下：

- `ai-scheduled-task-session-refactor-brief.md`：描述当前 AI scheduled task session 的设计与实现范围
- 本文档：记录为什么**当前不做 migration**，以及为什么**未来要单独建设 update-helper**

两者边界应保持清晰：

- task session 文档聚焦本次功能
- update-helper 文档聚焦未来升级体系需求

---

## 7. 本文档确认的需求边界

当前明确：

- **不在这次对话中展开 update-helper 的方案设计**
- **不在这次 task session 开发中引入完整升级框架**
- **当前数据库按开发期空库重建思路处理**
- **未来需要单独设计 update-helper，并覆盖数据库升级以外的升级能力**

---

## 8. 结论

本轮讨论后的统一结论是：

1. 当前项目仍处于开发早期，可接受删除并重建本地数据库。
2. 当前 AI scheduled task session 改造不需要为了兼容旧数据库而引入迁移逻辑。
3. 当前应把精力放在 task session 主能力落地，而不是提前做数据库升级系统。
4. 未来应单独设计一个专业的 **update-helper**，统一负责数据库、文件、配置及其它本地升级动作。
5. update-helper 是后续系统演进的重要基础设施，但**不属于本次 task session 功能的实现范围**。
