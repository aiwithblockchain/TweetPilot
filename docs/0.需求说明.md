## 1. 一句话定义

`TweetPilot` 是一个让企业按 token 雇佣一支会成长、24 小时在线的 Twitter 运营 AI 团队的平台。

### Slogan

拥有 `TweetPilot`，你就拥有一支会成长的 Twitter 运营 AI 团队。

## 2. 产品目标

`TweetPilot` 要解决的不是“如何再做一个 Twitter 工具”，而是“如何让不同规模的客户以更低的人力成本、更高的稳定性和更强的 AI 协同能力，完成 Twitter 的宣传、互动、客服、风险识别与运营交付”。

产品目标分为八层。

### 2.1 提高团队人效

- 让一个运营团队可同时管理更多账号
- 降低重复性操作的人力消耗
- 把高频互动工作从人工逐条处理升级为 AI 协同处理

### 2.2 提高互动质量与一致性

- 缩短评论回复时延
- 保持不同账号的风格一致性
- 让内容与互动更连续、更系统
- 让同一套运营目标可以由不同数字员工角色以不同风格完成

### 2.3 提高稳定性、可控性与可审计性

- 清楚知道当前在线账号、在线扩展、可控 tab 和正在执行的任务
- 清楚知道每次动作由谁、通过哪个实例、用哪个 AI 完成
- 清楚知道一次动作走的是本地扩展执行通道，还是 X 官方 API / 官方 MCP 执行通道
- 对高风险动作保留人工审核、追踪和回放能力

### 2.4 解放一人公司与创始人团队

- 让一人公司把时间集中在产品、销售和融资等高价值事项上
- 把 Twitter 的宣传、互动、客服和基础运营交给 AI 团队持续处理
- 用 token 成本替代部分人工成本，让小团队也能拥有接近运营团队的执行能力
- 让创始人只介入高风险、高价值和高判断难度事项

### 2.5 建立本地运营数据资产与长期状态能力

- 把推文、互动、任务、风险、账号变化等历史信息持续沉淀到本地
- 不只保存原始记录，还要保存具有运营意义的结构化状态和趋势快照
- 让 Agent、报表和管理洞察优先建立在本地历史数据之上，而不是每次都从零抓取和临时推理
- 让系统每天运行都会积累新的运营资产，而不是只完成一次性动作

### 2.6 建立客户专属知识驱动的专业能力

- 通过 MCP 与知识库能力，让 AI 读取每个客户独有的产品资料、FAQ、功能说明和品牌边界
- 让通用模型在客户私有知识之上生成更专业、更准确、更可执行的回复和宣传内容
- 把“通用模型 + 平台独有工作流 + 客户私有数据”沉淀成平台核心壁垒

### 2.7 建立可持续生长的客户定制能力

- 平台提供基础数据能力、基础统计能力、基础 Twitter 能力和基础展示容器
- Agent 在受控边界内基于这些基础能力生成新的扩展能力，例如报表、分析、策略和小功能
- 高级客户需求不应默认回到平台主程序发版，而应优先由 Agent 生成的扩展能力承接
- Agent 生成的扩展能力应可保留、可挂载、可复用、可接管，而不是一次性对话结果

### 2.8 提高交付与商业价值

- 把运营动作沉淀成报表、周报、月报和客户可读成果
- 从用户互动中提炼风险预警和产品反馈洞察
- 让产品既能卖软件，也能卖交付能力、运营方法论和企业服务

## 3. 目标用户

## 3.1 核心客户

### A. Twitter / X 代运营公司

这是最核心的目标客户。

特点：

- 同时管理多个客户账号
- 有团队协作和客户交付压力
- 需要报表、风险控制、任务编排和 AI 增效

### B. Web3 / AI / SaaS 增长团队

特点：

- 对 Twitter 高依赖
- 重视增长互动、品牌传播和高频内容运营
- 愿意尝试 AI 自动化与多账号编排

### C. 一人公司 / 创始人品牌团队

特点：

- 团队极小，通常由创始人自己兼任产品、增长、客服和内容工作
- 对 Twitter 的宣传和客服有真实需求，但没有预算长期养一支运营团队
- 更关注响应速度、专业度、稳定性和投入产出比
- 需要把有限时间投入到产品研发和业务增长，而不是被日常社媒运营占满

### D. 大集团 / 大客户 / 企业品牌团队

特点：

- 对稳定性、合规性、可审计性和长期可维护性要求更高
- 相比成本，更在意官方接入路径、故障边界和服务等级
- 更适合采用 X 官方 API 与官方 MCP 能力作为主执行通道
- 对权限、风控、报表、审计和服务保障有更高要求

## 4. 核心业务场景

`TweetPilot` 需要围绕真实高频工作流设计，而不是围绕孤立功能设计。

### 4.1 评论回复与客服运营

场景：

- 某账号发布推文后收到大量评论或提及
- 团队需要快速处理、统一语气、避免漏回
- 一人公司需要把基础客服和互动交给 AI 团队持续处理

产品需要支持：

- 发现新评论和新提及
- 读取上下文
- 调用 AI 生成候选回复
- 按角色输出不同风格版本
- 审核后发送或自动发送
- 记录执行结果并纳入报表

这是第一阶段最重要的场景。

### 4.2 内容生产、素材管理与定时发帖

场景：

- 团队提前准备好文案、图片、视频素材
- 需要按时间和账号批量发布
- 需要结合客户知识库生成更专业的宣传内容

产品需要支持：

- 管理文案和素材
- AI 辅助生成或改写内容
- 定时任务与批量发布
- 发布结果跟踪

### 4.3 基于历史数据的趋势分析与运营报表

场景：

- 平台长期保存了账号历史、推文历史、每日阅读量、点赞量、回复量、转发量等数据
- 团队不希望每次都从原始数据重新计算运营结论
- 管理层希望快速看到“增长在变好还是变差”“哪类内容有效”“哪些账号需要干预”

产品需要支持：

- 保存历史推文与互动记录
- 保存推文的周期性表现快照
- 保存账号增长和互动趋势快照
- 基于本地历史数据快速生成趋势结论和多维度报表
- 让 Agent 能直接基于这些历史状态做分析和建议

### 4.4 增长互动与关系管理

场景：

- 团队需要查看某账号关注了谁、被谁关注
- 需要展示某个账号下的关注列表与粉丝列表
- 需要对目标用户发起关注、点赞、留言、回复等增长任务

产品需要支持：

- 管理目标人群
- 管理关注关系与粉丝关系
- 针对目标人群执行互动任务
- 支持留言、点赞、关注等定时或批量任务
- 把关系管理升级为轻量增长 CRM

### 4.5 多账号、多实例与执行通道健康巡检

场景：

- 团队同时管理多个账号和多个扩展实例
- 需要知道哪些账号在线、哪些实例异常、哪些 tab 可控
- 企业客户需要知道官方 API / 官方 MCP 通道是否可用

产品需要支持：

- 账号状态总览
- 实例状态总览
- 标签页状态总览
- 官方 API / 官方 MCP 通道状态总览
- 异常检测、告警与提示

### 4.6 风险预警、产品反馈与管理洞察

场景：

- 用户评论与互动中包含大量产品反馈、抱怨和品牌风险信号
- 管理层需要及时知道风险并看到改进建议

产品需要支持：

- 提炼负面反馈
- 聚类投诉主题
- 输出风险预警
- 生成 AI 产品反馈报告
- 给出产品改进方向建议

### 4.7 客户交付与复盘

场景：

- 代运营公司需要定期向客户展示“做了什么、效果如何、风险如何、接下来做什么”

产品需要支持：

- 运营报表
- 客户周报 / 月报
- 任务执行总结
- 风险与洞察输出

### 4.8 企业级稳定运营通道

场景：

- 大客户不希望核心运营能力完全依赖本地浏览器扩展和桌面环境
- 更希望通过 X 官方提供的 API 与官方 Agent 友好接入方式完成关键动作

产品需要支持：

- 同时支持本地 `LocalBridge + 浏览器扩展` 通道与 X 官方 API / 官方 MCP 通道
- 按客户、账号、任务类型配置默认执行通道
- 在稳定性优先场景下优先使用官方执行通道
- 对两类执行通道的状态、限额、失败原因和回退策略做统一观测

### 4.9 客户定制报表与功能生成

场景：

- 平台提供了基础数据保存、基础统计和基础报表，但高级客户经常提出更细分、更独特的分析与功能需求
- 这些需求未必值得每次都升级平台主程序，但又确实具备长期业务价值
- 客户希望在已有数据和已有能力之上快速得到专属分析、专属报表和专属小功能

产品需要支持：

- Agent 基于平台沉淀的历史数据和基础能力生成新的扩展能力，例如报表、分析视角、策略逻辑和小功能
- 扩展能力运行在受控边界中，不直接改写平台核心
- 平台可以决定哪些扩展能力被保留、挂载到菜单、定时运行或交付给客户长期使用
- 扩展能力应具备可理解、可复用、可被其他模型继续接管的上下文说明
- 扩展能力应建立在平台已开放的数据、知识、模型、执行通道和展示容器之上，而不是以无边界系统权限为前提

### 4.10 团队协作、审核与责任接管

场景：

- 代运营团队不是单人操作，而是多人协作处理回复、发帖、风险事件和客户交付
- 某些任务需要由运营专员初审、账号经理复审、主管批准后再发送
- 当某个成员超时未处理、高风险任务升级或某个账号突然出现异常时，团队需要快速接管

产品需要支持：

- 任务指派、认领、转交与接管
- 审核队列与不同角色的审批权限
- 风险事件升级与责任人明确
- 团队协作过程可追踪、可审计、可复盘

### 4.11 客户空间隔离与企业治理

场景：

- 代运营公司通常同时服务多个客户，每个客户都有自己的账号、知识、素材、角色和报表
- 企业客户会要求凭证、知识、素材、报表和扩展能力都严格隔离
- 平台需要在同一个组织下支撑多个客户空间，并且清楚界定谁能看、谁能配、谁能执行

产品需要支持：

- 以客户空间为边界管理账号、知识、素材、报表和扩展能力
- 组织级、客户级、账号级权限分层
- 凭证、知识源和执行通道的可见范围与使用范围隔离
- 面向大客户的稳定治理和审计能力

## 5. 产品完整边界

这一章描述的是产品完整边界，而不是某一期边界。

- 哪个动作自动执行，哪个经过人工审核
- 哪个扩展能力由哪个 Agent 生成、启用、替换或停用
- 哪个实例、账号或通道出现异常

### 5.5 平台能力分层

为了让平台长期保持“可持续生长”而不是“边界失控”，平台中的能力至少应分为三类：

- 固有能力：由平台核心长期提供和维护，例如账号管理、数据沉淀、执行通道、内容与素材管理、模型管理、基础报表、调度、权限与审计
- 编排能力：由 Agent 和 AI 在平台边界内完成的理解、组合、判断和执行能力，例如模型路由、知识选择、通道选择、风险判断和任务策略生成
- 扩展能力：由 Agent 在受控沙箱中生成，并由平台决定是否保留、挂载、交付和长期运行的能力

这三类能力必须在需求定义上保持清楚区分，避免平台核心、Agent 行为和后天扩展混成一个不可控整体。

## 6. 关键业务规则

### 6.1 客户空间与组织边界规则

- 所有核心对象都必须有清晰归属边界，至少包括组织、客户空间和账号三个层级
- 账号、知识、素材、报表、凭证和扩展能力默认按客户空间隔离，而不是默认共享
- 扩展能力默认归属其所属客户空间，不应让所有客户共享同一套自由扩展空间
- 跨客户共享必须显式配置，并留下审计记录
- 组织级权限、客户级权限和账号级权限必须可区分，而不是混成一个角色体系

### 6.2 权限分层规则

- 权限设计不能只回答“谁能操作”，还必须回答“谁能读什么、谁能调用什么、谁能长期呈现什么”
- 平台至少应明确区分数据权限、能力权限和呈现权限
- 数据权限决定哪些事实层、状态层和派生层内容可被访问
- 能力权限决定哪些模型、知识源、执行通道和扩展能力可被调用
- 呈现权限决定哪些结果、报表和扩展入口可长期出现在用户界面中

### 6.3 多账号与多实例规则

- 所有功能默认支持多账号，而不是围绕单账号设计
- 同一个任务可以面向单账号或账号组
- 每个扩展实例有自己的名称和 ID
- 所有执行动作都应支持手动指定实例、自动选择实例、按账号绑定实例
- 当实例失联或能力不足时，系统应阻止高风险执行并给出提示

### 6.4 执行通道、凭证与配额规则

- 平台必须同时支持本地扩展通道与 X 官方 API / 官方 MCP 通道
- 每个账号和任务都应支持配置默认执行通道
- 对稳定性要求更高的客户，应优先支持官方通道
- 对需要浏览器上下文能力的动作，可继续走本地扩展通道
- 同一任务在必要时应支持通道回退和人工接管
- 所有执行结果都必须记录实际使用的通道
- 平台必须统一管理不同通道、不同 AI 和不同 MCP 所需的凭证，而不是分散在各个功能点里
- 凭证至少应支持组织级、客户级和账号级边界
- 平台必须支持按账号、按通道、按任务类型设置配额、限流、冷却和失败重试策略
- 当达到限额、触发限流或通道异常时，系统应阻止高风险动作继续放大，并给出降级、回退或人工接管路径

### 6.5 数据沉淀与状态抽象规则

- 原始推文、评论、互动、任务和通道事件必须可本地落库
- 高频核心指标应形成按天或按周期的快照，而不是只保留最后一次结果
- 报表、洞察和 Agent 分析应优先使用本地沉淀的历史数据和结构化状态，而不是每次都从原始数据重新推理
- 原始记录和结构化状态都要保留，前者用于追溯，后者用于快速分析
- 数据采集、状态抽象、模型推理和动作执行应彼此解耦，避免强耦合流程难以演进
- 平台事实数据和平台状态数据应被视为系统可信来源，沙箱内的分析结果、派生结果和扩展数据不应反向污染这些核心层级
- 报表、洞察和 Agent 运行时，应清楚区分事实层、状态层和派生层，而不是把所有数据混成一层

### 6.6 平台核心与沙箱边界规则

- 平台核心负责长期稳定、统一数据、统一权限、统一调度和统一展示宿主
- Agent 生成的扩展能力必须运行在受控沙箱中，而不是直接改写平台核心
- 沙箱对平台核心数据应以只读访问为主，避免污染平台事实层和状态层
- 沙箱生成的派生结果、分析逻辑和扩展能力，应与平台核心逻辑保持清晰边界
- 是否允许某个扩展能力进入主界面、长期挂载或自动执行，最终由平台控制
- 平台开放给沙箱的数据范围、能力范围和展示容器范围必须可配置、可审计、可收回
- 平台负责决定开放什么能力边界，Agent 负责在这些边界内组合能力，而不是反过来决定平台边界

### 6.7 AI、知识源与执行通道解耦规则

- AI 提供者、知识源和执行通道必须彼此解耦，而不是写死成一套组合
- 同一账号在不同任务中可以使用不同 AI、不同知识源和不同执行通道
- 平台应支持按客户、按角色、按任务配置默认组合
- 平台升级模型、知识源或执行通道时，不应破坏既有工作流

### 6.8 AI 规则

- 必须支持多种 AI 提供者
- 必须支持按任务切换 AI
- 必须支持按角色切换 AI
- 本地 Gemma、云端模型、REST API 型 AI 都应纳入统一管理
- 必须展示 AI 调用进展、失败原因和采用结果

### 6.9 知识与 MCP 规则

- 每个客户都应能接入自己的知识源，而不是共用一套通用知识
- AI 调用应支持绑定指定 MCP 或知识库范围
- 高价值回复和宣传内容应尽可能基于客户私有知识生成
- 系统应记录一次 AI 输出使用了哪些知识来源，方便追踪和审计
- 客户知识、平台工作流和基础模型必须彼此解耦，便于升级和替换

### 6.10 数字员工规则

- 不同账号可以绑定不同角色
- 不同客户可以配置不同品牌语气
- 同一任务可生成多个角色版本的候选内容
- 角色不只是提示词模板，还应有边界和禁区

### 6.11 Agent 与扩展能力规则

- Agent 必须围绕明确任务工作，而不是无边界运行
- Agent 必须有目标、权限、动作范围和审计记录
- Agent 应优先使用本地沉淀的历史运营数据、趋势快照和结构化状态作为分析输入
- Agent 产出的高级结果不应只停留在一次性会话中，而应能沉淀为可持续使用的扩展能力
- 扩展能力应具备被平台挂载、启停、替换和被其他模型接管的可能性
- 扩展能力应优先建立在平台已开放的固有能力之上，而不是假设自己拥有无边界系统权限
- 高风险动作必须支持人工审核
- 低风险动作可逐步开放自动执行
- 第一阶段先从 `Reply Agent` 开始，不做大而全通用 Agent

### 6.12 团队协作与审批规则

- 需要人工参与的任务必须支持指派责任人、审批人和接管人，而不是只停留在“待审核”状态
- 平台必须支持任务认领、转交、升级和超时接管
- 回复审批、发帖审批和风险事件审批可以有不同权限边界
- 高风险任务必须能升级到更高权限角色处理
- 团队协作过程必须可追踪、可审计、可复盘

### 6.13 风险规则

- 高风险评论和高风险回复必须标记
- 涉及品牌、产品缺陷、口碑风险的内容必须纳入风险预警
- 风险信号既影响任务审核，也影响报表输出

### 6.14 结果交付规则

- 所有任务必须有状态
- 所有 AI 调用必须可追踪
- 所有知识命中必须可追踪
- 所有自动化动作必须可复盘
- 所有关键运营动作都应可沉淀为报表和交付结果

### 6.15 平台能力分层规则

- 平台必须明确区分固有能力、编排能力和扩展能力，而不是把所有能力都视为同一层对象
- 固有能力由平台核心长期维护，扩展能力不得默认侵入式改写固有能力
- 编排能力负责组合模型、知识、执行通道和规则，不应直接取代平台的长期治理能力
- 扩展能力进入长期产品状态前，必须经过平台的挂载、展示、权限和审计边界约束
- 平台对扩展能力是否允许进入沙箱、进入主界面、长期挂载、自动运行、人工触发或跨边界访问拥有最终控制权
- 扩展能力是否值得长期保留，不应只看“能不能运行”，还应看它是否被转化为用户可见、可用、可交付的结果

## 8. 阶段计划

阶段计划描述的是“如何分期落地完整需求”，而不是定义产品边界。

### 8.1 第一阶段

第一阶段必须聚焦最有价值、最不可缺失、最容易形成销售闭环的部分。

第一阶段的核心是：

- 评论回复运营闭环
- 最小总控台与运行状态总览
- 最小账号资产、实例与执行通道管理
- 最小客户空间边界
- 最小权限分层边界
- 历史互动与推文数据本地沉淀
- 最小状态快照能力
- 最小 AI 编排与模型切换能力
- 最小 AI、知识与执行通道解耦框架
- 最小数字员工能力
- 最小知识接入能力
- 最小任务编排模型
- 最小平台能力分层边界
- 企业级执行通道预留
- 基础凭证与密钥治理
- 基础限流、冷却与失败保护
- 扩展能力边界预留
- 基础风险识别能力
- 基础审核指派与人工接管能力
- 基础任务与结果追踪
- 基础运营报表与审计能力
- Agent 可观测性

第一阶段的关键形态是：

**Reply Agent**

第一阶段闭环：

1. 发现某账号下的新评论或提及
2. 读取上下文
3. 读取本地历史数据、状态快照和客户知识
4. 结合账号角色和客户知识源调用 AI 生成候选回复
5. 按数字员工角色生成不同版本
6. 选择本地扩展通道或 X 官方通道执行
7. 审核或自动发送
8. 记录结果并回写本地数据资产
9. 进入报表

在第一阶段闭环稳定后，扩展到：

- 扩展能力基础挂载与展示
- 受控报表扩展能力
- 受控分析扩展能力
- MCP 知识接入增强
- 官方 API / 官方 MCP 通道增强
- 客户空间与组织能力增强
- 权限分层与治理能力增强
- 凭证治理与企业安全能力增强
- 通道配额与调度保护增强
- 团队协作与审批流增强
- 风险预警增强

## 9. 当前结论

`TweetPilot` 不应该被做成一个“桌面版 Twitter 自动化工具”。

它应该被做成：

**一支可按 token 雇佣、结合客户专属知识、支持双执行通道并逐步演化为垂直运营 Agent 平台的 Twitter 运营 AI 团队。**

如果只做执行工具，价格上不去。

如果把以下能力组合起来：

- 多账号执行能力
- 双执行通道能力
- 本地运营数据资产
- AI 协同能力
- 数字员工体系
- 客户专属知识能力
- 可持续生长的扩展能力
- Reply Agent
- 风险洞察
- 任务编排
- 报表交付
- 团队管理

那么 `TweetPilot` 才有机会成为：

- 代运营公司的操作系统
- 一人公司的 Twitter 运营外包团队
- 大客户的稳定社媒运营基础设施
- 同时卖软件、卖服务、卖方法论的高价值产品
