# 自定义数据积木 MVP 实施计划 v1

## 文档信息

- **版本**: v1.0
- **创建时间**: 2026-04-27
- **状态**: 实施计划
- **关联文档**: `docs/data-blocks-unified-roadmap.md`

---

## 一、文档目标

本文档只服务于一个目标：

> 落地“自定义数据积木 MVP”

也就是先跑通下面这条最小闭环：

> Python 写数据 → Python 注册卡定义 → 前端看到卡 → 前端读取数据 → 前端受控模板渲染

这份文档不讨论远期增强项，只关注第一阶段真正要做什么、改哪些地方、完成标准是什么。

---

## 二、MVP 范围

### 2.1 本阶段必须支持

- 用户通过 Python 向本地数据库写业务表
- 用户通过 Python 注册一张自定义数据卡定义
- 前端“添加积木”弹窗可以看到该卡
- 用户可以把该卡添加到页面
- 页面可以读取数据库数据并渲染卡片
- 渲染采用受控 HTML 模板
- 出错时显示统一错误态

### 2.2 本阶段明确不做

- 在线执行 Python
- 模板编辑器
- 复杂模板语法
- 图表脚本模板
- 模板市场
- 自定义卡共享能力
- 完整可视化配置器
- 高自由度样式系统

---

## 三、推荐实施策略

### 3.1 先做最小改动版本

MVP 阶段建议采用：

- **自定义卡定义先入数据库**
- **卡片实例和布局先尽量兼容现有逻辑**
- **优先跑通链路，不先做大迁移**

### 3.2 为什么不一开始就全量改造

如果一开始同时做：

- 自定义卡定义
- 卡片实例入库
- 统一实例模型
- 模板协议
- 配置系统

会导致第一阶段范围过大，验证周期变长。

所以 MVP 阶段建议把重点收敛到：

- 自定义卡能不能真正接进来
- 前后端链路能不能稳定跑通
- 模板渲染边界能不能控住

## 四、实施拆解

## 4.1 阶段一：数据库与数据结构

### 任务 1：新增自定义卡定义表

目标：先让系统有地方存“自定义卡是什么”。

建议表名：`custom_data_block_definitions`

### MVP 必需字段

- `id`
- `card_key`
- `name`
- `description`
- `source_type`
- `source_config`
- `template_type`
- `template_content`
- `status`
- `created_at`
- `updated_at`

### 字段说明

#### `card_key`
建议唯一，例如：

- `custom:my_daily_sales`
- `custom:task_summary`

#### `source_type`
MVP 先只支持：

- `sql_query`

#### `source_config`
建议保存 JSON，例如：

```json
{
  "query": "SELECT total_amount, order_count, updated_at FROM daily_sales_summary LIMIT 1",
  "result_mode": "single_row",
  "updated_at_field": "updated_at"
}
```

#### `template_type`
MVP 先只支持：

- `html_template`

### 完成标准

- 本地数据库中存在该表
- 可以插入一条自定义卡定义
- 可以按 `card_key` 查询出来

---

## 4.2 阶段二：后端接口能力

### 任务 2：新增“读取自定义卡定义列表”接口

目标：让前端可以获取所有可用自定义卡。

建议接口：

- `get_custom_data_block_definitions()`

### 返回建议

至少包含：

- `cardKey`
- `name`
- `description`
- `templateType`
- `status`

### 完成标准

- 前端可拿到定义列表
- 仅返回 `status = active` 的卡

---

### 任务 3：新增“读取单个自定义卡数据”接口

目标：让页面可以真正渲染自定义卡。

建议接口：

- `get_custom_data_block_data(cardKey, config?)`

### 职责

- 根据 `cardKey` 找到定义
- 读取 `source_config`
- 执行只读 SQL
- 返回统一数据结构

### 返回格式建议

```json
{
  "meta": {
    "title": "每日销售概览",
    "updatedAt": "2026-04-27T10:00:00Z"
  },
  "data": {
    "total_amount": 12580,
    "order_count": 42
  }
}
```

### 完成标准

- 已注册卡可以通过接口取到数据
- 查询失败时返回明确错误
- 无数据时返回可识别空结果

---

### 任务 4：增加 SQL 安全限制

目标：避免自定义卡能力直接变成任意 SQL 执行入口。

### MVP 规则

只允许：

- 单条 `SELECT`

不允许：

- `INSERT`
- `UPDATE`
- `DELETE`
- `DROP`
- `ALTER`
- `ATTACH`
- 多语句

### 建议校验项

- 必须以 `SELECT` 开头
- 禁止分号串联多语句
- 限制最大返回行数
- 限制查询超时

### 完成标准

- 非法查询被拒绝
- 合法只读查询可正常执行

---

## 4.3 阶段三：前端目录与实例接入

### 任务 5：让“添加积木”弹窗支持自定义卡目录

目标：用户在 UI 中能看到自定义卡。

主要改动文件：

- `src/components/AddCardDialog.tsx`
- 可能涉及 `src/config/data-blocks.ts`
- 可能涉及数据积木 service 层

### 调整方向

当前固定列表改成两类来源：

- 内置卡
- 自定义卡

### 完成标准

- 弹窗中能看到后端返回的自定义卡
- 内置卡与自定义卡都能正常展示

---

### 任务 6：支持把自定义卡加入页面

目标：用户不只是能看到卡，还能真正添加。

### 实例保存策略

MVP 建议先尽量兼容现有卡片结构。

推荐做法：

- `type` 设为 `custom`
- `config.definitionKey` 保存 `card_key`

例如：

```json
{
  "id": "card_123",
  "type": "custom",
  "position": 4,
  "config": {
    "definitionKey": "custom:my_daily_sales"
  },
  "lastUpdated": "2026-04-27T10:00:00Z"
}
```

### 为什么这样做

这样可以少改动当前模型，同时把“这是自定义卡”与“它具体对应哪个定义”分开。

### 完成标准

- 用户可以添加自定义卡到页面
- 刷新页面后实例仍存在

---

## 4.4 阶段四：前端渲染能力

### 任务 7：新增 `CustomDataCard` 组件

目标：独立承载自定义卡逻辑，不污染现有内置卡逻辑。

建议新增文件：

- `src/components/CustomDataCard.tsx`

### 职责

- 从 `card.config.definitionKey` 获取定义 key
- 请求后端读取数据
- 调模板渲染器
- 处理加载态、空态、错误态

### 完成标准

- 自定义卡能独立渲染
- 内置卡逻辑不受影响

---

### 任务 8：修改 `DataCard.tsx` 做渲染分发

目标：让内置卡和自定义卡共存。

### 调整方向

把当前纯 `switch(card.type)` 模式改成：

- 如果 `card.type === 'custom'`，走 `CustomDataCard`
- 否则继续走原有内置卡渲染逻辑

### 完成标准

- 现有 4 种内置卡行为不变
- 自定义卡进入新渲染路径

---

## 4.5 阶段五：模板渲染与安全控制

### 任务 9：实现最小模板渲染器

目标：支持最基础的受控模板渲染。

### MVP 支持

- `{{meta.xxx}}`
- `{{data.xxx}}`
- 基础静态 HTML 结构

### MVP 不支持

- 条件判断
- 循环
- JS 表达式
- 自定义脚本

### 完成标准

- 模板变量能正确替换
- 未知变量不会导致页面崩溃

---

### 任务 10：实现模板 sanitize

目标：阻止危险 HTML 被直接渲染。

### 至少处理

- 去掉 `script`
- 去掉事件属性
- 去掉危险标签
- 禁止外部资源
- 禁止内联脚本

### 完成标准

- 危险模板不会被原样执行
- 安全模板可以正常展示

---

## 4.6 阶段六：样例验证

### 任务 11：提供一份最小 Python 示例

目标：证明整个链路可以从外部 Python 接入。

### 示例流程

1. Python 创建一张业务表
2. Python 写入一行测试数据
3. Python 向 `custom_data_block_definitions` 注册一张卡
4. 前端展示该卡
5. 页面正确渲染数据

### 示例卡建议

优先选一个最简单的：

- `每日销售概览`
或
- `本地任务执行摘要`

### 完成标准

- 示例脚本执行后，前端页面能看到完整结果
- 不需要手工改数据库数据结构即可演示成功

## 七、阶段验收标准

以下标准全部满足，才算 MVP 完成。

### 功能验收

- Python 可以创建业务表并写入数据
- Python 可以注册一张自定义卡定义
- 前端可以加载自定义卡定义列表
- 用户可以将自定义卡加到页面
- 页面可以正确读取并显示数据

### 安全验收

- 后端不会执行非只读 SQL
- 模板不会执行危险脚本
- 错误不会导致整个页面崩溃

### 兼容性验收

- 现有 4 种内置卡不受影响
- 原有布局、刷新、删除、拖拽能力不被破坏

## 八、主要风险点

### 8.1 数据库接入方式不统一

如果当前项目里不同模块使用数据库的方式不一致，可能会影响新增表和查询逻辑的落地速度。

### 8.2 自定义卡实例与现有布局结构耦合

如果现有布局模型对 `type` 假设过强，可能需要对类型体系做一次小范围调整。

### 8.3 模板渲染安全边界容易失控

如果一开始放得太宽，后面会更难收回来，所以第一阶段必须严格保守。

### 8.4 Python 示例与真实场景差距

如果样例做得太简单，可能掩盖真实接入时的问题，所以建议示例至少覆盖：

- 写表
- 写数据
- 注册
- 前端显示

完整链路。
