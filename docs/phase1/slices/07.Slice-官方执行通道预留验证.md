## 1. 文档目的

本文档定义第一阶段第五个实际执行切片的正式开发边界。

这份文档要解决的问题不是“现在就把 X 官方 API 或 XMCP 完整接上”，而是：

- 平台是否已经把官方执行通道定义成正式对象
- 任务链路里是否已经有官方通道路由位
- 后续接入官方执行通道时，是否还需要重构 Reply Agent 主链路和任务模型

## 2. 切片目标

验证平台已经把官方执行通道保留为第一阶段正式执行对象，并在任务链路中预留官方通道路由位，从而保证后续企业级稳定通道的接入不会破坏当前主结构。

对第一阶段而言，这个切片优先证明：

- 平台的执行抽象不是只为本地扩展通道服务
- 官方通道已经是架构内对象，而不是未来再补的旁路
- 平台已经区分“当前主要可用通道”和“结构上正式存在的通道”

## 3. 主要涉及的工作包

`A + E`

补充关系：

- `A` 负责提供执行通道入口与控制面承载位置
- `E` 负责定义执行通道对象、任务对象和路由位

## 4. 前置依赖

依赖 Slice 4 已具备：

- 候选回复已进入任务体系
- 平台已形成最小风险分流与治理链路
- 平台已存在执行通道入口

同时依赖平台已具备：

- 至少一组客户空间、账号、实例与执行通道基础对象关系
- 平台中已经存在本地扩展执行通道对象的基础表达

## 5. 本切片的预留验证定义

## 5.1 第一阶段要验证的不是“接通”，而是“结构已成立”

本切片必须收窄到一个明确目标：

- 验证官方执行通道已经是正式对象
- 验证任务链路中已经有官方通道路由位
- 验证平台能够表达“本地扩展通道”和“官方通道”这两类通道

本切片当前不要求：

- 真实调用 X 官方 API
- 真实调用 XMCP
- 真实完成官方通道鉴权流程
- 真实完成官方通道发送动作

## 5.2 第一阶段官方执行通道应具备的最小产品属性

在本切片中，官方执行通道至少应具备以下属性：

- 它是平台中的正式执行通道对象
- 它能在界面中被看见或被识别
- 它能被任务模型引用
- 它能被路由逻辑识别为合法候选通道
- 它与本地扩展执行通道处于同一执行抽象层级

原则：

- 官方通道不能只是注释、文案或未来 TODO
- 官方通道必须进入平台对象模型

## 5.3 第一阶段任务链路中的最小官方通道路由位

本切片中，任务链路至少应能表达：

- 某个任务当前走的是本地扩展执行通道
- 某个任务结构上可以走官方执行通道
- 某个任务当前为什么还没有走官方执行通道

也就是说，平台至少应具备“路由位”，而不是等将来接入时再发明路由结构。

## 5.4 本切片不承担的官方通道深度

本切片当前不承担：

- 官方 API 真实调用
- XMCP 真实调用
- 企业级密钥治理落地
- 官方通道失败处理落地
- 官方通道限流策略落地
- 官方通道审计策略细化

这些内容不是不要，而是属于后续切片或后续阶段。

## 6. 当前轮廓交付

- 平台存在官方执行通道对象与入口
- 任务链路中存在官方通道路由位
- 不需要重构 Reply Agent 主链路即可表达官方通道
- 本地通道与官方通道的边界清楚

## 7. 当前明确不做

- 不在这一切片中完整接入 X 官方 API 或 XMCP
- 不在这一切片中实现企业级通道路由策略
- 不在这一切片中实现完整凭证治理体系
- 不在这一切片中实现真实官方发送动作
- 不在这一切片中实现复杂失败保护逻辑

## 8. 代码验收标准

本切片完成时，至少必须满足以下条件：

### 8.1 官方执行通道对象成立

- 平台中存在官方执行通道对象
- 官方执行通道对象不是纯展示文本
- 平台能够读取并展示该对象

### 8.2 执行抽象层级成立

- 本地扩展执行通道和官方执行通道处于同一执行抽象层级
- 二者都能被任务模型引用
- 二者都能被平台路由逻辑识别为合法通道

### 8.3 任务中的官方通道路由位成立

- 平台能表达任务当前通道和可选通道
- 平台能表达任务结构上可以走官方通道
- 平台不需要重构任务模型就能保留官方通道路径

### 8.4 与主链路兼容成立

- Reply Agent 主链路不需要为官方通道单独重写一套流程
- 治理链路不需要为官方通道单独重写一套流程
- 当前结构已经允许后续在官方通道位置接入真实实现

## 9. 自动测试要求

本切片进入开发前，至少应补齐以下自动测试：

## 9.1 单元测试

至少覆盖：

- 执行通道对象判定逻辑
- 本地通道与官方通道的抽象层级判断逻辑
- 任务通道路由位表达逻辑
- 通道类型切换或识别逻辑

## 9.2 集成测试

至少覆盖：

- 平台可读取并展示官方执行通道对象
- 任务对象可关联本地通道或官方通道路由位
- 路由结构不会因为官方通道存在而破坏当前任务链路

## 9.3 契约测试

至少覆盖：

- 执行通道对象与任务对象之间的引用契约
- 平台执行通道读取层的结构契约
- 路由位表达与任务状态之间的结构契约

说明：

- 本切片契约测试的重点不是外部官方接口
- 而是平台内部执行抽象的结构稳定性

## 9.4 端到端测试

至少覆盖一条最小链路：

1. 平台启动并进入执行通道管理或任务上下文
2. 界面中可见官方执行通道对象
3. 某个任务能够表达官方通道路由位或官方通道候选位

## 10. 人工测试手册

## 10.1 测试目的

验证平台已经把官方执行通道纳入正式执行抽象，而不是把它留在未来重构清单中。

## 10.2 测试前准备

- 平台应用可启动
- 平台中已存在执行通道入口
- 平台中已存在至少一个任务对象或任务容器

## 10.3 测试步骤

1. 打开平台
2. 进入执行通道相关入口
3. 检查是否存在官方执行通道对象
4. 进入任务相关入口
5. 检查任务是否可以表达官方通道路由位或官方通道候选位
6. 检查当前结构是否仍然兼容本地扩展执行通道

## 10.4 预期结果

- 官方执行通道对象存在
- 官方通道不是注释或静态文案
- 任务链路中存在官方通道路由位
- 平台当前主链路没有因此被破坏

## 10.5 异常判定

以下任一情况都视为未通过：

- 官方通道只是静态文本，不是平台对象
- 官方通道不能被任务模型引用
- 为了表达官方通道，主链路被分叉成单独流程
- 本地通道和官方通道处于不同抽象层级
- 平台在这一切片中错误尝试真实接入官方发送动作

## 11. 设计守卫清单

在开发本切片时，必须反复检查：

1. 是否把官方执行通道做成正式平台对象，而不是未来说明文字
2. 是否让本地通道和官方通道处于同一执行抽象层级
3. 是否只做“结构预留验证”，而不是偷跑真实接入
4. 是否保持任务、治理和主链路不需要为官方通道重写
5. 是否避免把企业级凭证、失败保护和复杂路由提前塞进这一切片

## 12. Definition of Done

满足以下条件才算本切片完成：

- 官方执行通道对象成立
- 任务中的官方通道路由位成立
- 本地通道与官方通道处于同一执行抽象层级
- 主链路和治理链路对官方通道保持兼容
- 自动测试覆盖到单元、集成、契约和最小端到端层
- 人工测试按手册执行后通过
- 没有把真实官方接入、企业级凭证治理和复杂路由策略错误塞入本切片
