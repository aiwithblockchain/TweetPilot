## 1. 文档目的

本文档定义第一阶段第二个纵向切片的正式开发边界。

这份文档要解决的问题不是“平台是否已经智能”，而是：

- 评论输入如何进入平台
- 输入如何归属到客户空间、账号、实例和执行通道
- 输入如何沉淀为平台可信事实
- 后续 Reply Agent 如何站在这条输入链路之上工作

## 2. 切片目标

把评论输入稳定接入平台，并写入本地事实层，形成第一条可重复触发、可追踪、可验证的输入链路。

对第一阶段而言，这个切片优先证明：

- 平台已经不是纯空壳
- 平台已经能接住真实或受控模拟的 X 输入
- 后续 Reply Agent 不需要直接面对浏览器原始世界

## 3. 主要涉及的工作包

`A + B`

补充关系：

- `A` 负责客户空间、账号、实例、执行通道和平台宿主入口
- `B` 负责事实层写入、状态准备和输入可追踪性

## 4. 前置依赖

依赖 Slice 1 已提供：

- 平台宿主
- 客户空间入口
- 账号入口
- 实例入口
- 执行通道入口
- 至少一组可用的客户空间、账号、实例与执行通道绑定关系

## 5. 本切片的输入链路定义

## 5.1 第一阶段优先支持的输入类型

第一阶段本切片应优先支持：

- 某条已监控推文下的回复评论输入

这是第一阶段最稳的真实输入来源，因为现有 `LocalBridge` 能直接提供：

- X 标签页状态
- X 实例列表
- 单条推文详情
- 单条推文的回复列表

相比之下，“全量提及流”或“通知中心流”虽然可能后续支持，但在当前阶段不应作为本切片的主输入链路。

因此本切片的原则是：

- 先把“推文回复评论输入”打稳
- 提及输入作为兼容方向保留
- 不把第一阶段输入范围做得过宽

## 5.2 第一阶段允许的输入方式

第一阶段允许两种输入方式：

### A. 真实输入链路

优先用于证明系统已能接住真实世界输入。

推荐链路：

1. 查询在线 X 实例
2. 查询当前 X 标签页状态
3. 绑定当前客户空间、账号、实例和标签页上下文
4. 读取目标推文详情
5. 读取目标推文下的回复列表
6. 从回复列表中识别出“待处理评论输入”
7. 归一化后写入平台事实层

### B. 受控模拟输入链路

用于自动测试、开发联调和本地重复验证。

推荐链路：

1. 构造标准化的评论输入样本
2. 走与真实输入一致的归一化流程
3. 写入相同事实层入口
4. 产出相同的可追踪结果

原则：

- 模拟输入不能走单独的数据模型
- 模拟输入必须复用真实输入的归一化与落库链路

## 5.3 第一阶段输入链路的最小结构

本切片在产品上必须至少形成以下逻辑链：

1. 某个客户空间下存在某个账号
2. 该账号关联某个实例和某个可用 X 标签页上下文
3. 平台能读取某个目标推文的回复评论
4. 平台能将该评论识别为一个平台输入对象
5. 该输入对象能被归属到明确的客户空间与账号
6. 该输入对象能被写入本地事实层
7. 平台能再次读取并展示该输入对象

## 5.4 本切片不承担的输入范围

本切片当前不承担：

- 全量提及流接入
- 私信输入接入
- 全量通知中心接入
- 多种评论来源统一去重
- 复杂分页与大规模历史回补

这些不是不要，而是不属于当前切片的最小成功边界。

## 6. 当前轮廓交付

- 至少一个账号的评论输入可进入平台
- 评论输入可以归属到明确的客户空间与账号
- 评论输入可以写入本地事实层
- 评论输入可在平台中被查看和追踪
- 至少存在一条可重复触发的评论输入链路，真实或模拟均可
- 最小状态输入准备路线成立

## 7. 当前明确不做

- 不在这一切片中实现回复生成
- 不在这一切片中实现执行与审核
- 不在这一切片中实现复杂趋势分析
- 不在这一切片中打通完整提及流
- 不在这一切片中打通复杂历史回补能力

## 8. 代码验收标准

本切片完成时，至少必须满足以下条件：

### 8.1 输入对象成立

- 平台可以定义“评论输入”这一平台对象
- 该对象不是页面临时状态，而是可复用平台事实

### 8.2 归属关系成立

- 每条评论输入都能归属到明确的客户空间
- 每条评论输入都能归属到明确的账号
- 如果输入来自真实链路，则能关联实例或标签页上下文

### 8.3 真实链路或模拟链路至少一条可重复运行

- 开发环境下必须至少有一条可重复触发的输入链路
- 这条链路可以是真实链路，也可以是模拟链路
- 但无论哪种链路，都必须最终进入同一事实层入口

### 8.4 落库与读取都成立

- 评论输入能够写入本地事实层
- 平台能够再次读取这些评论输入
- 平台能够在基础界面中展示这些评论输入

### 8.5 为下个切片提供最小输入准备

- Slice 3 需要的最小输入条件已经成立
- 至少可以从本地读取：评论内容、归属账号、基础上下文来源

## 9. 自动测试要求

本切片进入开发前，至少应补齐以下自动测试：

## 9.1 单元测试

至少覆盖：

- 评论输入归一化逻辑
- 客户空间与账号归属判定逻辑
- 输入重复识别或最小幂等逻辑
- 事实层写入前的最小校验逻辑

## 9.2 集成测试

至少覆盖：

- 通过模拟输入链路写入评论事实
- 从事实层再次读取评论事实
- 评论事实与客户空间、账号之间的关联关系成立
- 评论事实不会被错误写成状态层对象

## 9.3 契约测试

至少覆盖：

- X 实例查询适配层
- X 标签页状态查询适配层
- 推文回复读取适配层

说明：

- 契约测试的目标不是验证 Twitter 本身
- 而是验证平台适配层面对 `LocalBridge` 返回结构变化时不会静默失效

## 9.4 端到端测试

至少覆盖一条最小链路：

1. 平台存在客户空间和账号
2. 输入链路触发一条评论输入
3. 评论输入进入平台
4. 评论输入在界面中可见

## 10. 人工测试手册

## 10.1 测试目的

验证平台已经具备第一条可信输入链路，并且这条链路已可供后续 Reply Agent 使用。

## 10.2 测试前准备

- 平台应用可启动
- 已存在一个客户空间
- 已存在一个账号
- 已存在一个实例
- 已存在一个执行通道对象
- 具备真实输入条件，或准备好模拟输入样本

## 10.3 测试步骤

### 方案 A：真实输入验证

1. 打开平台
2. 确认目标客户空间、账号、实例都处于可见状态
3. 选择一条已监控推文或准备读取的目标推文
4. 触发读取该推文回复列表的输入流程
5. 观察评论是否进入平台列表或详情视图
6. 观察该评论是否已经归属到正确账号与客户空间

### 方案 B：模拟输入验证

1. 打开平台
2. 准备一条标准化评论样本
3. 通过模拟输入入口注入该样本
4. 观察该评论是否进入平台列表或详情视图
5. 观察该评论是否已经归属到正确账号与客户空间

## 10.4 预期结果

- 评论输入能够进入平台
- 评论输入能够被再次查看
- 评论输入归属关系正确
- 平台能够区分这是一条事实输入，而不是任务结果或状态对象

## 10.5 异常判定

以下任一情况都视为未通过：

- 评论只能在页面短暂出现，不能再次读取
- 评论不能归属到客户空间或账号
- 评论归属到错误账号
- 真实链路与模拟链路走了两套不同的落库逻辑
- 评论被错误写成状态结论或任务结果

## 11. 设计守卫清单

在开发本切片时，必须反复检查：

1. 是否让评论输入先变成平台事实，而不是页面状态
2. 是否让评论输入归属于明确客户空间和账号
3. 是否把真实输入链路和模拟输入链路收束到同一事实层入口
4. 是否提前把 Reply Agent、审核、执行逻辑塞进了输入切片
5. 是否误把评论输入直接写成状态结论
6. 是否为后续 Slice 3 保留了最小可复用输入

## 12. Definition of Done

满足以下条件才算本切片完成：

- 文档中定义的输入链路范围已按本切片边界落地
- 至少一条可重复触发的输入链路成立
- 评论输入可以稳定写入并读取
- 评论输入的客户空间与账号归属成立
- 自动测试覆盖到单元、集成、契约和最小端到端层
- 人工测试按手册执行后通过
- 没有绕过平台事实层边界
- 没有把回复生成、执行、审核错误塞入本切片
