## 1. 文档目的

本文档定义第一阶段第一个纵向切片的正式开发边界。

这份文档要解决的问题不是“平台是否已经有真实业务能力”，而是：

- 平台是否已经有稳定的桌面宿主
- 核心对象是否已经有统一入口和归属关系
- 第一阶段主链路后续是否已经有稳定承载面
- 后续切片开发是否可以建立在统一平台壳层之上，而不是继续堆临时页面

## 2. 切片目标

建立可承载第一阶段主链路的桌面宿主与平台控制面，并让最小客户空间、账号、实例和执行通道关系在平台中成立。

对第一阶段而言，这个切片优先证明：

- `TweetPilot` 已经是一个平台宿主，而不是零散页面集合
- 平台已经有后续所有能力的统一挂载面
- 平台已经能表达核心对象及其基础归属关系

## 3. 主要涉及的工作包

`A`

补充关系：

- `A` 负责平台宿主、主导航、对象入口和统一控制面
- 这个切片不直接承担数据沉淀、Agent 生成、执行动作和审核治理本体

## 4. 前置依赖

无。

这是第一阶段最前置切片。

但这不意味着它要把所有真实接入一次性做完。这个切片允许通过受控种子数据、初始化配置或本地假数据建立最小对象关系，只要这些对象关系后续可以被真实链路接管即可。

## 5. 本切片的壳层与控制面定义

## 5.1 第一阶段优先建立的壳层能力

本切片优先建立：

- 桌面应用主窗口
- 平台主导航
- 总控台入口
- 客户空间入口
- 账号入口
- 实例入口
- 执行通道入口
- 后续任务、报表和扩展能力挂载位

这意味着本切片的重点不是“界面好不好看”，而是“平台宿主是否成立”。

## 5.2 第一阶段必须成立的最小对象关系

本切片在产品上必须至少形成以下最小对象关系：

1. 至少存在一个客户空间
2. 该客户空间下至少存在一个账号
3. 该账号至少能关联一个实例
4. 该账号至少能关联一个执行通道
5. 平台能够把这些对象在统一界面中展示出来

这组对象关系可以来自：

- 真实本地配置
- 受控初始化数据
- 开发环境的本地种子数据

原则：

- 第一阶段允许用受控数据建立壳层
- 但这些对象不能只是静态 UI 文案，而必须是平台内部可读取、可切换、可挂载的对象

## 5.3 第一阶段的控制面边界

本切片中的控制面必须至少承担：

- 统一进入不同对象入口
- 统一查看平台运行状态总览
- 为后续切片保留明确挂载位置

当前不要求控制面承担：

- 复杂业务规则配置
- 复杂组织权限配置
- 复杂任务调度配置

## 5.4 本切片不承担的业务深度

本切片当前不承担：

- 真实评论输入链路
- Reply Agent 行为
- 执行动作打通
- 审核队列与治理逻辑
- 报表计算逻辑
- 扩展能力运行时

这些能力都要在后续切片中逐步接入，但后续所有能力都必须挂在这个壳层之上。

## 6. 当前轮廓交付

- 应用可启动并进入平台主界面
- 存在总控台、客户空间、账号、实例、执行通道的基础入口
- 存在统一运行状态总览入口
- 至少存在一组可用的客户空间、账号、实例与执行通道绑定关系
- 存在后续任务、报表、扩展能力的挂载位

## 7. 当前明确不做

- 不在这一切片中实现真实数据沉淀
- 不在这一切片中实现 Reply Agent 行为
- 不在这一切片中实现任务执行与审核流
- 不在这一切片中实现真实执行通道打通
- 不在这一切片中实现复杂权限体系

## 8. 代码验收标准

本切片完成时，至少必须满足以下条件：

### 8.1 平台宿主成立

- 应用可以稳定启动
- 用户可以进入统一主界面
- 平台存在明确的主导航与基础页面结构

### 8.2 核心对象入口成立

- 平台中存在客户空间入口
- 平台中存在账号入口
- 平台中存在实例入口
- 平台中存在执行通道入口

### 8.3 最小对象关系成立

- 至少存在一组客户空间、账号、实例和执行通道的绑定关系
- 这组关系不是硬编码到单个组件里的纯展示文本
- 平台能够读取并展示这组关系

### 8.4 统一状态入口成立

- 平台存在统一运行状态总览入口
- 平台至少能表达哪些对象可用、不可用或待接入

### 8.5 后续挂载位成立

- 平台界面结构中已经有任务、报表和扩展能力的挂载位
- 后续切片不需要重构主导航才能进入平台

## 9. 自动测试要求

本切片进入开发前，至少应补齐以下自动测试：

## 9.1 单元测试

至少覆盖：

- 平台壳层中的对象关系组装逻辑
- 客户空间切换或对象归属展示逻辑
- 导航路由或视图切换逻辑
- 运行状态展示的最小判定逻辑

## 9.2 集成测试

至少覆盖：

- 应用启动后进入主界面
- 平台可读取并展示至少一组对象关系
- 客户空间、账号、实例、执行通道入口可见
- 总控台与状态总览入口可见

## 9.3 契约测试

至少覆盖：

- 平台壳层读取最小对象关系的结构契约
- 主导航入口与对象入口之间的结构契约
- 运行状态总览入口与最小状态对象之间的结构契约

说明：

- 本切片契约测试的重点不是外部依赖
- 而是平台壳层对象结构和入口结构的稳定性

## 9.4 端到端测试

至少覆盖一条最小链路：

1. 启动平台
2. 进入主界面
3. 查看总控台
4. 进入客户空间或账号入口
5. 查看已存在的对象关系

说明：

- 本切片暂不要求真实外部依赖参与 E2E
- 允许使用初始化数据完成端到端验证

## 10. 人工测试手册

## 10.1 测试目的

验证平台壳层已经成立，且后续切片拥有统一承载面。

## 10.2 测试前准备

- 应用可启动
- 已准备一组最小初始化对象数据，真实或模拟均可

## 10.3 测试步骤

1. 启动平台
2. 观察是否进入统一主界面
3. 检查总控台入口是否存在
4. 检查客户空间、账号、实例、执行通道入口是否存在
5. 进入相关入口，确认至少存在一组对象关系
6. 检查是否存在任务、报表和扩展能力挂载位
7. 检查状态总览入口是否存在

## 10.4 预期结果

- 平台界面不是孤立页面，而是统一壳层
- 核心对象入口存在
- 最小对象关系存在
- 运行状态总览入口存在
- 后续挂载位存在

## 10.5 异常判定

以下任一情况都视为未通过：

- 应用无法稳定进入主界面
- 平台只有单页面，没有统一导航或控制面
- 核心对象入口缺失
- 客户空间、账号、实例、执行通道没有形成可见关系
- 后续挂载位不存在

## 11. 设计守卫清单

在开发本切片时，必须反复检查：

1. 是否先建立平台宿主，而不是先做某个业务页面
2. 是否把客户空间边界前置进入平台壳层
3. 是否让对象关系成为平台对象，而不是纯展示文本
4. 是否把后续挂载位留在平台壳层内，而不是临时放在某个业务页里
5. 是否提前把数据、Agent、执行、审核逻辑塞进了壳层切片

## 12. Definition of Done

满足以下条件才算本切片完成：

- 平台宿主、主导航和总控台入口成立
- 核心对象入口成立
- 至少一组最小对象关系成立
- 统一状态总览入口成立
- 任务、报表和扩展能力挂载位成立
- 自动测试覆盖到单元、集成、契约和最小端到端层
- 人工测试按手册执行后通过
- 没有把数据沉淀、Reply Agent、执行和审核逻辑错误塞入本切片
