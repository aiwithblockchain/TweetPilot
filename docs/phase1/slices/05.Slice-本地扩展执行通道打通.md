## 1. 文档目的

本文档定义第一阶段第六个实际执行切片的正式开发边界。

这份文档要解决的问题不是“平台是否已经拥有完整自动化运营能力”，而是：

- 已经进入可执行状态的回复任务，如何通过本地扩展执行通道真正发出
- 本地扩展执行结果如何回流到平台任务链路
- 平台如何为真实写动作建立最小失败保护、冷却与人工接管边界
- 平台是否已经具备一条可验证的真实执行闭环

## 2. 切片目标

让已经通过最小治理边界的回复任务，可通过本地扩展执行通道完成一次真实回复发送，并把执行结果、失败状态和人工接管信息回流平台，从而证明第一阶段主链路已经从“生成与治理”走到了“受控执行”。

对第一阶段而言，这个切片优先证明：

- 平台已经具备一条真实可写的本地执行通道
- 真实写动作必须建立在任务和治理链路之上
- 执行结果不是浏览器侧瞬时现象，而是平台正式结果
- 平台已经具备最小失败保护和人工接住能力

## 3. 主要涉及的工作包

`E + F`

补充关系：

- `E` 负责执行通道对象、执行请求、结果回流和最小失败保护
- `F` 负责处理责任归属、人工接管、审计和可观测性入口

## 4. 前置依赖

依赖 Slice 4 已具备：

- 候选回复已进入任务体系
- 高风险任务已被分流到审核队列
- 低风险任务或人工确认后的任务，已具备继续进入执行链路的资格
- 任务已经具备最小责任归属与接管入口

依赖 Slice 7 已具备：

- 平台执行模型已经同时表达本地扩展执行通道和官方执行通道
- 平台已确认当前打通的是本地扩展执行通道，而不是把它写成唯一执行模型

同时依赖平台已具备：

- 至少一个可用的本地扩展执行通道对象
- 至少一个可用于真实回复发送的受控测试账号或测试环境
- 至少一条可重复验证的评论任务输入链路

## 5. 本切片的真实执行定义

## 5.1 第一阶段优先支持的真实写动作

第一阶段本切片应优先支持：

- 针对单条评论任务，通过本地扩展执行通道发送一条回复

当前不要求：

- 发布独立推文
- 批量执行多个回复任务
- 点赞、关注、转发等其他写动作
- 自动化组合动作

原则：

- 第一阶段先把“单条评论回复”这一条真实写链路打稳
- 不把本切片扩展成通用动作引擎

## 5.2 第一阶段最小执行资格边界

进入本地扩展执行通道的任务，至少应满足：

- 任务来源明确
- 任务归属客户空间和账号明确
- 候选回复内容已确定
- 任务当前不处于高风险待审核状态
- 平台已决定该任务当前走本地扩展执行通道

原则：

- Agent 不能直接驱动真实写动作
- 真正进入执行的任务，必须已经成为平台可控制的对象
- 通道路由属于平台决定，不属于单个 Agent 自由决定

## 5.3 第一阶段最小执行请求与结果回流

本切片中，平台至少应形成一条最小执行闭环：

- 从任务中读取已确认的候选回复
- 调用本地扩展执行通道进行真实发送
- 读取本次执行的成功、失败或未完成结果
- 将结果回流到平台任务链路和审计链路

平台至少应能表达：

- 本次执行是否已发起
- 本次执行是否成功
- 本次执行为什么失败或未完成
- 任务当前是否需要人工接管或等待后续处理

## 5.4 第一阶段最小失败保护

本切片至少应具备：

- 最小失败状态记录
- 最小冷却或阻断能力
- 最小人工接管入口
- 最小重复执行保护

当前不要求：

- 复杂熔断系统
- 复杂重试编排系统
- 企业级通道弹性治理

原则：

- 第一阶段先解决“真实执行失败后不会失控”
- 不提前做复杂可靠性系统

## 5.5 本切片不承担的执行深度

本切片当前不承担：

- 官方执行通道真实接入
- 自动批量执行
- 高风险动作自动放行
- 复杂定时调度
- 多动作组合工作流
- 完整限流治理体系

因此本切片的边界是：

- 把一条最小真实写链路打通
- 把结果回流和失败保护接好
- 但不在这里把执行系统做宽做深

## 6. 当前轮廓交付

- 平台可将已具备执行资格的回复任务送入本地扩展执行通道
- 平台可获得并记录真实执行结果
- 平台对失败和重复执行具备最小保护
- 人工接管与执行结果共存于统一任务链路

## 7. 当前明确不做

- 不在这一切片中完整打通官方执行通道
- 不在这一切片中开放高风险自动写动作
- 不在这一切片中实现复杂调度策略
- 不在这一切片中实现批量执行与组合动作
- 不在这一切片中实现企业级限流与熔断治理

## 8. 代码验收标准

本切片完成时，至少必须满足以下条件：

### 8.1 执行资格成立

- 平台能够明确区分“可执行任务”和“不可执行任务”
- 只有已具备执行资格的任务才能进入本地扩展执行通道
- 平台能够读取该任务对应的候选回复、归属账号和当前通道路由

### 8.2 本地扩展真实发送成立

- 平台能够通过本地扩展执行通道发起真实回复发送
- 至少存在一条可受控验证的真实发送链路
- 平台不需要为真实发送绕过任务链路或审核链路

### 8.3 执行结果回流成立

- 平台能够记录本次执行是否成功
- 平台能够记录失败或未完成状态
- 平台能够把执行结果回写到任务对象和最小审计链路

### 8.4 最小失败保护成立

- 平台能够避免同一任务在异常情况下被无边界重复执行
- 平台能够对失败任务设置最小阻断、等待或人工接管状态
- 平台不会因为一次执行失败而阻塞主链路

### 8.5 与主结构兼容成立

- 本地扩展执行通道的真实打通不会破坏官方执行通道预留结构
- Reply Agent 不直接拥有真实写动作执行权
- 治理链路和执行链路仍然围绕统一任务对象运作

## 9. 自动测试要求

本切片进入开发前，至少应补齐以下自动测试：

## 9.1 单元测试

至少覆盖：

- 执行资格判定逻辑
- 任务到执行请求的转换逻辑
- 执行结果到任务状态的回写逻辑
- 最小失败保护逻辑
- 重复执行保护逻辑

## 9.2 集成测试

至少覆盖：

- 平台从可执行任务构造本地扩展执行请求
- 本地扩展执行结果能够回流平台
- 失败状态能够被记录并触发最小保护
- 人工接管状态与执行结果可以共存于统一任务链路

## 9.3 契约测试

至少覆盖：

- 平台执行请求对象与本地扩展执行适配层之间的契约
- 执行结果对象与任务回写层之间的契约
- 执行状态、审计状态与任务状态之间的结构契约

说明：

- 本切片契约测试的重点不是浏览器 UI
- 而是平台执行抽象与本地扩展适配层之间的结构稳定性

## 9.4 端到端测试

至少覆盖一条最小链路：

1. 平台存在一条已具备执行资格的回复任务
2. 任务进入本地扩展执行通道
3. 平台获得执行结果
4. 平台界面中可见任务状态和执行结果变化

说明：

- CI 中可以使用受控模拟扩展适配层完成最小端到端验证
- 真实浏览器扩展环境验证应由人工测试手册覆盖

## 10. 人工测试手册

## 10.1 测试目的

验证平台已经具备一条受控的本地扩展真实写链路，能够把可执行回复任务真正发送出去，并正确记录结果与失败状态。

## 10.2 测试前准备

- 平台应用可启动
- 平台中存在至少一个已具备执行资格的回复任务
- 本地扩展执行通道可用
- 至少一个测试账号和测试环境已准备完成
- 测试人员知道本轮只验证“单条评论回复发送”

## 10.3 测试步骤

1. 打开平台
2. 进入任务相关入口
3. 找到一条已具备执行资格的回复任务
4. 触发该任务通过本地扩展执行通道执行
5. 观察浏览器扩展侧是否完成真实回复动作
6. 回到平台，检查任务状态和执行结果是否更新
7. 如构造失败场景，检查平台是否进入最小失败保护或人工接管状态

## 10.4 预期结果

- 任务能够进入本地扩展执行通道
- 真实回复动作被触发
- 平台能够显示执行成功、失败或未完成状态
- 失败时平台不会失控重复执行
- 人工接管入口仍然存在

## 10.5 异常判定

以下任一情况都视为未通过：

- 平台绕过任务链路直接驱动发送
- Agent 直接控制真实写动作
- 本地扩展真实发送成功，但平台没有记录结果
- 发送失败后任务状态不清楚
- 同一任务在异常情况下被重复无边界触发
- 本地扩展通道打通后破坏了官方执行通道预留结构

## 11. 设计守卫清单

在开发本切片时，必须反复检查：

1. 是否把真实执行建立在任务资格和平台路由之上
2. 是否只打通“单条评论回复”这一条最小真实写链路
3. 是否保持 Agent 不直接拥有执行控制权
4. 是否把执行结果正式回流到任务和审计链路
5. 是否为失败、重复执行和人工接管保留最小保护
6. 是否避免把官方通道、批量执行和复杂调度提前塞进本切片

## 12. Definition of Done

满足以下条件才算本切片完成：

- 已具备执行资格的回复任务可进入本地扩展执行通道
- 平台可完成一条受控真实回复发送链路
- 执行结果可正式回流平台
- 最小失败保护和人工接管边界成立
- 本地通道真实打通后不破坏官方执行通道预留结构
- 自动测试覆盖到单元、集成、契约和最小端到端层
- 人工测试按手册执行后通过
- 没有把批量执行、复杂调度和企业级治理错误塞入本切片
