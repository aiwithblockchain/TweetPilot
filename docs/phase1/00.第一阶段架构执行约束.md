## 2. 第一阶段的架构目标

第一阶段的架构目标不是把 `TweetPilot` 做成完整平台，而是以最小范围证明以下结构已经成立：

- 平台核心是稳定宿主
- 本地事实与状态能够成为可信输入
- AI、知识和执行通道已经解耦
- Reply Agent 能在平台边界内完成生成工作
- 高风险结果可以被治理、接管和审计
- 任务结果可以回流到报表与交付层
- 后续扩展能力路线没有被第一阶段代码结构堵死

因此，第一阶段不是“先把功能做出来”，而是“先按正确结构把最小闭环做出来”。

## 3. 第一阶段必须守住的总体原则

### 3.1 平台核心先于业务闭环稳定

第一阶段所有业务能力都必须挂靠在平台核心之上。

不允许为了加快业务验证而让某个单独流程绕开平台核心，长成独立的临时系统。

### 3.2 任务先于动作

第一阶段所有对外执行的运营动作，都必须先进入任务体系，再进入执行或审核路径。

不允许把“模型输出”直接连成“立即动作”。

### 3.3 本地数据先于即时推理

第一阶段的 Agent、审核、报表和审计，应优先消费本地沉淀的事实与状态，而不是默认围绕一次性抓取结果工作。

### 3.4 平台控制权先于 Agent 自由度

第一阶段允许 Agent 生成候选结果、风险建议和执行建议。

但平台必须保留以下最终控制权：

- 是否允许执行
- 走哪条执行通道
- 是否要求人工审核
- 是否允许进入长期能力边界

### 3.5 预留路线不能污染当前核心

第一阶段必须为扩展能力和沙箱预留路线，但预留路线不能反向侵入当前平台核心，不允许以“为了未来可扩展”为理由把第一阶段代码结构做成无边界系统。

## 4. 第一阶段的执行主链路

第一阶段开发时，应始终围绕下面这条主链路组织实现和验证：

1. 评论或提及进入平台
2. 平台将其写入本地事实层
3. 平台整理最小状态输入
4. Reply Agent 读取角色、知识、历史和状态
5. Reply Agent 生成候选回复与风险建议
6. 平台将候选结果写入任务体系
7. 平台根据规则进入执行或审核路径
8. 执行、审核或接管结果回流本地数据层
9. 结果进入审计与报表层

第一阶段所有切片开发，都应能说明自己位于这条主链路的哪个位置。

如果某个功能无法挂回这条主链路，通常说明它不属于第一阶段核心闭环。

## 5. 第一阶段的架构分层约束

## 5.1 平台核心层

第一阶段平台核心层负责：

- 桌面宿主与主导航
- 客户空间边界
- 账号、实例、执行通道等核心对象入口
- 权限边界入口
- 凭证与密钥边界入口
- 任务、审计、报表、扩展能力的统一挂载控制

开发约束：

- 平台核心层不能依赖某个单一 Agent 流程存在
- 平台核心层不直接承载业务提示词和业务策略逻辑
- 平台核心层不被扩展能力直接改写

## 5.2 数据层

第一阶段数据层必须至少区分：

- 事实层
- 状态层

事实层用于承接：

- 历史推文
- 历史互动
- 候选回复结果
- 审核结果
- 执行结果
- 接管结果

状态层用于承接：

- 账号基础状态
- 实例基础状态
- 执行通道基础状态
- 与 Reply Agent 直接相关的最小互动状态

开发约束：

- 原始事实不能被状态结论覆盖
- 状态输出不能反向修改原始事实
- 第一阶段不允许把未来扩展能力的派生数据写入平台事实层和状态层

## 5.3 AI 与知识层

第一阶段 AI 与知识层负责：

- 统一纳管本地模型、云端模型和第三方 REST AI
- 统一纳管客户知识与 MCP 输出
- 为 Reply Agent 提供统一调用入口
- 追踪 AI 调用过程与知识命中结果

开发约束：

- 业务流程中不得写死某个唯一模型来源
- 知识接入必须是平台能力，不得散落在单个业务页面里临时拼接
- 模型来源、知识来源和执行通道不得耦合成单条不可替换流程

## 5.4 Agent 层

第一阶段 Agent 层主要对应 Reply Agent。

它负责：

- 读取平台开放的输入
- 组合角色、知识、历史和状态
- 生成候选回复
- 输出风险建议与执行建议

开发约束：

- Agent 不直接持有平台原始凭证
- Agent 不直接决定最终执行通道
- Agent 不绕过审核边界
- Agent 不直接改写平台核心层状态定义

## 5.5 执行与任务层

第一阶段执行与任务层负责：

- 回复任务对象化
- 执行通道对象化
- 通道路由承接
- 执行结果记录
- 限流、冷却和失败保护

开发约束：

- 执行动作必须挂在任务对象上
- 本地扩展执行通道与官方执行通道预留必须是平台对象，不是页面分支
- 通道路由属于平台策略，不属于单个 Agent 自由决定

## 5.6 治理与审计层

第一阶段治理与审计层负责：

- 审核队列
- 指派、接管、升级
- 最小权限分层
- 最小审计记录
- Agent 基础可观测性

开发约束：

- 高风险任务不得绕过治理层
- 所有治理动作必须围绕任务对象发生
- 审计必须是平台级记录，而不是页面提示信息

## 5.7 报表与交付层

第一阶段报表与交付层负责：

- 组织任务结果
- 组织审核和接管结果
- 组织基础风险摘要
- 形成基础客户可读输出

开发约束：

- 报表层只能组织结果，不能反向承载治理规则
- 报表层应建立在本地数据、任务结果和审计结果之上
- 客户可读结果必须从第一阶段出现，但不承担复杂客户门户职责

## 5.8 扩展能力与沙箱预留层

第一阶段这一层不落完整运行时，只承接结构边界。

它负责：

- 平台能力分层预留
- 扩展能力挂载位预留
- 受控展示容器预留
- 派生数据边界预留
- 平台最终控制权预留

开发约束：

- 第一阶段不得把扩展能力误做成平台核心能力
- 第一阶段不得开放无边界代码执行
- 第一阶段不得让扩展能力写入平台核心事实层与状态层

## 6. 第一阶段的横向解耦约束

## 6.1 客户空间边界必须全链路存在

第一阶段所有核心对象都必须能落到客户空间边界内，包括：

- 账号
- 角色
- 知识
- 任务
- 审计记录
- 报表结果
- 未来扩展能力归属

不允许出现“先做全局共享，后面再补客户空间”的临时做法。

## 6.2 模型、知识、执行通道必须三者解耦

第一阶段任何业务链路都不得默认写成：

- 固定模型
- 固定知识源
- 固定执行通道

三者必须由平台分别纳管，再在任务流中组合。

## 6.3 权限必须分层，而不是单一开关

第一阶段至少要保留以下三类权限边界：

- 数据权限
- 能力权限
- 呈现权限

即使具体权限模型仍然很简化，这三层边界也必须提前进入架构。

## 6.4 凭证和密钥必须与 Agent 隔离

第一阶段必须确保：

- 平台负责管理凭证与密钥边界
- Agent 消费的是平台开放后的能力，而不是原始凭证
- 界面层和 Agent 层都不应直接持有和传播原始密钥

## 6.5 结果回流必须是统一链路

第一阶段所有关键结果都必须能回流到统一平台数据链路，包括：

- 候选回复采用结果
- 审核结果
- 执行结果
- 接管结果

如果某个结果只停留在页面内存里或单次会话里，就不符合第一阶段架构要求。

## 7. 第一阶段开发时明确不允许的做法

以下做法即使能加快短期功能实现，也不应在第一阶段采用：

- 在页面层直接拼接业务主逻辑
- 让 Reply Agent 直接决定最终执行通道
- 让 Reply Agent 直接调用原始凭证
- 把审核、接管做成脱离任务体系的独立流程
- 把报表建立在页面临时状态而不是平台数据上
- 把未来扩展能力代码直接写进平台核心层
- 用“先全局共享，后续再补隔离”的方式绕过客户空间边界
- 用“先打通本地通道，后面再说”完全跳过官方执行通道预留

## 8. 第一阶段开发前的架构自检

每个纵向切片在进入开发前，都应至少回答下面八个问题：

1. 这个切片位于主链路的哪个位置
2. 它会写入事实层、状态层，还是只消费这两层
3. 它是否绕过了客户空间边界
4. 它是否把模型、知识、执行通道写死在一起
5. 它是否让 Agent 越权决定平台控制逻辑
6. 它是否绕过了任务对象或治理链路
7. 它是否把结果稳定回流到了平台数据层
8. 它是否误把未来扩展能力写进了平台核心

只要有一项回答不清楚，这个切片就不应直接进入开发。

## 9. 第一阶段完成时，架构上应出现什么结果

第一阶段完成时，从架构角度看，至少应出现以下结果：

- 平台核心已经稳定承载主闭环
- 本地事实层和状态层已经形成最小可信输入
- AI、知识和执行通道已经具备可替换结构
- Reply Agent 已经是受控的生成层，而不是自由执行层
- 任务、审核、接管、审计和报表已经围绕同一条主链路组织
- 官方执行通道虽然未完整落地，但路径已经是正式对象
- 扩展能力与沙箱虽然未完整实现，但后续路线不需要重构平台核心
