## 1. 文档目的

本文档用于拆解第一阶段工作包 B：本地数据与状态沉淀。

本文档只定义：

- 第一阶段需要沉淀哪些最小数据资产
- 第一阶段需要形成哪些最小状态能力
- 这一工作包与报表、Agent、执行之间的关系
- 第一阶段做到什么程度才足够支撑 Reply Agent 闭环

本文档不展开表结构、采集字段和具体存储实现。

## 2. 工作包定位

工作包 B 是第一阶段的数据底座。

它的价值不是“多存一点数据”，而是把 `TweetPilot` 和普通 Agent 产品拉开差距：

- 平台不依赖一次性抓取和一次性推理
- 平台能够积累本地运营数据资产
- Agent 与报表都优先建立在本地历史数据与状态之上

如果工作包 B 不成立，第一阶段就只能是一个“会调模型的回复工具”，而不是一个平台。

## 2A. 第一阶段数据形态

第一阶段的数据底座，不应被理解为“把原始数据存下来”。

它更准确的产品形态应是：

- 平台对第一阶段关键运营事实的本地沉淀层
- 平台对这些事实形成的最小状态层
- Reply Agent、审核、报表和审计可共同复用的可信输入层

第一阶段真正要证明的，不是“已经有复杂数据系统”，而是：

- 平台已经开始积累自己的本地运营资产
- 平台已经能把原始记录转成最小状态快照
- Agent 和报表已经不再完全依赖实时临时抓取

## 3. 本工作包要解决的问题

### 3.1 让历史成为资产

第一阶段必须开始沉淀：

- 历史推文
- 历史评论 / 提及 / 回复
- 任务执行结果
- 账号和实例的关键运行状态

这些历史信息应成为平台可持续积累的资产，而不是临时上下文。

### 3.2 让状态成为可复用输入

第一阶段不要求做复杂分析体系，但必须开始形成：

- 最小状态快照
- 最小趋势判断基础
- 基础运行状态输出

这保证 Agent 和报表后续不必每次都直接从原始事件推理。

### 3.3 让平台具备事实层与状态层

第一阶段至少要把两层分开：

- 事实层：原始互动、原始推文、原始任务结果
- 状态层：平台可复用的结构化运行状态和最小快照

第一阶段不要求建立完整派生层体系，但必须为后续派生层保留边界。

### 3.4 让关键结果形成回流闭环

第一阶段的数据工作不只是“采集输入”，还必须承接关键结果回流：

- 候选回复采用结果
- 审核结果
- 执行结果
- 接管结果

只有回流链路成立，平台的数据底座才不是单向采集层，而是完整闭环的一部分。

## 4. 第一阶段交付目标

工作包 B 在第一阶段至少应交付以下结果：

### 4.1 历史互动落库

至少需要沉淀：

- 评论
- 提及
- 历史回复
- 与回复处理相关的上下文信息

### 4.2 历史推文落库

至少需要沉淀：

- 已发布推文
- 与当前账号相关的重要历史内容
- 后续内容效果分析需要的基础历史记录

### 4.3 任务结果落库

至少需要沉淀：

- 生成过什么候选回复
- 最终采用了哪个版本
- 是否发送成功
- 是否进入审核
- 是否失败、升级或被人工接管

### 4.4 最小事实层与状态层分层

至少需要做到：

- 平台能区分原始互动记录和结构化状态输出
- Reply Agent、审核、报表和审计读取的是同一组可信平台数据
- 第一阶段不把扩展能力派生结果混入平台核心层

### 4.5 最小状态快照

至少需要形成：

- 账号基础运行状态
- 实例基础运行状态
- 执行通道基础状态
- 与 Reply Agent 直接相关的最小互动状态快照

### 4.6 基础运行状态输出

平台至少能够输出：

- 当前有哪些数据已被沉淀
- 当前哪些账号和实例处于可运行状态
- 当前有哪些任务结果可被追踪

### 4.7 关键结果回流

至少需要支持：

- 候选回复采用结果回流
- 审核结果回流
- 执行结果回流
- 接管结果回流

## 5. 范围边界

### 5.1 本工作包负责什么

工作包 B 负责：

- 本地历史数据沉淀
- 最小事实层与状态层分层
- 最小状态快照
- 事实层与状态层的最小分层
- 为 Agent、报表和审计提供统一输入底座
- 关键结果回流底座

### 5.2 本工作包不负责什么

工作包 B 不负责：

- 复杂趋势分析模型
- 高级多维报表体系
- 完整风险主题聚类
- 完整派生层体系
- 沙箱派生数据空间
- 自动化策略扩展
- 高级数据仓库或复杂指标体系

### 5.3 本工作包与其他工作包的边界

- 与工作包 A 的边界：A 提供宿主和对象入口，B 负责让这些对象有历史和状态数据可承载
- 与工作包 C 的边界：B 提供数据与状态输入，C 负责 AI 与知识编排
- 与工作包 D 的边界：B 提供 Reply Agent 所需的历史与状态基础，D 负责基于这些输入做角色化回复生成
- 与工作包 F 的边界：B 提供任务、审核和接管所需的历史与回流数据，F 负责治理动作本体
- 与工作包 G 的边界：B 提供报表可复用数据底座，G 负责业务结果展示

## 6. 核心设计约束

### 6.1 优先沉淀业务闭环直接需要的数据

第一阶段不追求大而全的数据仓库，应优先沉淀与 Reply Agent 闭环直接相关的数据资产。

### 6.2 事实层与状态层必须区分

第一阶段即使不展开完整层次，也必须明确：

- 原始记录不能等同于状态结论
- 状态输出不能反向篡改原始事实

### 6.3 本地历史数据优先于临时抓取

第一阶段应优先让 Agent 和报表消费本地已沉淀的数据，而不是默认每次都从实时通道抓取。

### 6.4 为后续派生层和扩展能力预留边界

第一阶段的数据底座必须允许后续：

- 扩展能力只读平台核心数据
- 扩展能力在自己的派生层保存结果

但第一阶段本工作包本身不承担扩展层实现。

### 6.5 平台可信数据必须先于高级分析出现

第一阶段可以不做复杂分析，但必须先建立一层可信平台数据输入，否则后续 Agent、报表和扩展能力都会围绕不稳定输入演化。

### 6.6 数据底座的价值在于复用，而不是堆积

第一阶段沉淀的数据，必须能被 Reply Agent、审核、审计和报表共同消费，而不是分别维护各自一套记录。

## 7. 依赖关系

### 7.1 本工作包的前置依赖

工作包 B 依赖工作包 A 提供：

- 客户空间边界
- 账号资产入口
- 实例入口
- 执行通道入口
- 状态展示挂载位置

### 7.2 本工作包的输出依赖

工作包 B 的输出会被下列工作包直接依赖：

- 工作包 D：Reply Agent 读取历史与状态
- 工作包 E：任务执行结果回写与追踪
- 工作包 F：审核与审计读取执行历史
- 工作包 G：生成基础报表和客户可读结果
- 工作包 H：后续扩展能力只读平台核心数据

## 8. 完成标准

工作包 B 完成至少应满足：

- 平台能够稳定保存历史互动与历史推文
- 任务结果可回写并可查询
- 平台能够承接关键治理和执行结果回流
- 至少存在一组可被 Reply Agent 和报表复用的最小状态快照
- 平台可以区分原始记录和状态输出
- 第一阶段业务闭环不再依赖纯实时临时数据

## 9. 后续补充方向

后续补充应优先围绕：

- 第一阶段最小数据资产范围
- 第一阶段关键结果回流范围
- 状态快照的节奏与刷新规则
- 风险与内容效果的最小沉淀口径
- 审计数据与业务数据的边界
- 平台核心数据与扩展派生数据的关系
