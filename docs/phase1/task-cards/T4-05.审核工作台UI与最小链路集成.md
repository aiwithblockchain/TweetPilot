# T4-05: 审核工作台 UI 与最小链路集成

- 版本: `v0.2.0`
- 创建日期: `2026-04-10`
- 文档状态: `草稿`
- 对应切片: Slice 4 - 风险分流与审核队列
- 前置依赖: `T4-01`, `T4-03`, `T4-04`

## 1. 任务目标

实现 Slice 4 的审核工作台界面，把任务列表、上下文详情、审核动作、责任归属和人工接管串成一条可操作的最小治理链路。

## 2. 核心交付物

### 2.1 领域对象

- 复用 `ReviewQueueItem`、`ReplyTask` 和 `TaskAssignment`。
- 本任务不新增核心领域模型。

### 2.2 仓储接口

- 复用 `IReplyTaskRepository`、`commentInputRepository`、`candidateReplyRepository`。
- 通过查询服务而非组件内部拼装复杂数据。

### 2.3 服务层

- `src/features/review-queue/useReviewQueue.ts`
- `src/features/review-queue/useTaskGovernance.ts`

export interface ReviewQueueViewModel {

UI 最小要求：

- 左侧待审核任务列表
- 右侧任务详情，包括评论输入、候选回复、风险标签、责任归属、事件时间线
- 审核通过、拒绝、重新入队、指派、人工接管按钮
- 加载中显示骨架屏，查询失败显示可恢复错误态

## 3. 实施步骤

1. 设计审核工作台页面结构和 hooks，避免在组件内直接拼接多仓储逻辑。
2. 将详情区域拆分为 `TaskMetadata`、`TaskTimeline`、`TaskGovernanceActions` 等子组件。
3. 实现待审核任务列表、骨架屏、错误态和任务详情视图。
4. 接入审核、指派、人工接管动作，并在成功后刷新列表与详情。
5. 在主导航或挂载位中增加审核队列入口。
6. 补齐集成测试和 E2E 测试，验证完整最小链路。

## 4. 验收标准

- [ ] 平台中存在明确的审核队列入口。
- [ ] 用户能看到待审核任务列表和关联上下文。
- [ ] 用户能在界面中执行审核、指派和接管动作。
- [ ] 动作完成后，列表和详情会反映最新状态与责任归属。
- [ ] 详情组件已按元数据、时间线、动作区拆分，避免单组件承载过多逻辑。
- [ ] UI 对加载态和错误态有明确展示。

## 5. 测试要求

### 5.1 单元测试

- 验证 `useReviewQueue()` 的加载、错误和刷新状态。
- 验证治理动作 hook 在成功后会触发列表刷新。
- 验证 `ReviewQueueSkeleton` 和 `ReviewQueueErrorState` 的渲染条件。
- 验证详情视图正确展示风险、责任归属和事件时间线。
- 验证 `TaskMetadata`、`TaskTimeline`、`TaskGovernanceActions` 的分层渲染。

### 5.2 集成测试

- 验证审核队列视图能读取 `ReviewQueueItem` 并渲染详情。
- 验证审批通过后列表移除该任务。
- 验证指派和接管后详情区域显示最新责任归属。
- 验证分页参数变化时列表和总数同步刷新。

### 5.3 端到端测试

- 验证“候选回复生成 -> 创建任务 -> 高风险进入审核队列 -> 人工接管”最小链路。
- 验证低风险任务不会出现在审核队列中。
- 验证权限不足时 UI 会展示明确错误提示而不是静默失败。
