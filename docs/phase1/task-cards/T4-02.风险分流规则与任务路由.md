## 1. 任务目标

实现第一阶段最小风险分流规则，让任务能够根据风险标记进入“待审核”或“待后续处理”路径，并把路由结果写回任务状态。

## 2. 核心交付物

### 2.1 领域对象

- `src/domain/taskRouting.ts`
- `TaskRoutingDecision`
- `TaskRiskRule`

```typescript
export interface TaskRoutingDecision {
  taskId: string;
  riskLevel: "low" | "medium" | "high";
  route: "pending_review" | "ready_for_execution";
  reason: string;
}

export interface TaskRiskRule {
  riskLevel: "low" | "medium" | "high";
  defaultRoute: "pending_review" | "ready_for_execution";
  fallbackRoute: "pending_review";
}
```

### 2.2 仓储接口

- 复用 `IReplyTaskRepository`。
- 可选增加 `findByRoute(route)`，便于后续页面读取。

### 2.3 服务层

- `src/services/taskRoutingService.ts`

```typescript
export interface TaskRoutingService {
  routeTask(taskId: string, actorId: string): Promise<TaskRoutingDecision>;
}
```

规则约束：

- `high` 风险必须进入 `pending_review`。
- `low` 风险进入 `ready_for_execution`。
- `medium` 风险第一阶段默认进入 `pending_review`，防止风险边界过松。
- `medium` 风险规则必须通过配置对象声明，不能直接散落在条件分支里。
- 路由失败时统一回退到 `pending_review`，保证高风险不会漏出治理边界。
- 所有路由都必须写入 `risk_routed` 事件。

### 2.4 错误处理策略

- `TASK_ROUTE_NOT_FOUND`
- `TASK_ROUTE_RULE_MISSING`
- `TASK_ROUTE_FALLBACK_APPLIED`

### 2.5 UI 组件

- 本任务不落完整 UI。
- 可在任务详情或调试面板中保留最小路由结果展示位。

## 3. 实施步骤

1. 定义最小风险规则对象和路由决策对象。
2. 实现 `TaskRoutingService`，读取任务并根据风险等级产出路由结果。
3. 将中风险策略做成可配置规则对象，但默认仍保持保守审核策略。
4. 将路由结果回写到任务 `route`、`status` 和 `events`。
5. 为路由失败补齐回退策略，默认回退到 `pending_review`。
6. 补齐路由服务单元测试与集成测试。

## 4. 验收标准

- [ ] 高风险任务不会进入可继续处理路径。
- [ ] 中风险任务默认进入审核路径，规则清晰且可测试。
- [ ] 低风险任务会进入 `ready_for_execution` 状态。
- [ ] 路由结果和原因能在任务事件中被追踪。
- [ ] 路由失败时任务会被保守地送入审核路径。

## 5. 测试要求

### 5.1 单元测试

- 验证 `low` 风险路由为 `ready_for_execution`。
- 验证 `medium` 和 `high` 风险路由为 `pending_review`。
- 验证任务不存在时服务返回 `TASK_ROUTE_NOT_FOUND`。
- 验证路由动作会追加 `risk_routed` 事件。
- 验证规则缺失或异常时返回回退结果并标记 `TASK_ROUTE_FALLBACK_APPLIED`。

### 5.2 集成测试

- 验证“候选回复任务化 -> 风险路由”链路可跑通。
- 验证高风险任务能被 `findPendingReview()` 读取。
- 验证低风险任务不会出现在审核队列结果中。
- 验证中风险策略切换时，读取结果跟随规则变化。
