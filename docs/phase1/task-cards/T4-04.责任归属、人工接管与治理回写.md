## 1. 任务目标

建立任务责任归属、人工接管和治理结果回写能力，确保高风险任务不会停留在无人负责状态，并形成第一阶段最小审计链路。

## 2. 核心交付物

### 2.1 领域对象

- `src/domain/taskOwnership.ts`
- `TaskAssignment`
- `TaskTakeoverResult`

```typescript
export interface TaskAssignment {
  taskId: string;
  assigneeId: string;
  assignedBy: string;
  assignedAt: Date;
}

export interface TaskTakeoverResult {
  taskId: string;
  actorId: string;
  action: "take_over" | "complete_takeover";
  result?: "ready_for_execution" | "rejected" | "completed";
  expectedVersion?: number;
  note?: string;
}
```

### 2.2 仓储接口

- 复用 `IReplyTaskRepository`。
- 不新增独立审计仓储，审计通过 `ReplyTask.events` 最小化沉淀。

### 2.3 服务层

- `src/services/taskOwnershipService.ts`

```typescript
export interface TaskOwnershipService {
  assignTask(
    taskId: string,
    assigneeId: string,
    actorId: string,
    options?: { force?: boolean },
  ): Promise<ReplyTask>;
  takeOverTask(
    taskId: string,
    actorId: string,
    note?: string,
    options?: { checkPermission?: boolean; expectedVersion?: number },
  ): Promise<ReplyTask>;
  completeTakeover(
    taskId: string,
    actorId: string,
    result: "ready_for_execution" | "rejected" | "completed",
    note?: string,
  ): Promise<ReplyTask>;
}
```

服务职责：

- 指派任务并记录 `task_assigned` 事件。
- 发起人工接管并记录 `task_taken_over` 事件。
- 接管前必须进行最小权限检查，默认只允许当前指派人、任务接管人或管理员操作。
- 接管时必须进行版本或时间戳校验，避免多人同时接管同一任务产生冲突覆盖。
- 将治理结果回写到任务状态和事件流。

### 2.4 错误处理策略

- `TASK_ASSIGNMENT_PERMISSION_DENIED`
- `TASK_TAKEOVER_PERMISSION_DENIED`
- `TASK_TAKEOVER_CONFLICT`
- `TASK_TAKEOVER_RESULT_INVALID`

### 2.5 UI 组件

- 本任务只定义动作接口和状态回写结果。
- UI 在 `T4-05` 中落地。

## 3. 实施步骤

1. 定义责任归属和人工接管相关值对象。
2. 实现指派服务，支持把待审核任务分配给具体处理人，并支持受控重指派。
3. 实现人工接管服务，支持接管中和接管完成两类状态。
4. 为接管动作加入最小权限检查和版本冲突校验。
5. 明确 `completeTakeover()` 后只允许进入 `ready_for_execution`、`rejected` 或 `completed` 三种结果状态。
6. 将指派和接管动作统一写入任务事件流，形成最小审计记录。
7. 补齐动作约束、回写逻辑和集成测试。

## 4. 验收标准

- [ ] 任务可以被指派给具体处理人。
- [ ] 人工接管动作会改变任务状态并记录操作者。
- [ ] 接管完成结果只会回写到允许的结果状态。
- [ ] 未授权的操作者无法接管或重指派任务。
- [ ] 多人并发接管时系统会检测冲突并拒绝后到请求。
- [ ] 任务事件流能反映创建、路由、审核、指派、接管等关键治理动作。

## 5. 测试要求

### 5.1 单元测试

- 验证 `assignTask()` 会写入 `assigneeId` 和 `task_assigned` 事件。
- 验证无权限操作者调用 `assignTask()` 或 `takeOverTask()` 时返回权限错误。
- 验证 `takeOverTask()` 会将任务置为 `in_takeover`。
- 验证版本不匹配时返回 `TASK_TAKEOVER_CONFLICT`。
- 验证 `completeTakeover()` 会记录完成信息并只写入允许的结果状态。
- 验证非法场景，例如对 `rejected` 任务再次接管，会返回明确失败。

### 5.2 集成测试

- 验证待审核任务可被指派并在再次读取时看到责任归属。
- 验证接管动作会写入事件流并可从任务详情中读取。
- 验证审核动作与接管动作组合执行后，状态回写保持一致。
- 验证两名操作者并发接管同一任务时，只有一个请求成功。
