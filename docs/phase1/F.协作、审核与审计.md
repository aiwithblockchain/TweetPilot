## 1. 文档目的

本文档用于拆解第一阶段工作包 F：协作、审核与审计。

本文档只定义：

- 第一阶段最小团队治理能力需要做到什么程度
- 第一阶段审核与人工接管的边界是什么
- 第一阶段最小权限分层与审计能力需要做到什么程度
- Agent 可观测性在第一阶段的最小落点是什么

本文档不展开权限模型细节、审计字段和日志实现。

## 2. 工作包定位

工作包 F 负责让第一阶段从“可运行 demo”变成“团队可用系统”。

它解决的问题不是“能不能回复”，而是：

- 谁来审核
- 谁来接管
- 哪些任务必须升级
- 谁拥有哪类权限
- 关键动作如何被审计
- Agent 的关键运行状态如何被观察

如果工作包 F 不成立，第一阶段就只能作为个人演示系统存在，很难真正进入代运营团队和企业场景。

## 2A. 第一阶段治理形态

第一阶段的协作、审核与审计，不应被定义为“补充页面”或“运营后台附属功能”。

它更准确的产品形态应是：

- 平台对高风险任务的最小治理链路
- 团队成员接手 Agent 结果的协作入口
- 平台对关键动作保留最终解释能力的记录层

第一阶段真正要证明的，不是“流程有多复杂”，而是：

- 高风险任务不会绕过人工边界
- 团队可以接住 Agent 生成结果
- 平台可以解释关键任务是如何被处理的

## 3. 本工作包要解决的问题

### 3.1 建立最小审核队列

第一阶段必须有一个明确入口，让高风险任务可以进入人工审核流程。

### 3.2 建立最小协作流

第一阶段即使不做复杂团队流，也必须有：

- 基础指派
- 基础人工接管
- 高风险任务升级

### 3.3 建立最小权限分层边界

第一阶段至少需要明确：

- 谁能看什么
- 谁能审什么
- 谁能发什么
- 谁能接管什么

### 3.4 建立最小审计能力

第一阶段必须能够回答：

- 某次回复是谁触发的
- 用了哪个角色和哪个执行通道
- 是否经过审核
- 最终是谁执行或接管的

### 3.5 建立最小 Agent 可观测性

第一阶段必须至少能够看到：

- 某个任务是否经过 Agent 生成
- Agent 生成是否成功
- Agent 输出结果是否被采用
- Agent 结果进入了哪个后续路径

### 3.6 让治理链路形成闭环

第一阶段的治理能力不应只是“多一个审核页面”，而应形成最小闭环：

- 高风险任务进入审核
- 审核结果进入执行或接管路径
- 执行或接管结果进入审计记录
- 审计结果进入报表与交付输出

## 4. 第一阶段交付目标

工作包 F 在第一阶段至少应交付以下结果：

### 4.1 基础审核队列

至少需要支持：

- 让高风险任务进入审核列表
- 让团队成员看到待处理审核项
- 让审核动作与任务结果产生关联

### 4.2 基础处理责任归属

至少需要支持：

- 一个任务存在明确处理责任
- 审核、执行和接管环节存在明确责任归属
- 第一阶段任务不会停留在无人负责的中间状态

### 4.3 基础指派与人工接管

至少需要支持：

- 任务可被指派
- 任务可被人工接管
- 接管结果可被追踪

### 4.4 高风险任务升级

至少需要支持：

- 高风险任务进入升级路径
- 升级结果可被记录
- 升级后的处理结果可进入审计与结果输出

### 4.5 最小权限分层边界

至少需要支持：

- 在第一阶段区分基础查看、审核、执行和管理边界
- 让协作流不依赖所有人拥有同等权限

### 4.6 基础审计记录

至少需要支持：

- 关键任务有审计记录
- 审核、接管和执行结果可被追踪
- 后续报表和客户交付可读取审计结果

### 4.7 Agent 基础可观测性

至少需要支持：

- 查看 Agent 是否参与某任务
- 查看 Agent 结果是否被采用
- 查看 Agent 输出结果进入了哪个流程分支

### 4.8 治理结果回流

至少需要支持：

- 审核结果回流到任务体系
- 接管结果回流到任务体系
- 审计结果进入报表与交付体系

## 5. 范围边界

### 5.1 本工作包负责什么

工作包 F 负责：

- 审核队列
- 基础处理责任归属
- 基础协作流
- 最小权限分层边界
- 基础审计记录
- Agent 基础可观测性
- 治理结果回流

### 5.2 本工作包不负责什么

工作包 F 不负责：

- 完整企业权限体系
- 复杂组织角色体系
- 高级审计分析系统
- 完整客户交付门户
- 多客户空间的大规模协作治理
- 报表与客户可读结果的最终组织方式

### 5.3 本工作包与其他工作包的边界

- 与工作包 E 的边界：E 负责把任务和执行结果送入治理流程，F 负责治理本体
- 与工作包 G 的边界：F 负责沉淀审核、接管、升级和审计结果，G 负责把这些结果组织成可读输出

## 6. 核心设计约束

### 6.1 高风险动作不能绕过治理流程

第一阶段即使只做最小协作流，也必须保证高风险任务存在人工审核和接管路径。

### 6.2 权限边界必须早于复杂功能出现

第一阶段即使不做完整权限体系，也必须先建立最小权限边界，否则后续协作流会被迫返工。

### 6.3 审计必须是平台级能力

审计不应作为某个页面上的附属信息存在，而应作为平台可追踪能力存在。

### 6.4 Agent 可观测性必须进入第一阶段

第一阶段不能只让 Agent 生成结果，而看不到它的参与情况和采用情况。

### 6.5 审核和接管必须以任务为中心，而不是以页面为中心

第一阶段不应把审核和接管做成脱离任务体系的孤立功能。所有治理动作都应围绕任务对象发生，并可回写任务结果。

### 6.6 治理能力的目标是“可控协同”，不是“流程复杂化”

第一阶段治理链路应尽量短，但必须有效。重点是把高风险动作拦住，把责任归属建立起来，而不是一开始做复杂审批系统。

## 7. 依赖关系

### 7.1 本工作包的前置依赖

工作包 F 依赖：

- 工作包 D 提供候选回复与风险分流结果
- 工作包 E 提供任务对象、执行状态和执行结果

### 7.2 本工作包的输出依赖

工作包 F 的输出会被下列工作包直接依赖：

- 工作包 G：需要汇总审核、接管、审计和 Agent 采用结果
- 工作包 H：后续扩展能力的自动运行与展示挂载，也需要建立在平台治理能力之上

## 8. 完成标准

工作包 F 完成至少应满足：

- 高风险任务可以进入审核队列
- 高风险任务存在明确责任归属
- 任务可以被基础指派和人工接管
- 高风险任务可以进入升级路径
- 最小权限分层边界可用
- 基础审计记录可用
- Agent 基础可观测性可用
- 治理结果可以回流到任务和结果输出体系

## 9. 后续补充方向

后续补充应优先围绕：

- 审核队列设计
- 责任归属与接管规则
- 任务接管和升级规则
- 权限分层方式
- 审计结果组织方式
- Agent 可观测性面板设计
