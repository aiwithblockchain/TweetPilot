# 新窗口打开工作目录问题分析

## 文档信息
- **版本**: v1.0
- **创建时间**: 2026-04-27
- **目的**: 单独梳理 `Open in New Window...` / “新窗口打开工作目录”能力当前暴露的问题、根因判断、临时产品策略，以及后续独立重构方向。
- **当前结论**: 该能力暂不继续修补，先从产品入口移除；当前版本只收口“单窗口切换工作目录”主链路。

---

## 一、问题定义

这里讨论的不是普通的“多开一个窗口”，而是以下目标能力：

> 用户在现有窗口之外，再打开一个新窗口，并让这个新窗口独立进入另一个工作目录，同时不影响原窗口的 workspace、AI 会话、任务、定时器、文件浏览与工具调用。

用户对这个能力的直觉预期类似 VS Code：

- 窗口 A 绑定 workspace A
- 窗口 B 绑定 workspace B
- 两个窗口互不影响
- 关闭某个窗口时，只清理该窗口自己的运行时资源

当前 TweetPilot 还没有完整满足这个预期。

---

## 二、当前观察到的实际问题

在最近的真实使用和日志排查中，已经暴露出以下问题。

### 2.1 新窗口初始化体验不正确

当前通过 `set-initial-workspace` 打开新工作目录时，新窗口不会直接稳定进入目标工作区，而是：

1. 先显示首页
2. 首页里能看到目标工作目录
3. 然后再进入该工作区，或者表现出中间态

这说明新窗口启动链路与 workspace 初始化链路并没有形成一个完整、原子、明确的“直接进入目标工作区”流程。

### 2.2 AI 会话可能跨窗口串状态

已实际出现过：

- A 窗口打开 workspace A
- B 窗口打开 workspace B
- 回到 A 操作历史会话时，第一次闪过错误
- 第二次点击后，直接显示 B 的 AI 对话内容

这类问题说明：

- 会话列表、当前 workspace、active session、active request 等状态，只要有一层没有真正按窗口隔离，就会发生跨窗污染。

### 2.3 Explore / 文件访问能力会绑定错误工作目录

用户已经反馈：

- 在 Explore 查看文件时，提示“目标路径不在当前工作目录内”

这类现象说明，多窗口下“当前工作目录”的判定仍可能落到错误窗口或错误上下文上。

AI、文件浏览、文档读取、工作区文件操作，本质上都依赖同一个前提：

> 当前这个窗口，到底绑定的是哪个 workspace。

只要这个归属关系不稳定，上层工具就都会表现异常。

### 2.4 多窗口与任务 / 定时器边界没有真正闭合

目前的设计文档已经说明：

- `TaskState` 仍然更偏单 context 模型
- `WorkspaceContext` 仍然持有 db / executor / timer_manager
- 定时器与工作区生命周期强耦合

这意味着即使前端窗口表现正常，后端运行时资源是否真正做到“一个窗口一个完整 workspace context”，仍然没有彻底完成。

### 2.5 问题会误导用户

`Open in New Window...` 最危险的地方，不是它一定立刻崩，而是它会制造一种“这个能力应该可以正常使用”的错觉。

实际情况是：

- UI 能开出窗口
- 但资源归属未必正确
- 某些操作可用，某些操作会串状态
- 错误往往在用户进行第二步、第三步操作时才出现

这类问题最伤，因为它不是明确不可用，而是“看起来能用，实际上不可信”。

---

## 三、为什么这不是一个小 bug

从表象看，它像是几个分散问题：

- 新窗口先显示首页
- AI 串会话
- Explore 提示路径不在当前工作目录内
- 文档读取或文件操作落到错误目录

但这些问题本质上属于同一类架构问题：

```text
窗口身份
  → 当前 workspace 归属
    → AI runtime 归属
    → 文件工具归属
    → task/timer/db context 归属
```

只要这条链路中有任何一层仍然是：

- app-global 单例
- 假隔离
- label 不稳定
- 初始化顺序不确定
- 清理流程不完整

最终都会表现成“多窗口随机串状态”。

所以它不是一个适合继续局部打补丁的问题，而是一个需要单独定义边界、再整体推进的重构任务。

---

## 四、当前根因判断

### 4.1 当前系统真实擅长的是单窗口切换 workspace

经过最近几轮修改，系统已经越来越接近这个稳定模型：

- 应用启动后进入首页
- 用户在当前窗口里选择工作目录
- 当前窗口切换到目标 workspace
- AI / recent / settings / 当前会话围绕这个窗口内的 workspace 工作

这条路径虽然还需要继续收口，但边界已经清晰，而且可以验证。

### 4.2 “新窗口打开新工作目录”需要窗口级完整隔离

这个能力要求的不是“再开一个 Webview”这么简单，而是至少要有：

1. **窗口级 current workspace state**
2. **窗口级 AI runtime state**
3. **窗口级 task/timer/workspace context**
4. **窗口级文件/文档/Explore 工具归属**
5. **窗口关闭时只清理对应窗口资源**
6. **新窗口初始化过程直接进入目标 workspace，而不是先首页再跳转**

只要其中任何一项没有闭合，多窗口能力就不可信。

### 4.3 当前后端状态模型还没有完整升级到这个层级

虽然近期已经做过一部分 per-window 改造，但整体上仍然没有形成完整的：

```text
TaskState / Runtime State / AI State / WorkspaceContext / Tool Access
全部按 WindowLabel 稳定绑定
```

因此现在继续暴露新窗口入口，等于让用户踩进一个“部分隔离、部分共享”的中间态系统。

## 五、短期产品策略

### 5.1 当前版本只承诺单窗口 workspace 切换

当前正式支持的能力应明确收口为：

- 使用 `Open...` 在当前窗口打开或切换工作目录
- recent workspace 仅作为历史记录和单窗口切换入口
- 所有稳定性验证都围绕这条主链路进行

### 5.2 暂时移除 `Open in New Window...`

短期内不再把 `Open in New Window...` 作为公开可用能力保留在菜单中。

原因：

- 它会误导用户以为多窗口独立工作区已经可用
- 它会持续制造跨窗串状态问题
- 它会干扰当前主线收口
- 它对应的是一个独立架构任务，不应继续混在单窗口修复中处理

### 5.3 当前修复边界

当前阶段只继续处理：

- 单窗口切换 workspace 的正确性
- 单窗口 AI 会话与 workspace 的一致性
- 单窗口 recent / settings / 文件访问行为

不再把多窗口问题混入这一轮收尾。

---

## 六、后续独立重构应覆盖的范围

如果未来要重新支持“新窗口打开新工作目录”，建议作为独立任务推进，至少覆盖以下范围。

### 6.1 运行时状态模型

需要系统性检查并统一：

- `RuntimeWorkspaceState`
- `AiState`
- `TaskState`
- `WorkspaceContext`
- 可能存在的其他 app-global 状态

目标是明确：

```text
每个窗口 = 一套稳定的 workspace runtime 归属
```

### 6.2 新窗口初始化链路

需要重新设计：

- 新窗口创建
- 初始事件投递
- workspace 初始化
- 前端 ready 状态
- 是否 reload
- 是否先显示首页

目标是做到：

- 新窗口直接进入目标工作目录
- 中间态可控
- 不出现“先首页再跳转”的错觉路径

### 6.3 工具与文件访问归属

需要核对所有依赖当前 workspace 的能力，例如：

- Explore
- 文档读取
- workspace 文件读写
- AI 工具调用中的路径校验
- 其他 workspace-scoped command

确保它们取到的是“当前这个窗口的 workspace”，而不是某个全局状态或别的窗口状态。

### 6.4 task / timer / db 生命周期

需要决定是否真正升级到类似：

```text
HashMap<WindowLabel, WorkspaceContext>
```

否则多窗口下仍会有以下风险：

- 切换一个窗口时影响另一个窗口的 timers
- 数据库连接归属不清
- executor / context 生命周期互相覆盖
- 窗口关闭时无法精确释放资源

### 6.5 测试策略

不能只靠手工点几次，需要有明确回归矩阵：

1. 窗口 A workspace A，窗口 B workspace B
2. 两边分别执行 AI 对话
3. 两边分别打开历史会话
4. 两边分别浏览文件 / 文档
5. 关闭其中一个窗口
6. 验证另一个窗口状态不受影响
7. 验证 task / timer / db 不串状态

---

## 七、推荐结论

### 当前结论

现在不应该继续把“新窗口打开工作目录”当作顺手可修的小问题。

更合理的做法是：

1. **立即移除 `Open in New Window...` 菜单入口**
2. **把多窗口独立工作区视为单独架构任务**
3. **当前版本只收口单窗口切换 workspace 主链路**

### 为什么这样更对

因为这会让当前产品边界重新变清楚：

- 用户知道什么是正式支持的
- 我们知道什么需要继续修
- 日志、测试、回归也都有清晰范围

不然所有“AI 问题”“Explore 问题”“文档路径问题”都会继续混在一起，浪费时间，而且很难得到稳定结论。
