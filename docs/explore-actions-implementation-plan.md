## 背景

当前 Explore / Explorer 区域顶部的 3 个操作按钮分别为：

1. 新建文件
2. 新建文件夹
3. 刷新工作区

经过重新核查运行时行为与代码链路，当前问题需要更准确地定义为：

- 3 个按钮在代码层面已经存在配置与处理入口
- 刷新按钮具备基础刷新逻辑
- 新建文件、新建文件夹虽然已接入事件处理，但运行时点击后无可见反馈
- 根因不是后端缺少创建能力，而是前端创建流程依赖 `window.prompt` 获取名称
- 在当前 Tauri / WebView 运行环境中，`window.prompt` 未产生有效交互，导致创建流程在入口处中断

因此，用户实际体验上会认为“新建文件”和“新建文件夹”按钮没有实现。这份文档基于这个修正后的结论，给出后续实施方案。

## 当前问题分析

### 1. 按钮配置与点击链路是存在的

文件：`src/config/layout.ts`

Workspace 侧边栏已定义 3 个 action：

- `new-file`
- `new-folder`
- `refresh-workspace`

文件：`src/components/LeftSidebar.tsx`

按钮点击后会统一调用：

- `onAction?.(action.id)`

文件：`src/App.tsx`

主界面将侧边栏 action 绑定到：

- `handleSidebarAction`

这说明按钮并不是纯展示控件。

### 2. 创建逻辑在运行时卡在 `window.prompt`

文件：`src/hooks/useAppLayoutState.ts`

当前逻辑：

- `new-file` → `handleWorkspaceCreate('file')`
- `new-folder` → `handleWorkspaceCreate('folder')`

而 `handleWorkspaceCreate(...)` 在真正调用创建接口之前，先执行：

- `window.prompt(...)`

用于让用户输入名称。

这意味着：

1. 只要 `window.prompt` 不弹出
2. 就拿不到文件名/文件夹名
3. 也就不会继续调用 `workspaceService.createFile` 或 `workspaceService.createFolder`
4. 用户侧看起来就是“点击没有任何反应”

### 3. 后端创建能力已经具备，但前端没有真正走到那里

前端服务文件：`src/services/workspace/tauri.ts`

已实现：

- `createFile`
- `createFolder`
- `listDirectory`
- `readFile`
- `getFolderSummary`

后端文件：`src-tauri/src/commands/workspace.rs`

已实现：

- `create_workspace_file`
- `create_workspace_folder`
- `list_workspace_directory`
- `read_workspace_file`
- `get_workspace_folder_summary`

所以问题不是“没有创建接口”，而是“当前前端交互入口失效，导致创建接口没有被触发”。

### 4. 当前准确结论

当前 Explore 区域这 3 个按钮的真实状态应定义为：

- **刷新按钮**：已实现基础功能
- **新建文件按钮**：代码链路存在，但运行时创建入口失效，用户体验等同于未实现
- **新建文件夹按钮**：代码链路存在，但运行时创建入口失效，用户体验等同于未实现

### 5. 当前版本的核心问题

当前版本主要问题：

- 新建文件/文件夹依赖 `window.prompt`
- `window.prompt` 在当前桌面运行环境中未产生有效交互
- 创建流程因此无法继续执行
- 缺少创建失败或交互失效时的显式反馈
- 刷新逻辑仅覆盖已展开目录，完整性不足
- 缺少更完整的 Explorer 常见能力，例如重命名、删除、右键菜单

## 目标

将当前基础版能力升级为可交付的 Explorer 工具栏交互，目标包括：

- 保留现有前后端链路
- 修复新建文件、新建文件夹按钮点击无响应的问题
- 升级操作交互体验
- 提升刷新的一致性和可预期性
- 补齐状态反馈
- 为后续扩展右键菜单、重命名、删除等能力预留清晰结构

---

## 实施原则

1. **优先修复真实阻塞点**：先替换 `window.prompt`，解决“点击无反应”的实际问题。
2. **优先复用现有链路**：不推翻现有 `workspaceService` 与 Tauri 命令。
3. **先改善体验，再扩展能力**：优先解决输入、刷新、状态反馈问题。
4. **保持改动收敛**：主要修改 Explorer 相关组件和工作区状态管理，不做无关重构。
5. **前后端职责清晰**：前端负责交互与即时校验，后端负责文件系统安全与最终约束。

## 分阶段实施方案

# Phase 1：修复点击无响应并升级交互

## 1. 替换 `window.prompt`，改为内联输入

### 当前问题

目前通过 `window.prompt` 获取名称，存在以下问题：

- 在当前 Tauri / WebView 环境中无法提供可用交互
- 用户点击按钮后没有可见反馈
- 无法提供实时校验
- 不支持键盘流畅操作，如 Enter 提交、Esc 取消
- 不符合 Explorer/IDE 的交互习惯

### 改造方案

在左侧 Explore 区域增加临时创建态：

- `creatingType: 'file' | 'folder' | null`
- `draftName: string`
- `creatingParentPath: string | null`

交互流程：

1. 点击“新建文件”或“新建文件夹”
2. 在目标父目录下方插入一个临时输入行
3. 自动聚焦输入框
4. Enter 提交创建
5. Esc 取消创建
6. 创建成功后自动选中新节点
7. 创建失败时在当前区域或 toast 中给出明确反馈

### 涉及文件

- `src/components/LeftSidebar.tsx`
- `src/hooks/useAppLayoutState.ts`

### 验收标准

- 不再使用 `window.prompt`
- 点击新建文件/新建文件夹后，界面立即出现输入控件
- 可以通过 Enter 提交、Esc 取消
- 创建成功后新节点立即出现在树中并被选中
- 创建失败时有明确提示，不再表现为“没反应”

---

## 2. 增加前端名称校验

### 当前问题

当前校验主要依赖后端返回错误，用户只能在提交后得到反馈。

### 改造方案

前端增加基础校验规则：

- 名称不能为空
- 不能为 `.` 或 `..`
- 不能包含 `/` 或 `\\`
- 长度不能超过合理上限，例如 255
- 对当前目录下重名进行即时检测

后端仍保留最终校验，形成双层防线。

### 涉及文件

- `src/hooks/useAppLayoutState.ts`
- 可新增轻量校验工具，例如：`src/services/workspace/validators.ts`

### 验收标准

- 非法输入在前端立即提示
- 已存在同名项时可提前阻止提交
- 后端错误仍可兜底提示

---

## 3. 刷新改为真正重载工作区树

### 当前问题

当前刷新逻辑主要针对已展开目录执行强制加载，存在感知问题：

- 用户可能认为“刷新了但没完全刷新”
- 外部变更同步不够直观
- 对未展开目录的状态恢复不够明确

### 改造方案

刷新时执行以下步骤：

1. 重新拉取工作区根目录内容
2. 记录当前展开状态 `expandedWorkspacePaths`
3. 对已展开节点递归重新加载
4. 保留当前选中项
5. 如果选中项已不存在，则回退到最近可用父节点或工作区根节点

### 涉及文件

- `src/hooks/useAppLayoutState.ts`

### 验收标准

- 刷新后目录树与真实文件系统状态一致
- 已展开目录尽量保持展开
- 当前选中项尽量保持不变
- 选中项已失效时有合理回退策略

# Phase 2：状态反馈完善

## 4. 增加创建中/刷新中状态

### 当前问题

用户在创建或刷新期间缺少明确反馈，可能重复点击。

### 改造方案

增加状态字段：

- `workspaceCreating: boolean`
- `workspaceRefreshing: boolean`

在 UI 上表现为：

- 创建期间禁用新建文件、新建文件夹按钮
- 刷新期间禁用刷新按钮
- 刷新图标持续旋转
- 输入框提交中显示 loading 样式或不可编辑状态

### 涉及文件

- `src/hooks/useAppLayoutState.ts`
- `src/components/LeftSidebar.tsx`

### 验收标准

- 创建过程中不会重复提交
- 刷新过程中不会重复触发刷新
- 用户能明确感知操作进行中

---

## 5. 优化错误提示语义

### 当前问题

目前错误多依赖 toast 原样显示，语义不够聚焦。

### 改造方案

对常见错误做统一映射，例如：

- 同名文件已存在
- 同名文件夹已存在
- 路径无效
- 无权限创建
- 目录已不存在
- 输入被取消

错误提示应尽量明确动作对象和原因，例如：

- “创建文件失败：当前目录下已存在同名文件”
- “创建文件夹失败：目录不可写”
- “刷新失败：目录已被外部删除”

### 涉及文件

- `src/hooks/useAppLayoutState.ts`
- `src/services/workspace/tauri.ts`

### 验收标准

- 常见错误具有清晰可理解的中文提示
- 用户能快速判断是重试、改名还是切换目录
