# 3.Agent 产品方案

## 1. 文档目的

本文档用于定义 `TweetPilot` 的 Agent 产品方向。

重点回答以下问题：

1. `TweetPilot` 为什么适合做 Agent
2. `TweetPilot` 的 Agent 最终形态应该是什么
3. 平台与 Agent 的关系是什么
4. Agent 应该拆成哪些子能力和子模块
5. 哪些内容必须第一期就做
6. 哪些内容可以分期推进

本文档不是技术实现细节文档，而是产品方案文档。

## 2. 一句话结论

`TweetPilot` 不应该只是一个“可调用 Twitter 扩展的桌面管理平台”，而应该逐步演化为：

**一个面向 Twitter 代运营团队的垂直领域 Agent Operating Platform。**

也就是说：

- 平台负责管理、配置、权限、报表、审计、调度
- Agent 负责观察、判断、生成、建议、执行、复盘

最终卖点不是“支持接大模型”，而是：

**帮助客户拥有一组可被管理、可被监督、可被复盘的 Twitter 数字员工与运营 Agent。**

## 3. 为什么 `TweetPilot` 适合做 Agent

一个真正可商用的 Agent，不只是“会聊天的模型”，而需要：

- 明确目标
- 可调用的工具
- 可持续的状态
- 可观测的执行过程
- 可审计的结果

`TweetPilot` 已经天然具备了这些条件：

### 3.1 已有真实工具层

当前产品已经有明确的工具能力基础：

- 浏览器扩展实例
- LocalBridge API
- Twitter 读写动作
- 本地模型和云模型接入能力

这意味着 Agent 不是空转推理，而是能真正执行动作。

### 3.2 已有真实执行环境

我们不是停留在“生成文案”，而是可以：

- 读取评论
- 回复评论
- 发推文
- 点赞、转发、关注
- 切换和管理 Twitter tab
- 路由到不同扩展实例

这让 Agent 天然具备“手和脚”。

### 3.3 已有真实业务闭环

代运营场景本身就适合 Agent：

- 评论需要持续监控
- 内容需要持续生产
- 任务需要持续执行
- 风险需要持续预警
- 报表需要持续生成

这些都不是一次性动作，而是长期运行型工作流。

### 3.4 已有明确客户价值

Agent 的价值必须能转化成客户愿意付费的结果。

在 `TweetPilot` 里，这个结果非常明确：

- 更快回复
- 更少人力
- 更高互动效率
- 更稳定的交付
- 更强的客户可见性
- 更好的风险识别与产品反馈洞察

## 4. Agent 的最终产品形态

从长期看，`TweetPilot` 的最终形态不应是一个单一总 Agent，而应是：

**一个由多个垂直 Agent + 数字员工角色系统 + 任务编排系统 + 风险与报表系统组成的平台。**

### 4.1 平台层与 Agent 层的关系

#### 平台层负责

- 账号管理
- 扩展与实例管理
- 标签页状态管理
- 任务创建与调度
- AI 提供者配置
- 报表
- 风控
- 审计
- 团队权限

#### Agent 层负责

- 持续观察数据与状态
- 识别机会或异常
- 生成候选内容
- 提出执行建议
- 在授权范围内执行动作
- 产出总结和复盘

### 4.2 最终不应该只有一个 Agent

我们不建议做“一个万能 Agent”。

正确方向是做多个垂直 Agent，每个 Agent 解决一类清晰、高频、可量化的问题。

建议的长期 Agent 体系如下。

#### 1. Reply Agent

职责：

- 监控评论和提及
- 识别待回复内容
- 生成候选回复
- 自动或半自动发送
- 记录结果与质量

#### 2. Content Agent

职责：

- 根据素材和账号人设生成内容
- 生成发帖草稿
- 给出多个版本建议
- 参与排期和发布流程

#### 3. Growth Agent

职责：

- 发现值得互动的人群和账号
- 跟踪关注、被关注、互动对象
- 生成增长任务建议
- 驱动互动和关注计划

#### 4. Risk Agent

职责：

- 识别负面评论与高风险互动
- 聚类投诉主题
- 输出风险预警
- 生成 AI 产品反馈与改进建议报告

#### 5. Account Health Agent

职责：

- 检查账号健康状态
- 检查实例在线状态
- 检查标签页状态
- 检查任务失败和异常
- 输出巡检与修复建议

#### 6. Delivery Agent

职责：

- 自动整理客户周报 / 月报
- 汇总任务执行结果
- 输出客户可读成果说明

### 4.3 数字员工与 Agent 的关系

数字员工不是 Agent 的替代品，而是 Agent 的人格层和执行风格层。

可以这样理解：

- Agent 是“工作职责”
- 数字员工是“表达方式和行为风格”

例如：

- `Reply Agent` 负责回复任务
- 但它可以调用不同数字员工角色生成不同风格的回复：
  - 品牌客服型
  - 创始人口吻型
  - 社群互动型
  - 危机公关型

所以最终系统不是：

- 一个大模型

而是：

- 多个 Agent
- 多个角色
- 多种 AI 提供者
- 多种执行策略

## 5. Agent 的核心组成

为了真正把 Agent 做成产品，而不是宣传概念，Agent 至少要由以下能力组成。

### 5.1 观察层

负责持续感知：

- 账号状态
- 实例状态
- tab 状态
- 评论 / 提及 / 时间线
- 任务状态
- AI 调用结果
- 风险信号

### 5.2 记忆层

负责沉淀长期上下文：

- 账号历史
- 品牌语气
- 角色设定
- 过去互动
- 高价值用户
- 已处理记录
- 风险主题

### 5.3 决策层

负责决定：

- 是否需要回复
- 是否需要人工审核
- 使用哪个 AI 提供者
- 使用哪个数字员工角色
- 路由到哪个实例和账号

### 5.4 执行层

负责调用实际工具：

- LocalBridge API
- 浏览器扩展实例
- 模型 API
- 外部 REST API

### 5.5 审核层

负责高风险动作的人机协同：

- 哪些动作必须人工审核
- 哪些动作可自动执行
- 哪些账号启用更严格策略

### 5.6 观测与复盘层

负责输出：

- 执行日志
- 采用率
- 成功率
- 风险命中情况
- 报表和复盘总结

## 6. 第一性原则：第一期必须做什么

虽然最终形态可以很大，但第一期必须极度克制。

第一期必须满足三个标准：

1. 最有价值
2. 最不可缺失
3. 最容易形成闭环并卖出去

按照这个标准，**第一期不应该做“大而全 Agent 平台”**。

第一期应该只做：

## 6.1 第一优先级：Reply Agent

这是整个 Agent 方向中最重要、最不可缺失、最有商业价值的部分。

原因：

- 评论回复是高频刚需
- 对团队人效提升最直接
- 最容易形成“AI + 扩展 + 审核 + 报表”的闭环
- 最容易被客户感知到价值
- 最容易切入收费和交付

### 第一阶段 Reply Agent 的最小闭环

1. 监控某账号的评论或提及
2. 发现新的待处理回复
3. 调用 AI 生成候选回复
4. 按数字员工角色输出不同版本
5. 人工审核或低风险自动发送
6. 记录结果
7. 汇总到报表

这条链路跑通，产品就已经从工具升级到了“最小可售卖 Agent”。

## 6.2 第一优先级：数字员工角色基础能力

如果只有 Reply Agent，而没有角色系统，那么客户感知到的仍然只是“自动回复工具”。

所以第一期必须包含最小角色系统。

最小角色系统只需要支持：

- 账号绑定一个默认角色
- 角色包含：
  - 名称
  - 语气
  - 人设
  - 禁止事项
- 生成回复时可切换角色

第一期不需要做非常复杂的角色管理市场，但必须把“数字员工”这个价值点立起来。

## 6.3 第一优先级：风险识别基础能力

如果第一期完全不做风险识别，产品就还是停留在“执行工具”。

第一期至少应该做最小版本的风险能力：

- 标记高风险评论
- 标记负面反馈
- 把高风险评论提升到人工审核
- 对评论主题做简单聚类
- 输出基础风险摘要

这能直接把产品价值抬高到“运营 + 风险感知”。

## 6.4 第一优先级：Agent 可观测性

第一期即使功能少，也必须有可观测性。

至少要让用户知道：

- Agent 看到了什么
- Agent 为何这样判断
- Agent 调用了哪个 AI
- Agent 是否调用了扩展
- Agent 最终执行结果是什么

没有这个能力，Agent 就不可控，也很难卖给团队客户。

## 7. 第一阶段明确不做什么

为了保证第一期能打穿，不建议一开始就做：

- 通用大而全总 Agent
- 多 Agent 协同编排
- 自动完成所有运营动作
- 高复杂度长期记忆系统
- 全量风控中心
- 完整交付门户
- 复杂多租户权限

这些都很有想象力，但不适合第一期。

## 8. 分期路线图

## 8.1 第一阶段：Reply Agent MVP

目标：

- 做出第一个可卖的 Agent 闭环

包含：

- Reply Agent
- 最小数字员工角色
- 人工审核流
- 风险识别基础能力
- AI 调用进展展示
- 回复结果记录
- 基础报表

第一阶段的销售话术应是：

**“让一个运营团队拥有一位可以 24/7 监控评论、生成回复并辅助互动的 Twitter 数字员工。”**

## 8.2 第二阶段：多 Agent 扩展

目标：

- 从单点回复，扩展到内容与增长

包含：

- Content Agent
- Growth Agent
- 风险预警增强
- 产品反馈洞察增强
- 定时任务结合 Agent
- 多角色体系增强

第二阶段的价值会从“评论回复”扩展到“内容生产 + 增长互动”。

## 8.3 第三阶段：Agent Operating Platform

目标：

- 从“若干 Agent 功能”升级为“完整 Agent 平台”

包含：

- 多 Agent 协同
- Agent 策略中心
- Agent 权限中心
- Agent 审计中心
- Agent 评估体系
- 企业级风控与报表
- 客户交付与汇报自动化

## 9. 商业价值与价格提升逻辑

Agent 方向最大的意义，不只是技术先进，而是能把价格带抬高。

### 9.1 如果只是平台

客户理解为：

- 账号管理工具
- 自动化执行工具
- AI 接口接入工具

这类产品通常会被拿去和各种普通 SaaS 工具比较价格。

### 9.2 如果是垂直 Agent 平台

客户理解为：

- 我拥有了一组可工作的数字员工
- 我不仅能执行动作，还能持续感知和优化
- 平台会帮我减少人力、缩短响应时间、识别风险、产出报告

这时定价逻辑就可以从“工具订阅”升级成：

- 软件订阅
- 数字员工席位
- Agent 模块包
- 运营增强服务
- 企业交付服务

### 9.3 长期最强卖点

长期最强的外部价值表达不是：

- 我们支持很多模型
- 我们支持很多 API

而是：

**“TweetPilot 让 Twitter 代运营公司拥有一组可管理、可审计、可交付的 AI 数字员工和垂直运营 Agent。”**

## 10. 当前结论

结论很明确：

1. `TweetPilot` 非常适合做 Agent 化升级
2. 但不应抛弃平台，而应做成“平台 + Agent 系统”
3. 第一阶段最重要、最不可缺失、最有价值的部分是：
   - `Reply Agent`
   - `数字员工角色基础能力`
   - `基础风险识别能力`
   - `Agent 可观测性`
4. 只有先打穿这个最小闭环，后续的 Content Agent、Growth Agent、Risk Agent、Delivery Agent 才有意义

因此，产品执行策略应该是：

**先做最小可售卖 Agent，再逐步做成完整的垂直 Agent Operating Platform。**
