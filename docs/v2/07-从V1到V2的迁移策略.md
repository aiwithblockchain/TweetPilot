## 1. 迁移策略总览

### 1.1 核心决策

**建议：V1 代码可以完全抛弃，不是必须保留**

**理由**：
1. **架构完全不同**：V2 采用产品中心化 + Claurst Host Manager + MCP 服务器，与 V1 的单产品 + 直接调用 Claude API 完全不同
2. **复用率低**：V1 代码复用率约 20-30%，强行复用反而增加复杂度和技术债务
3. **从零开始更清晰**：V2 是全新的架构设计，从零开始可以避免 V1 的设计包袱

**但是**：
- ✅ **保留** LocalBridge HTTP Client（可以直接复用）
- ✅ **参考** V1 的业务逻辑和设计思路（任务路由、执行保护等）
- ✅ **参考** V1 的 UI 组件（审核队列、任务管理等）

### 1.2 迁移原则

1. **不强求复用**：不为了复用而复用，只复用真正有价值的部分
2. **参考优于复用**：参考 V1 的设计思路和业务逻辑，而不是直接复制代码
3. **渐进式迁移**：先实现核心功能，再逐步完善
4. **保持简洁**：避免引入 V1 的技术债务

**参考方案**：

3. 在 V2 中重新实现（适配新架构）

**具体步骤**：

```typescript

// 步骤 2: 提取业务规则
// 例如：任务路由规则
// - 低风险任务 → 自动执行
// - 中风险任务 → 人工审核
// - 高风险任务 → 人工接管

// 步骤 3: 在 V2 中重新实现
// src/services/TaskRoutingService.ts (V2)
export class TaskRoutingService {
  routeTask(task: Task): TaskRoute {
    // 参考 V1 的路由规则，但使用 V2 的数据模型

      return 'auto_execute';
    } else if (task.riskLevel === 'medium') {
      return 'review_queue';
    } else {
      return 'manual_takeover';
    }
  }
}
```

## 4. 迁移风险和缓解措施

### 4.1 架构变化风险

**风险**：V2 架构与 V1 完全不同，迁移成本高

**影响**：高

**概率**：确定

**缓解措施**：
- 接受架构变化，不强求复用 V1 代码
- 参考 V1 的业务逻辑和设计思路
- 逐步迁移，先实现核心功能
- 充分测试，确保功能正确

### 4.2 Claurst 集成风险

**风险**：Claurst 版本不稳定（0.0.9），可能有 bug

**影响**：高

**概率**：中

**缓解措施**：
- 锁定 Claurst 版本
- 准备 fallback 方案（直接调用 Claude API）
- 充分测试 Claurst 集成
- 监控 Claurst 进程健康状态

### 4.3 开发时间风险

**风险**：V2 开发时间可能超出预期

**影响**：中

**概率**：中

**缓解措施**：
- 采用接口优先 + UI 优先开发策略
- 并行开发，提高效率
- 使用 Mock 数据快速验证
- 定期评审进度，及时调整

### 4.4 团队学习曲线风险

**风险**：团队需要学习新架构和新技术（Claurst、MCP）

**影响**：中

**概率**：中

**缓解措施**：
- 提供详细的技术文档
- 组织技术分享会
- 结对编程，互相学习
- 逐步迁移，降低学习压力

## 6. 迁移决策建议

### 6.1 推荐方案：完全重写

**建议**：**从零开始开发 V2，不强求复用 V1 代码**

**理由**：
1. **架构完全不同**：V2 的产品中心化 + Claurst Host Manager 架构与 V1 完全不同
2. **复用率低**：V1 代码复用率约 20-30%，强行复用反而增加复杂度
3. **技术债务**：V1 可能存在技术债务，从零开始可以避免
4. **开发效率**：从零开始可以更快地实现 V2 的核心功能

**但是**：
- ✅ **保留** LocalBridge HTTP Client（可以直接复用）
- ✅ **参考** V1 的业务逻辑和设计思路
- ✅ **参考** V1 的 UI 组件

### 6.2 替代方案：渐进式迁移

**建议**：**逐步迁移 V1 代码到 V2**

**理由**：
1. **降低风险**：渐进式迁移可以降低迁移风险
2. **保持连续性**：用户可以继续使用 V1，同时开发 V2
3. **复用现有工作**：可以复用 V1 的部分代码和测试

**缺点**：
- ❌ **复杂度高**：需要同时维护 V1 和 V2
- ❌ **时间长**：渐进式迁移时间更长
- ❌ **技术债务**：可能引入 V1 的技术债务

**不推荐**：由于 V2 架构与 V1 完全不同，渐进式迁移的复杂度和成本较高，不推荐采用。
