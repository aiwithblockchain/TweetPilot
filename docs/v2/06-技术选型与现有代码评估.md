## 1. V1 代码现状评估

### 1.1 V1 Phase 1 完成情况

TweetPilot V1 已经完成了 Phase 1 的开发（T1-T5 任务卡），包括：

### 1.3 V1 vs V2 架构对比

| 维度 | V1 架构 | V2 架构 | 差异程度 |
|------|---------|---------|---------|
| **核心理念** | 单一产品运营 | 产品中心化（多产品工作区） | 🔴 完全不同 |
| **AI 集成** | 直接调用 Claude API | Claurst Host Manager（Extension Host 模式） | 🔴 完全不同 |
| **Agent 实现** | TypeScript 实现 Reply Agent | Claurst Coordinator + Worker Agents | 🔴 完全不同 |
| **知识库** | 空实现（预留接口） | MCP 知识库服务器 | 🟡 接口可参考 |
| **LocalBridge** | HTTP Client + 执行器 | HTTP Client + MCP 封装 | 🟢 可以复用 |
| **领域对象** | 任务、执行请求、通道等 | 产品、角色实例、任务会话等 | 🟡 部分概念可保留 |
| **服务层** | 任务路由、执行编排等 | 产品管理、Claurst Host Manager 等 | 🟡 业务逻辑可参考 |
| **UI 组件** | 审核队列、执行工作台 | 产品选择器、产品工作台等 | 🟢 部分可复用 |

**图例**：
- 🔴 完全不同 - 无法复用，需要重新设计和实现
- 🟡 部分相似 - 接口或概念可参考，但实现需要重构
- 🟢 可以复用 - 可以直接复用或稍作修改

## 2. V1 代码复用性详细评估

### 2.1 AI Provider 层（src/ai/）

**V1 实现**：
- 直接调用 Claude API（`@anthropic-ai/sdk`）
- 提供 `IAIProvider` 接口抽象
- 支持同步和流式生成

**V2 需求**：
- 通过 Claurst Host Manager 管理 Claurst 进程
- 通过 IPC（stdin/stdout + JSON）与 Claurst 通信
- Claurst 内部管理所有 Worker Agent

**复用评估**：❌ **完全不同，无法复用**

**原因**：
- V1 是直接调用 Claude API，V2 是管理 Claurst 进程
- V1 是同步/流式 API 调用，V2 是 IPC 通信
- V1 的 `IAIProvider` 接口不适用于 V2 的 Claurst Host Manager

**建议**：
- V1 的 `src/ai/` 目录可以完全抛弃
- V2 需要开发全新的 `ClaurstHostManager`（参考 [02-Claurst技术评估与集成方案.md](docs/v2/02-Claurst技术评估与集成方案.md)）

### 2.2 知识库层（src/knowledge/）

**V1 实现**：
- `IKnowledgeBase` 接口定义
- `EmptyKnowledgeBase` 空实现（返回空数组）

**V2 需求**：
- MCP 知识库服务器（Node.js 进程）
- 通过 MCP 协议提供知识库能力
- 支持产品专属知识库

**复用评估**：⚠️ **接口可参考，实现需要重构**

**原因**：
- V1 的 `IKnowledgeBase` 接口设计思路可以借鉴
- 但 V2 需要实现 MCP 服务器，而不是简单的 TypeScript 类

**建议**：
- 保留 V1 的 `IKnowledgeBase` 接口定义作为参考
- V2 需要开发 MCP 知识库服务器（`mcp-servers/knowledge/index.ts`）
- MCP 服务器内部可以参考 V1 的接口设计

### 2.3 Agent 层（src/agents/）

**V1 实现**：
- `ReplyAgent.ts`：TypeScript 实现的 Reply Agent
- 依赖注入 `IAIProvider` 和 `IKnowledgeBase`
- 支持角色注入、知识注入、风险评估

**V2 需求**：
- Claurst Coordinator 管理所有 Worker Agent
- Worker Agent 由 Claurst 内部创建和管理
- 通过 MCP 工具调用获取数据

**复用评估**：❌ **完全不同，无法复用**

**原因**：
- V1 是 TypeScript 实现的 Agent，V2 由 Claurst 管理
- V1 的 Agent 逻辑（prompt 构建、风险评估）在 V2 中由 Claurst 处理

**建议**：
- V1 的 `src/agents/` 目录可以完全抛弃
- V2 的 Agent 能力由 Claurst 提供，不需要自己实现

### 2.4 LocalBridge 适配器层（src/adapters/localBridge/）

**V1 实现**：
- `client.ts`：LocalBridge HTTP Client
- `localBridgeReplyExecutor.ts`：执行器实现
- `types.ts`：类型定义

**V2 需求**：
- LocalBridge HTTP Client（封装 REST API 调用）
- Twitter MCP Server（封装 LocalBridge Client）
- Claurst 通过 MCP 协议调用

**复用评估**：✅ **可以复用 HTTP Client 部分**

**原因**：
- V1 的 `client.ts` 已经实现了 LocalBridge REST API 的封装
- V2 只需要在此基础上添加 MCP 服务器层

**建议**：
- **保留** V1 的 `src/adapters/localBridge/client.ts`
- **重构** 为 `src/modules/LocalBridgeClient.ts`（参考 [02-Claurst技术评估与集成方案.md](docs/v2/02-Claurst技术评估与集成方案.md)）
- **新增** `mcp-servers/twitter/index.ts`（封装 LocalBridgeClient）
- **抛弃** `localBridgeReplyExecutor.ts`（V2 不需要执行器）

### 2.5 领域对象层（src/domain/）

**V1 实现**：
- `replyTask.ts`：回复任务
- `candidateReply.ts`：候选回复
- `executionRequest.ts`：执行请求
- `executionChannel.ts`：执行通道
- `commentInput.ts`：评论输入
- 其他领域对象

**V2 需求**：
- `Product`：产品（新增）
- `AgentInstance`：角色实例（新增）
- `TaskSession`：任务会话（新增）
- `Task`：任务（概念类似）
- `ExecutionRequest`：执行请求（概念类似）

**复用评估**：⚠️ **部分概念可保留，但需要重构**

**原因**：
- V1 的任务、执行请求等概念在 V2 中仍然存在
- 但 V2 增加了产品中心化的概念（Product、AgentInstance、TaskSession）
- 数据模型需要重新设计

**建议**：
- **参考** V1 的领域对象设计思路
- **重新设计** V2 的领域对象（参考 [03-模块化架构设计.md](docs/v2/03-模块化架构设计.md)）
- **保留** 部分通用概念（如任务状态、执行请求）

### 2.6 服务层（src/services/）

**V1 实现**：
- `replyTaskCreationService.ts`：任务创建
- `taskRoutingService.ts`：任务路由
- `executionService.ts`：执行服务
- `executionEligibilityService.ts`：执行资格判定
- `executionProtectionService.ts`：执行保护
- 其他服务

**V2 需求**：

**复用评估**：⚠️ **业务逻辑可参考，但需要适配新架构**

**原因**：
- V1 的服务层包含了很多业务逻辑（任务路由、执行保护等）
- 这些业务逻辑在 V2 中仍然需要，但需要适配新架构

**建议**：
- **参考** V1 的服务层设计思路和业务逻辑
- **重构** 为 V2 的服务层（适配产品中心化和 Claurst Host Manager）
- **保留** 通用的业务逻辑（如任务路由规则、执行保护边界）

### 2.7 数据层（src/data/）

**V1 实现**：

- 其他内存仓储

**V2 需求**：

**原因**：

**建议**：

## 4. 技术选型

### 4.1 前端框架

**选择**：React + TypeScript

**理由**：
- V1 已经使用 React + TypeScript，团队熟悉
- React 生态丰富，组件库完善
- TypeScript 提供类型安全

**保持不变**：✅

### 4.2 后端框架

**选择**：Node.js + TypeScript

**理由**：
- V1 已经使用 Node.js + TypeScript
- 与前端技术栈统一，降低学习成本
- MCP 服务器需要 Node.js 实现

**保持不变**：✅

### 4.3 数据库

**选择**：SQLite（第一阶段）

**理由**：
- 本地优先，数据存储在本地
- SQLite 轻量级，无需额外服务
- 第二阶段可以升级到 PostgreSQL

**V1 使用**：内存仓储（InMemory）

**V2 变化**：升级到 SQLite

### 4.4 UI 组件库

**选择**：Shadcn/ui + Tailwind CSS

**理由**：
- 现代化 UI 组件库
- 与 Tailwind CSS 集成良好
- 可定制性强

**V1 使用**：自定义 CSS

**V2 变化**：引入 Shadcn/ui

### 4.5 Claurst 集成

**选择**：Claurst Host Manager（Extension Host 模式）

**理由**：
- 借鉴 VS Code Extension Host 架构
- 单一 Claurst Host 进程管理所有 Worker
- 通过 IPC（stdin/stdout + JSON）通信

**V1 使用**：直接调用 Claude API

**V2 变化**：完全不同的架构

## 5. 技术风险评估

### 5.1 架构变化风险

**风险**：V2 架构与 V1 完全不同，迁移成本高

**影响**：高

**概率**：确定

**缓解措施**：
- 接受架构变化，不强求复用 V1 代码
- 参考 V1 的业务逻辑和设计思路
- 逐步迁移，先实现核心功能

### 5.2 Claurst 集成风险

**风险**：Claurst 版本不稳定（0.0.9），可能有 bug

**影响**：高

**概率**：中

**缓解措施**：
- 锁定 Claurst 版本
- 准备 fallback 方案（直接调用 Claude API）
- 充分测试 Claurst 集成

### 5.3 开发时间风险

**风险**：V2 开发时间可能超出预期

**影响**：中

**概率**：中

**缓解措施**：
- 采用接口优先 + UI 优先开发策略
- 并行开发，提高效率
- 使用 Mock 数据快速验证

## 6. 推荐方案

### 6.1 总体策略

**建议**：**V1 代码可以完全抛弃，不是必须保留**

**理由**：
1. V2 架构与 V1 完全不同（产品中心化 + Claurst Host Manager）
2. V1 代码复用率低（约 20-30%），强行复用反而增加复杂度
3. V2 是全新的架构设计，从零开始更清晰

**但是**：
- **保留** LocalBridge HTTP Client（可以直接复用）
- **参考** V1 的业务逻辑和设计思路（任务路由、执行保护等）
- **参考** V1 的 UI 组件（审核队列、任务管理等）

### 6.2 实施建议

**阶段 1：核心架构搭建（2-3 周）**
- 开发 Claurst Host Manager
- 开发 MCP 服务器（Twitter、知识库、数据访问）
- 开发产品管理模块

**阶段 2：UI 设计和实现（2-3 周）**
- 使用 ui-ux-pro-max-skill 设计 UI
- 实现产品选择器、产品工作台
- 复用 V1 的审核队列、任务管理 UI（稍作修改）

**阶段 3：业务逻辑实现（4-6 周）**
- 参考 V1 的业务逻辑
- 实现任务管理、执行服务
- 实现审核队列、角色实例管理

**阶段 4：集成测试（2-3 周）**
- 端到端测试
- 性能测试
- 用户验收测试

**总时间**：约 10-15 周
