# P0-01-需求说明

## 文档概述

本文档将 P0 阶段的功能需求划分为 A、B 两个开发阶段，确保优先交付核心刚需功能，后续再补充增强功能。

## 阶段划分原则

- **阶段 A**：刚需的、不可或缺的核心功能，每个功能都有明确的使用场景和操作流程
- **阶段 B**：增强型功能，提升用户体验但非首次交付必需

---

## 阶段 A：核心刚需功能

### 1. 工作目录管理（Workspace Management）

**核心价值**：确定当前工作的根目录，所有数据、配置、AI 对话都基于此目录

**必需功能**：
- 选择/创建本地工作目录
- 从 GitHub 克隆项目作为工作目录
- 显示最近使用的工作目录历史
- 工作目录路径显示与切换

**数据存储**：
- 工作目录路径
- 最近使用历史（最多 10 条）

---

### 2. Twitter 账号状态监控（Account Status）

**核心价值**：实时了解管理的 Twitter 账号连接状态，确保任务可正常执行

**架构说明**：
- TweetPilot 是 Tauri 桌面应用，不直接管理浏览器 Cookie/Session
- 用户在真实浏览器（Chrome/Edge）中登录 Twitter
- TweetPilot 通过 LocalBridge 与浏览器扩展通信
- 账号映射基于 `screen_name`（@username）

**必需功能**：
- 映射已登录的 Twitter 账号（通过 screen_name）
- 删除账号映射
- 实时显示账号连接状态（在线/离线）
- 账号状态异常时的提示
- 手动刷新账号状态

**数据存储**：
- 账号映射配置（screen_name → extension_id + tab_id）**注意：映射配置是临时的**
- 账号基本信息（用户名、头像、状态）

**技术实现**：
- 通过 LocalBridge 获取已连接的浏览器扩展 ID
- 查询扩展中当前登录的 Twitter 账号信息
- 用户选择 screen_name 建立映射关系
- 执行任务时通过 LocalBridge + tab_id 与浏览器交互

**映射配置的临时性和检查机制**：
- extension_id 和 tab_id 是临时的，浏览器重启或扩展重载后会变化
- **启动时检查**：TweetPilot 启动时必须检查并更新所有账号的映射配置
- **定时检查**：定期（如每 5 分钟）验证映射配置的有效性
- **失败重试**：LocalBridge 请求失败时，立即重新检查并更新连接信息
- **状态同步**：TweetPilot 和浏览器扩展是两套独立系统，必须及时掌握对方的变化

---

### 3. 定时任务管理（Task Management）

**核心价值**：管理 Python 脚本任务的执行，查看任务健康状况和统计数据

**必需功能**：

#### 3.1 任务列表
- 显示所有任务（名称、类型、状态、下次执行时间）
- 任务状态标识（运行中/已暂停/已完成/失败）
- 任务类型标识（定时任务/即时任务）

#### 3.2 任务操作
- 暂停任务
- 恢复任务
- 立即执行任务
- 删除任务

#### 3.3 任务配置
- 添加新任务（上传 Python 脚本文件）
- 设置任务名称和描述
- 配置任务类型（定时/即时）
- 配置定时规则（cron 表达式或简单间隔）
- 配置任务参数（传递给 Python 脚本的参数）

#### 3.4 任务详情
- 查看任务配置信息
- 查看执行历史（最近 20 条）
- 查看执行结果（成功/失败、输出日志）
- 查看统计数据（总执行次数、成功率、平均耗时）

**数据存储**：
- 任务配置（名称、描述、类型、脚本路径、参数、定时规则）
- 执行历史（时间、状态、输出、耗时）
- Python 脚本文件（存储在工作目录下的 `scripts/` 目录）

**技术约束**：
- 阶段 A 不提供 Python 脚本编辑器
- 用户需在外部编辑脚本，通过界面上传
- 系统只负责脚本的执行和结果处理

---

### 4. 设置界面（Settings）

**核心价值**：集中管理系统配置

**必需功能**：
- Twitter 账号管理（添加/删除账号）
- 工作目录设置
- 基础系统设置（语言、主题等）

---

### 5. 数据积木（Data Blocks）

**核心价值**：以可视化方式展示 Twitter 数据的简单加工结果

**必需功能**：

#### 5.1 Grid 布局
- 使用网格布局展示多个数据卡片
- 每个卡片展示一种数据类型
- 支持卡片的添加/删除/排序

#### 5.2 数据卡片类型（阶段 A 提供 6 种）

1. **账号推文统计**
   - 显示每个账号的推文总数
   - 按账号分组展示

2. **最新推文列表**
   - 显示每个账号最新的 10 条推文
   - 显示基本信息：内容、点赞数、回复数、转发数、浏览数

3. **账号基础数据**
   - 显示每个账号的：
     - 关注数
     - 被关注数
     - 蓝V 关注数
     - 总推文数

4. **账号互动数据**
   - 显示每个账号的：
     - 总浏览量
     - 总点赞数
     - 总转推数

5. **推文时间分布**
   - 显示最近 7 天的推文发布数量
   - 简单柱状图展示

6. **任务执行统计**
   - 显示最近 24 小时的任务执行情况
   - 成功/失败数量统计

**数据存储**：
- 数据卡片配置（类型、位置、参数）
- 数据缓存（避免频繁请求）

**技术约束**：
- 阶段 A 限定最多 9 种数据卡片类型
- 数据卡片类型需预定义，不支持用户自定义
- 数据更新采用手动刷新或定时刷新

## 技术架构要求

### 阶段 A
- 使用 Tauri 2.x (Rust 后端 + React 前端)
- 前端：React 19 + TypeScript + Vite + Tailwind CSS + shadcn/ui
- 后端：Rust (Tauri Commands)，数据存储使用本地 JSON 文件
- Python 脚本执行使用 tauri-plugin-shell + tokio::process::Command
- 详见 docs/v2/p0/ENGINEERING-01-工程搭建.md

### 阶段 B
- 考虑引入 SQLite 数据库
- 考虑引入 WebSocket 实时通信
- 考虑引入更复杂的状态管理
