# P0-01-需求说明

## 文档概述

本文档定义 TweetPilot P0 阶段的需求边界，并划分为 A、B 两个阶段：

- **阶段 A**：首版必须交付的核心能力
- **阶段 B**：增强能力，后续按优先级推进

本版已根据当前进度更新：
- Tauri 技术路线已确定
- UI 基础已完成
- 下一步以“模块接口化 + Mock-first + 真实实现替换”为主线

---

## 阶段划分原则

1. 阶段 A 只做刚需功能，不为未来过度设计
2. 每个功能都要有明确用户场景和可验证结果
3. 模块边界优先于实现细节，接口契约优先于临时代码

---

## 阶段 A：核心刚需功能

## 1. 工作目录管理（Workspace Management）

**核心价值**：确定当前实例的工作根目录，所有任务、账号映射、数据积木都围绕该目录组织。

**必需功能**：
- 选择本地工作目录
- 从 GitHub 克隆项目作为工作目录
- 最近使用工作目录历史（最多 10 条）
- 在新窗口打开工作目录（当前实例不受影响）
- 当前工作目录路径展示

**数据存储（真实实现阶段）**：
- `~/.tweetpilot/config.json`
- `~/.tweetpilot/recent-workspaces.json`

---

## 2. Twitter 账号管理与状态监控（Account Management）

**核心价值**：确保任务能绑定可用账号并持续可执行。

**架构约束**：
- TweetPilot 不直接管理浏览器 Cookie/Session
- 通过 LocalBridge 与浏览器扩展通信
- 映射主键为 `screen_name`

**必需功能**：
- 查询并映射可用账号
- 删除账号映射（解除绑定）
- 查看账号状态（在线/离线/验证中）
- 刷新账号状态
- 账号设置：查看账号详情、编辑 personality
- 彻底删除账号与本地关联数据（二次确认）

**数据存储（真实实现阶段）**：
- 账号映射配置（`screen_name -> extension_id + tab_id`）
- 账号 profile
- 账号设置（如 personality）

**关键说明**：
- `extension_id` / `tab_id` 是临时映射，可能随浏览器重启变化
- 启动检查 + 定期检查是必需能力

---

## 3. 定时任务管理（Task Management）

**核心价值**：管理 Python 脚本任务的创建、调度、执行、统计。

**必需功能**：

### 3.1 任务列表
- 展示任务名称、类型、状态、下次执行时间
- 支持按状态查看（运行中/暂停/失败等）

### 3.2 任务操作
- 创建任务
- 更新任务配置
- 暂停/恢复任务
- 立即执行任务
- 删除任务

### 3.3 任务详情
- 查看任务配置
- 查看执行历史（最近 20 条）
- 查看执行结果日志
- 查看统计数据（总次数、成功率、平均耗时）

**技术约束（阶段 A）**：
- 不提供内置脚本编辑器
- 用户在外部编辑脚本后上传/指定路径
- 系统负责执行与结果管理

**数据存储（真实实现阶段）**：
- 任务配置
- 执行历史（jsonl）
- 执行日志

---

## 4. 数据积木（Data Blocks）

**核心价值**：将账号与任务数据以可视化卡片形式展示，支持快速洞察。

**必需功能**：
- 网格布局展示卡片
- 卡片添加/删除/排序
- 卡片数据加载与刷新
- 布局和缓存管理

### 阶段 A 预定义卡片（6 种）
1. 账号推文统计
2. 最新推文列表
3. 账号基础数据
4. 账号互动数据
5. 推文时间分布
6. 任务执行统计

**技术约束（阶段 A）**：
- 卡片类型预定义，不支持用户自定义类型
- 数据可手动刷新，缓存策略可简化

---

## 5. 设置管理（Settings）

**核心价值**：集中管理系统配置，避免配置分散。

**必需功能**：
- Twitter 账号管理入口
- 工作目录相关配置入口
- LocalBridge 配置（默认 `127.0.0.1:10088`）
- 基础偏好设置（主题/语言，按阶段推进）

---

## 阶段 B：增强功能（非首版必须）

### 1. 工作目录增强
- 模板、标签、批量导入

### 2. 账号管理增强
- 账号分组、标签、批量操作、导出

### 3. 任务管理增强
- 任务依赖、执行链、任务模板
- 内置脚本编辑器
- 脚本版本管理
- 高级日志过滤与性能分析

### 4. 数据积木增强
- 自定义卡片
- 更多预定义卡片类型（20+）
- 数据导出与对比
- 实时更新（WebSocket）

### 5. AI 辅助增强
- 基于工作目录的对话与建议
- 任务脚本生成辅助
- 数据解读辅助

### 6. 高级设置
- 代理、日志级别、备份恢复等

---

## 开发优先级

1. **P0-A（当前）**：核心可用，流程闭环
2. **P0-B（后续）**：体验和能力增强

---

## 技术架构要求

### 阶段 A
- Tauri 2.x（Rust 后端 + React 前端）
- 前端：React 19 + TypeScript + Vite + Tailwind + shadcn/ui
- 后端：Rust Commands + 本地存储
- Python 执行：`tauri-plugin-shell` + `tokio::process::Command`
- 分层：`UI -> Service Interface -> Mock/Tauri -> Tauri Commands -> Rust`

### 阶段 B
- 可评估引入 SQLite、实时通信、更复杂状态管理
