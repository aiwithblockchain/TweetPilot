## 1. 核心架构原则

### 1.1 工作区优先（Workspace-First）

**TweetPilot V2 采用“工作区优先”架构。工作区不是应用内部再创建的一层产品壳，而是用户选中的本地工作目录。**

```
应用启动
  ↓
工作区选择器（唯一入口）
  ├─ 显示最近使用的工作区
  ├─ 搜索已有工作目录
  ├─ 新建工作区
  └─ 打开已有工作区目录
  ↓
选择目录后
  ↓
将该目录设置为当前工作区根目录
  ├─ 后续所有操作都以该目录为根
  ├─ 工作区名称取该目录最后一级文件夹名
  └─ 切换工作区 = 切换当前工作根目录
```

### 1.2 工作区的准确定义

1. **工作区 = 用户选中的本地目录**
2. **当前工作区 = 当前会话绑定的工作根目录**
3. **工作区名称 = `basename(workspaceRoot)`**
4. **新建工作区的核心动作只有一个：设置当前工作目录**

这意味着：
- 新建工作区时，不需要输入产品名称、描述、图标
- 不需要在该步骤绑定 Twitter 账号
- 不需要先建立“产品记录”再进入工作台

如果界面上仍保留“新建工作区”文案，其实际含义应为：

**选择一个目录作为新的工作区根目录，并在首次进入时完成 TweetPilot 所需初始化。**

### 1.3 为什么要这样设计

1. **符合 IDE 心智模型**：像 VS Code 的“打开文件夹”，而不是在应用内部再造一层项目系统
2. **根目录唯一**：数据库、上下文配置、skills、日志等都相对当前工作区根目录组织
3. **可移植性强**：迁移工作区本质上就是迁移一个目录
4. **实现更简单**：无需维护额外的产品主表和全局产品元数据

## 2. 工作区根目录模型

### 2.1 根目录即边界

每个工作区就是一个真实存在的本地目录。TweetPilot 的工作数据统一放在该目录下。

推荐结构：

```
/Users/me/projects/my-saas/
├── src/
├── docs/
├── package.json
└── .tweetpilot/
    ├── context.json
    ├── data/
    │   └── tweetpilot.db
    ├── config/
    │   └── settings.json
    ├── skills/
    └── logs/
        └── app.log
```

说明：
- `/Users/me/projects/my-saas/` 是工作区根目录
- `my-saas` 是工作区名称
- TweetPilot 自身产生的工作数据统一放在 `.tweetpilot/` 下
- 业务代码、文档、素材等保持原项目结构，不需要被 TweetPilot 改写

### 2.2 工作区名称与标识

工作区名称不再由用户录入，而是从目录名推导：

```typescript
import path from 'node:path';

function getWorkspaceName(workspaceRoot: string) {
  return path.basename(workspaceRoot);
}
```

例如：
- `/Users/me/projects/tweetpilot-demo` -> `tweetpilot-demo`
- `/Users/me/work/ai-writer` -> `ai-writer`

如果系统内部需要稳定标识，可以使用绝对路径或由绝对路径派生哈希值作为内部 ID，但这只是实现细节，不应影响用户创建流程。

### 2.3 工作区内的特定数据

以下内容都必须以当前工作区根目录为基准：

- 数据库
- 上下文配置
- workspace 级 settings
- skills
- logs
- 与当前工作区相关的缓存、索引、生成结果

统一要求：

```typescript
const metadataDir = path.join(workspaceRoot, '.tweetpilot');
const databasePath = path.join(metadataDir, 'data', 'tweetpilot.db');
const settingsPath = path.join(metadataDir, 'config', 'settings.json');
const skillsDir = path.join(metadataDir, 'skills');
```

## 3. 应用入口：工作区选择器

### 3.1 设计理念

**工作区选择器是应用的唯一入口。** 用户必须先选择一个本地目录作为当前工作区，才能进入工作台。

这类似于：
- **VS Code**：打开文件夹
- **IntelliJ IDEA**：打开项目目录
- **Cursor / Windsurf**：选择当前工作目录

### 3.2 工作区选择器功能

1. **显示最近使用的工作区**
   - 按最后使用时间排序
   - 显示目录名和目录路径
   - 点击即可进入

2. **搜索工作区**
   - 按目录名或目录路径搜索
   - 实时过滤结果

3. **新建工作区**
   - 选择一个新的本地目录，或先创建目录再选中
   - 将该目录设置为当前工作区根目录
   - 使用目录最后一级文件夹名作为项目名称
   - 若 `.tweetpilot/` 不存在，则自动初始化所需结构

4. **打开已有工作区**
   - 选择一个已有本地目录
   - 检查其中是否已有 `.tweetpilot/`
   - 有则直接加载，无则按首次工作区初始化

### 3.3 工作区选择器界面

参考原型：[product-selector.html](docs/v2/p0/prototypes/product-selector.html)

该原型可以继续作为交互参考，但语义需要统一为“选择工作目录”：
- “新建工作区”本质上是选择一个新的目录作为根目录
- “打开工作区”本质上是选择一个已有目录作为根目录

## 4. 工作台：工作区上下文

### 4.1 工作区上下文验证

**工作台必须建立在当前工作区根目录之上，不能脱离工作区上下文独立访问。**

验证逻辑：

```typescript
async function loadCurrentWorkspace() {
  const workspaceInitialized = localStorage.getItem('workspaceInitialized');
  const currentWorkspacePath = localStorage.getItem('currentWorkspacePath');

  if (!workspaceInitialized || !currentWorkspacePath) {
    alert('请先选择一个工作区');
    window.location.href = 'product-selector.html';
    return;
  }

  const exists = await fs.exists(currentWorkspacePath);
  if (!exists) {
    clearWorkspaceContext();
    alert('工作区目录不存在，请重新选择');
    window.location.href = 'product-selector.html';
    return;
  }

  await loadWorkspaceByPath(currentWorkspacePath);
}
```

### 4.2 工作区上下文对象

```typescript
interface WorkspaceContext {
  workspaceRoot: string;
  workspaceName: string;
  metadataDir: string;
  databasePath: string;
  settingsPath: string;
  skillsDir: string;
}

function buildWorkspaceContext(workspaceRoot: string): WorkspaceContext {
  return {
    workspaceRoot,
    workspaceName: path.basename(workspaceRoot),
    metadataDir: path.join(workspaceRoot, '.tweetpilot'),
    databasePath: path.join(workspaceRoot, '.tweetpilot', 'data', 'tweetpilot.db'),
    settingsPath: path.join(workspaceRoot, '.tweetpilot', 'config', 'settings.json'),
    skillsDir: path.join(workspaceRoot, '.tweetpilot', 'skills'),
  };
}
```

### 4.3 切换工作区

切换工作区 = 切换当前工作根目录：

1. 保存当前工作区状态
2. 清除当前工作区上下文
3. 返回工作区选择器
4. 用户重新选择目录
5. 按新目录构建新的 `WorkspaceContext`

## 5. 数据持久化设计

### 5.1 每个工作区独立数据库

**每个工作区拥有独立数据库文件**，路径位于：

`<workspaceRoot>/.tweetpilot/data/tweetpilot.db`

优点：
- 天然按目录隔离
- 备份和迁移简单
- 不需要额外全局产品表
- 当前工作区数据总是与目录绑定

### 5.2 表结构原则

由于数据库已经按工作区分库，P0 阶段不再要求每张表额外带 `product_id` 字段。

示例：

```sql
CREATE TABLE tasks (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  name TEXT NOT NULL,
  status TEXT NOT NULL,
  config TEXT NOT NULL,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL
);

CREATE TABLE tweets (
  id TEXT PRIMARY KEY,
  account_id TEXT NOT NULL,
  content TEXT NOT NULL,
  author TEXT NOT NULL,
  likes INTEGER NOT NULL,
  retweets INTEGER NOT NULL,
  replies INTEGER NOT NULL,
  created_at DATETIME NOT NULL
);

CREATE TABLE comments (
  id TEXT PRIMARY KEY,
  tweet_id TEXT NOT NULL,
  content TEXT NOT NULL,
  author TEXT NOT NULL,
  likes INTEGER NOT NULL,
  sentiment TEXT,
  created_at DATETIME NOT NULL
);
```

## 6. 接口设计原则

### 6.1 通过 WorkspaceContext 传递上下文

所有模块应依赖 `WorkspaceContext`，而不是传递 `productId`。

```typescript
interface ITaskManagement {
  setContext(context: WorkspaceContext): void;
  listTasks(): Promise<Task[]>;
  createTask(data: CreateTaskDTO): Promise<Task>;
}

interface ISettingsService {
  setContext(context: WorkspaceContext): void;
  loadSettings(): Promise<WorkspaceSettings>;
  saveSettings(data: WorkspaceSettings): Promise<void>;
}
```

### 6.2 路径访问规则

所有工作区相关访问都必须从 `workspaceRoot` 派生，禁止硬编码全局路径：

```typescript
// ❌ 错误：写死到用户主目录
const dbPath = '~/.tweetpilot/workspaces/demo/data/tweetpilot.db';

// ✅ 正确：从当前工作区上下文派生
const dbPath = context.databasePath;
```

## 7. 实现要点

### 7.1 应用启动流程

```typescript
async function startApp() {
  const currentWorkspacePath = localStorage.getItem('currentWorkspacePath');

  if (!currentWorkspacePath) {
    showWorkspaceSelector();
    return;
  }

  const exists = await fs.exists(currentWorkspacePath);
  if (!exists) {
    clearWorkspaceContext();
    showWorkspaceSelector();
    return;
  }

  await ensureWorkspaceInitialized(currentWorkspacePath);
  await loadWorkspaceByPath(currentWorkspacePath);
  showWorkbench();
}
```

### 7.2 选择工作区流程

```typescript
async function selectWorkspace(workspaceRoot: string) {
  await ensureWorkspaceInitialized(workspaceRoot);

  localStorage.setItem('workspaceInitialized', 'true');
  localStorage.setItem('currentWorkspacePath', workspaceRoot);
  localStorage.setItem('currentWorkspaceName', path.basename(workspaceRoot));

  window.location.href = 'workbench.html';
}
```

### 7.3 首次初始化流程

```typescript
async function ensureWorkspaceInitialized(workspaceRoot: string) {
  const metadataDir = path.join(workspaceRoot, '.tweetpilot');

  await fs.mkdir(path.join(metadataDir, 'data'), { recursive: true });
  await fs.mkdir(path.join(metadataDir, 'config'), { recursive: true });
  await fs.mkdir(path.join(metadataDir, 'skills'), { recursive: true });
  await fs.mkdir(path.join(metadataDir, 'logs'), { recursive: true });

  const contextPath = path.join(metadataDir, 'context.json');
  const context = {
    workspaceRoot,
    workspaceName: path.basename(workspaceRoot),
    createdAt: new Date().toISOString(),
    updatedAt: new Date().toISOString(),
    version: '2.0.0',
  };

  await fs.writeFile(contextPath, JSON.stringify(context, null, 2));

  const db = await initDatabase(path.join(metadataDir, 'data', 'tweetpilot.db'));
  await db.migrate();
}
```

### 7.4 清除工作区上下文

```typescript
function clearWorkspaceContext() {
  localStorage.removeItem('workspaceInitialized');
  localStorage.removeItem('currentWorkspacePath');
  localStorage.removeItem('currentWorkspaceName');
}
```
