## 3. 执行总原则（必须遵守）

1. **接口先行**：每个模块先完成 `types.ts`（类型 + interface），再写实现。
2. **Mock 优先**：先用内存假数据跑通全流程，不依赖真实外部系统。
3. **UI 只依赖 service**：组件层禁止继续新增 `invoke()`。
4. **单模块闭环**：每次开发必须包含“实现 + 测试 + 验收记录”。
5. **小步可回归**：每个模块完成后可独立回归，不等待全量完成。
6. **不扩 scope**：禁止新增 P0 以外功能。

## 5. 统一开发流程（每个模块都按此顺序）

### Step 1：冻结接口

- 在 `types.ts` 定义：
  - 领域类型
  - Service Interface
- 与 [P0-03-接口设计规范.md](./P0-03-接口设计规范.md) 对齐。

### Step 2：补齐 mock-data

- 在 `src/services/mock-data/*.ts` 提供默认数据。
- 覆盖：空态、正常态、错误态、长文本/多数据量（按模块需要）。

### Step 3：实现 `mock.ts`

- 使用内存状态（运行时可变）。
- 所有方法返回 Promise。
- 模拟延迟：
  - 读：50-150ms
  - 写：100-250ms
  - 执行：300-1500ms
- 提供可控错误分支（账号不存在/任务不存在/配置非法等）。

### Step 4：实现 `tauri.ts`（替换位）

- 只做适配层，不写业务逻辑。
- 做字段映射（snake_case / camelCase）。

### Step 5：改造 UI 接入

- 页面组件改为调用 `xxxService`。
- 删除组件中的直接 `invoke()`。

### Step 6：模块测试与验收

- 执行构建检查：`npm run build`
- 执行模块场景测试（见第 8 节）
- 记录验收结果（见第 10 节模板）

## 7. 每次开发任务的最小交付标准（DoD）

每次 AI 完成一个模块/子模块，必须同时满足：

1. **代码完成**：接口、mock、UI 接入都完成
2. **构建通过**：`npm run build` 成功
3. **行为通过**：模块测试清单全部通过
4. **回归通过**：不破坏已完成模块关键路径
5. **记录完整**：提交“修改文件 + 测试结果 + 未解决问题”

## 8. 测试与验收清单（执行版）

### 8.1 通用检查（每次必做）

- [ ] `npm run build` 成功
- [ ] 页面无明显运行时报错
- [ ] 关键交互无阻塞（弹窗、保存、删除、刷新）

### 8.2 状态覆盖检查（每模块必做）

- [ ] 空状态
- [ ] 正常状态
- [ ] 错误状态
- [ ] 长文本/大数据量（适用时）

### 8.3 关键场景回归（阶段性）

- [ ] 工作目录选择 + 进入主界面
- [ ] 账号映射 + 状态刷新 + 账号设置保存
- [ ] 任务创建 + 执行 + 查看详情/历史
- [ ] 数据积木添加/拖拽/刷新/删除
- [ ] 偏好设置修改与生效

## 9. 明确禁止事项

1. 禁止在页面组件中继续新增 `invoke()`。
2. 禁止跳过 `types.ts` 直接写实现。
3. 禁止把 mock 写成不可变死数据（必须支持运行时状态变化）。
4. 禁止在本阶段接入真实 Twitter API / LocalBridge / Python 实际执行。
5. 禁止新增 P0 范围外功能。
