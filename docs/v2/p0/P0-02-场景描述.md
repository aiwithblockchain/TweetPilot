# P0-02-场景描述

## 文档概述

本文档详细描述 P0 阶段 A 的核心使用场景和操作流程，确保每个功能都有明确的用户路径和交互逻辑。

## 场景 1：确定工作目录

### 场景描述

用户首次启动应用或需要切换项目时，必须先确定当前的工作目录。工作目录是整个应用的根目录，所有数据、配置、AI 对话都基于此目录进行。

### 用户目标

- 快速选择或创建工作目录
- 从历史记录中恢复之前的工作环境
- 从 GitHub 克隆项目作为新的工作目录

### 操作流程

#### 1.1 首次启动应用

```
用户启动应用
  ↓
显示欢迎界面
  ↓
提供三个选项：
  - 选择本地目录
  - 从 GitHub 克隆
  - 查看最近使用
  ↓
用户选择其中一个选项
  ↓
确认工作目录
  ↓
进入主界面
```

**详细步骤**：

1. **选择本地目录**
   - 点击"选择本地目录"按钮
   - 系统打开文件选择器
   - 用户浏览并选择目标目录
   - 系统验证目录权限
   - 确认并设置为当前工作目录
   - 将目录路径添加到历史记录

2. **从 GitHub 克隆**
   - 点击"从 GitHub 克隆"按钮
   - 输入 GitHub 仓库 URL（支持 HTTPS 和 SSH）
   - 选择本地存储位置
   - 系统执行 `git clone` 操作
   - 显示克隆进度
   - 克隆完成后自动设置为当前工作目录
   - 将目录路径添加到历史记录

3. **查看最近使用**
   - 点击"最近使用"标签
   - 显示最近 10 个工作目录列表
   - 每个条目显示：
     - 目录名称
     - 完整路径（压缩显示，如 `~/projects/...`）
     - 最后访问时间
   - 点击任一条目即可切换到该工作目录

#### 1.2 打开新的工作目录

```
用户在主界面
  ↓
点击菜单栏"文件" → "打开工作目录"
  ↓
显示工作目录选择器（与首次启动相同）
  ↓
选择新的工作目录
  ↓
系统启动新的 TweetPilot 实例
  ↓
新实例加载选定的工作目录
  ↓
原实例继续运行，互不干涉
```

**架构说明**：
- 一个 TweetPilot 实例 = 一个工作目录
- 每个实例是独立的进程，互不影响
- 类似 IDE 的"打开工作目录"，而不是"切换"
- 用户可以同时运行多个 TweetPilot 实例，管理不同的工作目录

### 界面元素

**欢迎界面**：
- 应用 Logo 和标题
- 三个大按钮：
  - "选择本地目录"（图标：文件夹）
  - "从 GitHub 克隆"（图标：GitHub logo）
  - "最近使用"（图标：时钟）

**主界面顶部栏**：
- 工作目录路径显示（只读，不可点击）
- 路径压缩显示（如：`~/projects/twitter-bot`）
- 悬停显示完整路径

**菜单栏**：
- "文件" → "打开工作目录"：启动新的 TweetPilot 实例

### 数据存储

- 当前工作目录路径：`~/.tweetpilot/config.json`
- 历史记录：`~/.tweetpilot/recent-workspaces.json`（最多 10 条）

## 场景 2：管理 Twitter 账号状态

### 场景描述

用户需要映射、删除和监控 Twitter 账号的连接状态，确保任务可以正常执行。当账号掉线时，用户需要能够快速发现并重新建立连接。

**架构说明**：
- TweetPilot 是 Electron 桌面应用，不直接管理浏览器 Cookie/Session
- 用户在真实浏览器（Chrome/Edge）中登录 Twitter
- TweetPilot 通过 LocalBridge 与浏览器扩展通信
- 账号映射基于 `screen_name`（@username）

### 用户目标

- 映射已登录的 Twitter 账号
- 查看所有账号的实时状态
- 删除不再使用的账号映射
- 重新建立掉线账号的连接

### 操作流程

#### 2.1 映射 Twitter 账号

```
用户在真实浏览器中登录 Twitter
  ↓
用户进入 TweetPilot 设置界面
  ↓
点击"映射账号"按钮
  ↓
系统通过 LocalBridge 获取已连接的扩展 ID
  ↓
查询扩展中当前登录的 Twitter 账号列表
  ↓
显示可映射的账号列表（screen_name）
  ↓
用户选择要映射的账号
  ↓
系统记录映射关系（screen_name → extension_id + tab_id）
  ↓
验证账号连接状态
  ↓
显示账号信息（用户名、头像、状态）
  ↓
账号映射成功
  ↓
返回设置界面
```

**详细步骤**：

1. **前置条件**：用户已在浏览器中登录 Twitter 账号
2. 点击"映射账号"按钮
3. 系统通过 LocalBridge 获取已连接的浏览器扩展 ID
4. 系统查询扩展中当前登录的 Twitter 账号信息
5. 显示可映射的账号列表，每个账号显示：
   - screen_name（@username）
   - 显示名称
   - 头像
6. 用户选择要映射的账号
7. 系统建立映射关系：
   - screen_name → extension_id
   - screen_name → tab_id
8. 系统通过 LocalBridge 验证账号连接状态
9. 将映射配置存储到工作目录的 `accounts/` 目录
10. 在账号列表中显示新映射的账号

#### 2.2 查看账号状态

```
用户进入设置界面
  ↓
显示所有账号列表
  ↓
每个账号显示：
  - 头像
  - 用户名
  - 状态指示器（在线/离线）
  - 最后验证时间
  ↓
用户可点击"刷新状态"按钮
  ↓
系统重新验证所有账号
  ↓
更新状态显示
```

**状态指示器**：
- 🟢 在线：账号连接正常，可以执行任务
- 🔴 离线：账号连接失败，需要重新建立连接
- 🟡 验证中：正在验证账号状态

**映射配置的检查机制**：
- **启动时检查**：TweetPilot 启动时自动检查并更新所有账号的映射配置
- **定时检查**：每 5 分钟自动验证映射配置的有效性
- **失败重试**：LocalBridge 请求失败时，立即重新检查并更新连接信息
- **状态同步**：extension_id 和 tab_id 是临时的，浏览器重启或扩展重载后会变化，必须及时更新

#### 2.3 重新建立掉线账号的连接

```
用户发现账号状态为"离线"
  ↓
用户在浏览器中重新登录该 Twitter 账号
  ↓
返回 TweetPilot，点击账号卡片上的"重新连接"按钮
  ↓
系统通过 LocalBridge 重新查询扩展状态
  ↓
验证账号连接
  ↓
更新 tab_id 映射
  ↓
状态更新为"在线"
```

#### 2.4 删除账号映射

```
用户点击账号卡片上的"删除"按钮
  ↓
显示确认对话框
  ↓
用户确认删除
  ↓
系统删除账号映射配置
  ↓
从列表中移除账号
```

### 界面元素

**设置界面 - 账号管理区域**：
- 标题："Twitter 账号管理"
- "映射账号"按钮（主要按钮，显眼位置）
- 账号列表（网格布局）

**账号卡片**：
- 头像（圆形，64x64）
- 用户名（@username）
- 状态指示器（带颜色的圆点 + 文字）
- 最后验证时间（相对时间，如"5 分钟前"）
- 操作按钮：
  - "刷新状态"（图标按钮）
  - "重新连接"（仅在离线时显示）
  - "删除"（图标按钮，次要样式）

### 数据存储

- 账号映射配置：`{workspace}/accounts/accounts.json`
- 账号映射关系：`{workspace}/accounts/{screen_name}/mapping.json`（包含 extension_id 和 tab_id）
- 账号基本信息：`{workspace}/accounts/{screen_name}/profile.json`

## 场景 3：管理定时任务

### 场景描述

用户需要创建、配置和监控 Python 脚本任务的执行。任务可以是定时执行（如每小时发一条推文）或即时执行（如立即抓取某个用户的推文）。

### 用户目标

- 创建新任务并上传 Python 脚本
- 配置任务参数和执行规则
- 查看任务执行状态和历史
- 暂停/恢复/删除任务
- 立即执行任务并查看结果

### 操作流程

#### 3.1 创建新任务

```
用户进入任务管理界面
  ↓
点击"创建任务"按钮
  ↓
显示任务配置对话框
  ↓
填写任务信息：
  - 任务名称
  - 任务描述
  - 上传 Python 脚本文件
  - 选择任务类型（定时/即时）
  - 配置执行规则（如果是定时任务）
  - 配置任务参数
  ↓
点击"创建"按钮
  ↓
系统验证配置
  ↓
保存任务配置和脚本文件
  ↓
任务添加到列表
```

**详细步骤**：

1. **填写基本信息**
   - 任务名称：必填，最多 50 字符
   - 任务描述：可选，最多 200 字符

2. **上传 Python 脚本**
   - 点击"上传脚本"按钮
   - 选择本地 `.py` 文件
   - 系统验证文件格式
   - 显示文件名和大小

3. **选择任务类型**
   - 定时任务：按照设定的时间规则自动执行
   - 即时任务：仅在用户手动触发时执行

4. **配置定时规则**（仅定时任务）
   - 简单模式：
     - 每 N 分钟/小时/天执行一次
     - 下拉选择时间单位
     - 输入数字
   - 高级模式：
     - 输入 Cron 表达式
     - 显示下次执行时间预览

5. **配置任务参数**
   - 参数以键值对形式输入
   - 支持添加多个参数
   - 参数会作为命令行参数传递给 Python 脚本
   - 示例：`--target-user=elonmusk --max-tweets=100`

6. **创建任务**
   - 点击"创建"按钮
   - 系统验证所有必填字段
   - 保存任务配置到 `{workspace}/tasks/{task_id}/config.json`
   - 复制脚本文件到 `{workspace}/tasks/{task_id}/script.py`
   - 任务初始状态为"已暂停"（用户需手动启动）

#### 3.2 查看任务列表

```
用户进入任务管理界面
  ↓
显示所有任务列表
  ↓
每个任务显示：
  - 任务名称
  - 任务类型（定时/即时）
  - 当前状态（运行中/已暂停/已完成/失败）
  - 下次执行时间（定时任务）
  - 最后执行时间
  - 快速操作按钮
```

**任务状态**：
- 🟢 运行中：任务正在执行
- ⏸️ 已暂停：任务已暂停，不会自动执行
- ✅ 已完成：任务执行成功
- ❌ 失败：任务执行失败

**快速操作**：
- 暂停/恢复按钮
- 立即执行按钮
- 查看详情按钮
- 删除按钮

#### 3.3 查看任务详情

```
用户点击任务卡片或"查看详情"按钮
  ↓
显示任务详情侧边栏
  ↓
显示内容：
  - 任务配置信息
  - 执行统计数据
  - 执行历史列表
  ↓
用户可以：
  - 编辑任务配置
  - 查看执行日志
  - 下载执行结果
```

**任务详情内容**：

1. **配置信息**
   - 任务名称和描述
   - 任务类型
   - 执行规则（定时任务）
   - 任务参数
   - 脚本文件名

2. **统计数据**
   - 总执行次数
   - 成功次数
   - 失败次数
   - 成功率（百分比）
   - 平均执行时长

3. **执行历史**（最近 20 条）
   - 执行时间
   - 执行状态（成功/失败）
   - 执行时长
   - 输出摘要（前 100 字符）
   - 点击可查看完整日志

#### 3.4 立即执行任务

```
用户点击"立即执行"按钮
  ↓
系统验证账号状态
  ↓
显示执行确认对话框
  ↓
用户确认
  ↓
系统启动 Python 脚本
  ↓
显示执行进度（实时输出）
  ↓
执行完成
  ↓
显示执行结果
  ↓
更新任务统计和历史
```

**执行过程**：

1. 验证关联的 Twitter 账号是否在线
2. 如果账号离线，提示用户先重新登录
3. 显示确认对话框，包含任务名称和参数
4. 用户确认后，系统执行：
   ```bash
   python {workspace}/tasks/{task_id}/script.py {params}
   ```
5. 实时显示脚本的标准输出和标准错误
6. 执行完成后，记录执行结果：
   - 执行时间
   - 执行时长
   - 退出码
   - 完整输出
7. 更新任务统计数据
8. 在执行历史中添加新记录

#### 3.5 暂停/恢复任务

```
用户点击"暂停"按钮
  ↓
任务状态更新为"已暂停"
  ↓
定时任务停止自动执行
  ↓
按钮文字变为"恢复"
```

```
用户点击"恢复"按钮
  ↓
任务状态更新为"运行中"
  ↓
定时任务恢复自动执行
  ↓
按钮文字变为"暂停"
```

#### 3.6 删除任务

```
用户点击"删除"按钮
  ↓
显示确认对话框
  ↓
用户确认删除
  ↓
系统停止任务（如果正在运行）
  ↓
删除任务配置和脚本文件
  ↓
从列表中移除任务
```

### 界面元素

**任务管理主界面**：
- 标题："任务管理"
- "创建任务"按钮（主要按钮）
- 任务列表（卡片布局）
- 筛选器：全部/运行中/已暂停/失败

**任务卡片**：
- 任务名称（大字体）
- 任务类型标签（定时/即时）
- 状态指示器（带颜色的图标 + 文字）
- 下次执行时间（定时任务）
- 最后执行时间
- 快速操作按钮行：
  - 暂停/恢复
  - 立即执行
  - 查看详情
  - 删除

**任务详情侧边栏**：
- 关闭按钮
- 标签页：
  - 配置
  - 统计
  - 历史
- 编辑按钮（在配置标签页）

### 数据存储

- 任务配置：`{workspace}/tasks/{task_id}/config.json`
- Python 脚本：`{workspace}/tasks/{task_id}/script.py`
- 执行历史：`{workspace}/tasks/{task_id}/history.jsonl`（每行一条记录）
- 执行日志：`{workspace}/tasks/{task_id}/logs/{execution_id}.log`

## 场景 5：数据积木

### 场景描述

用户需要以可视化方式查看 Twitter 数据的简单加工结果，如推文统计、账号数据、互动数据等。数据以网格卡片形式展示，用户可以自由添加、删除和排列卡片。

### 用户目标

- 快速了解账号的数据概况
- 查看推文的基本统计信息
- 监控账号的互动数据
- 自定义数据展示布局

### 操作流程

#### 5.1 查看数据积木

```
用户进入数据积木界面
  ↓
显示网格布局的数据卡片
  ↓
每个卡片显示一种数据类型
  ↓
用户可以：
  - 查看卡片内容
  - 刷新卡片数据
  - 删除卡片
  - 添加新卡片
  - 拖拽调整卡片位置
```

#### 5.2 添加数据卡片

```
用户点击"添加卡片"按钮
  ↓
显示卡片类型选择器
  ↓
显示所有可用的卡片类型（最多 9 种）
  ↓
用户选择一种类型
  ↓
卡片添加到网格末尾
  ↓
自动加载数据并显示
```

#### 5.3 刷新数据

```
用户点击卡片上的"刷新"按钮
  ↓
显示加载状态
  ↓
系统重新获取数据
  ↓
更新卡片内容
  ↓
显示最后更新时间
```

### 数据卡片类型

#### 类型 1：账号推文统计

**显示内容**：
- 卡片标题："推文统计"
- 每个账号一行：
  - 账号头像（小）
  - 账号用户名
  - 推文总数（大字体）

**数据来源**：
- 调用 Twitter API 获取每个账号的推文数量
- 或从本地数据库统计

#### 类型 2：最新推文列表

**显示内容**：
- 卡片标题："最新推文"
- 账号选择器（下拉菜单）
- 推文列表（最新 10 条）：
  - 推文内容（截断显示，最多 100 字符）
  - 发布时间（相对时间）
  - 互动数据：
    - 👁️ 浏览数
    - ❤️ 点赞数
    - 💬 回复数
    - 🔄 转发数

**交互**：

## 场景流程总结

### 典型用户工作流

```
1. 启动应用 → 选择工作目录（场景 1）
   ↓
2. 进入设置 → 映射 Twitter 账号（场景 2）
   ↓
3. 进入任务管理 → 创建新任务（场景 3）
   ↓
4. 配置任务参数 → 立即执行测试（场景 3）
   ↓
5. 查看执行结果 → 调整任务配置（场景 3）
   ↓
6. 启动定时任务 → 任务自动执行（场景 3）
   ↓
7. 进入数据积木 → 查看数据统计（场景 5）
   ↓
8. 监控任务执行 → 查看账号状态（场景 2）
```

**多工作目录管理**：
- 用户可以通过"文件" → "打开工作目录"启动新的 TweetPilot 实例
- 每个实例独立运行，管理各自的工作目录
- 实例之间互不干涉，可以同时运行多个实例
