# P0-04-阶段A开发计划

## 文档概述

本文档描述 TweetPilot P0 阶段 A 的**当前执行计划**。

和早期版本不同，本版基于当前项目状态更新：
- 核心 UI 已基本完成
- 下一步重点从“搭界面”转为“模块抽象 + 接口对齐 + Mock 驱动”
- 开发流程采用：**现状收敛 → 接口冻结 → Mock 跑通 → 真实实现替换 → 联调测试**

---

## 当前状态（2026-04）

### 已完成（高优先）
- Tauri + React 基础工程已可运行
- 工作目录主流程、账号管理主流程、任务/数据积木/设置页面均有可用 UI
- 账号设置弹窗、账号映射弹窗、LocalBridge 配置项等关键交互已落地

### 当前主要风险
1. 组件中仍存在直接 `invoke()` 调用，接口边界不稳定
2. 文档与代码存在轻微漂移，容易让后续 AI 误解当前阶段
3. 还未形成完整的 Mock-first 服务层，真实实现替换成本偏高

---

## 阶段A执行原则

1. **先稳接口，再做实现**：先冻结模块接口，再写 Mock 与真实实现
2. **UI 不直连后端**：UI 只能依赖前端 service interface
3. **Mock 先跑通全流程**：所有核心场景先在内存假数据下跑通
4. **逐模块替换真实逻辑**：按模块从 Mock 切换到 Tauri/Rust 实现
5. **每次改动都可验证**：每个阶段产出都可单独验收

---

---

## 阶段 2：Service 层与 Mock 实现（下一步）

### 目标
搭建前端服务层，完成 Mock 版全链路。

### 工作内容
1. 建立目录：
   - `src/services/{workspace,account,task,data-blocks,settings}/`
2. 每个模块落地：
   - `types.ts`（接口和类型）
   - `mock.ts`（内存假数据实现）
   - `index.ts`（模块导出）
3. 统一入口：`src/services/index.ts`
4. 提供默认假数据：`src/services/mock-data/*`

### 交付物
- 可运行的 Mock 服务层
- 所有 UI 页面可在无真实后端情况下完整运行

### 验收标准
- [ ] 所有核心接口都有 mock 实现
- [ ] 用户操作能改变内存数据并即时反馈 UI
- [ ] 覆盖空态/正常态/错误态

---

## 阶段 3：UI 接口切换与收敛

### 目标
把组件中的直接调用切换到 service interface。

### 工作内容
- 替换散落在组件中的 `invoke()`
- 统一改为 `workspaceService / accountService / taskService / dataBlocksService / settingsService`
- 收敛重复状态逻辑，减少组件层业务代码

### 交付物
- UI 完全通过 service interface 工作

### 验收标准
- [ ] 页面组件无直接 `invoke()`
- [ ] 页面行为与改造前一致
- [ ] 关键流程回归通过（账号映射、账号设置、任务执行、数据积木刷新、设置保存）

---

## 阶段 4：真实实现替换（Tauri/Rust）

### 目标
逐模块把 Mock 替换为真实后端能力。

### 推荐替换顺序
1. workspace
2. settings
3. account
4. task
5. data-blocks

### 工作内容
- 每个模块新增 `tauri.ts`
- 对接对应 Tauri Commands
- 做字段映射（camelCase/snake_case）
- 逐模块切换运行实现

### 交付物
- 可在真实本地环境运行的阶段 A 核心功能

### 验收标准
- [ ] 每个模块都有真实实现
- [ ] mock 与真实行为语义一致
- [ ] 切换不需要改 UI 组件

---
