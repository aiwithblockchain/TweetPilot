## 1. 文档目的

本文档定义 TweetPilot V2 在 P0 阶段 A 的模块边界、接口契约和前后端通信规范。

它的目标不是直接描述 UI，也不是直接描述 Rust 实现细节，而是把下面几件事对齐：

1. UI 需要什么能力
2. 前端 Service Interface 应该长什么样
3. Tauri Commands 应该暴露什么边界
4. Mock 实现和真实实现如何共用同一份契约

## 2. 核心设计原则

### 2.1 UI 驱动，但不让 UI 直接绑定实现

接口由 UI 和场景驱动，但 UI 不直接依赖：

- 某个 Mock 类
- 某个具体 Tauri Command 名称
- 某个临时数据结构

UI 只能依赖稳定的前端 service interface。

### 2.2 单一职责

每个模块只做自己的事：

- workspace 只管工作目录
- account 只管账号映射和状态
- task 只管任务配置和执行
- data-blocks 只管数据积木布局和数据
- settings 只管系统设置

不要让一个模块偷偷接管另一个模块的状态。

### 2.3 Mock 先于真实实现

P0 当前阶段的顺序固定为：

```text
UI 场景确认
  ↓
定义前端 Service Interface
  ↓
定义对应的 Tauri Commands 契约
  ↓
实现 Mock 版本
  ↓
UI 接入 Mock
  ↓
实现真实 Tauri / Rust 逻辑
```

### 2.4 前端接口与 Tauri Command 分层

统一分层如下：

```text
UI 组件
  ↓
Service Interface
  ↓
Mock 实现 / Tauri 实现
  ↓
Tauri Commands
  ↓
Rust Services / Store / External IO
```

说明：

- `Service Interface` 是 UI 的稳定依赖
- `Tauri Commands` 是真实运行时的跨边界调用接口
- Mock 和 Tauri 实现必须在语义上保持一致

## 4. 通用类型约定

### 4.1 时间字段

所有时间字段统一使用 ISO 8601 字符串：

```typescript
export type ISODateString = string
```

例如：
- `2026-04-16T08:00:00Z`

### 4.2 Result 风格

前端 service interface 统一直接返回业务数据或抛错：

```typescript
const tasks = await taskService.getTasks()
```

而 Tauri Command 侧统一使用：

```rust
Result<T, String>
```

例如：

```rust
#[tauri::command]
pub async fn get_tasks() -> Result<Vec<Task>, String>
```

### 4.3 命名规范

#### 前端
- Interface 使用 PascalCase，以 `I` 开头
- 类型使用 PascalCase
- 方法使用 camelCase

#### Rust
- Command 函数使用 snake_case
- Struct 使用 PascalCase
- 字段序列化时通过 serde 与前端字段对齐

## 10. Mock 设计规范

### 10.1 文件位置

建议位置：`src/services/{module}/mock.ts`

### 10.2 数据要求

Mock 数据必须覆盖：
- 正常态
- 空态
- 错误态
- 长文本
- 多条记录
- 状态变化

### 10.3 行为要求

Mock 必须模拟：
- 异步延迟
- 常见错误
- 状态更新
- 列表增删改
- 历史记录增长

### 10.4 禁止事项

Mock 阶段禁止：
- 真实文件写入
- 真实外部网络请求
- 真实 Python 执行
- 真实 LocalBridge 调用

## 11. Tauri 实现适配规范

### 11.1 前端 Tauri 适配层

建议位置：`src/services/{module}/tauri.ts`

职责：
- 调用 `src/lib/tauri-api.ts` 或直接封装 `invoke`
- 把 Tauri Command 的返回值映射为前端 service interface 定义的数据
- 处理字段命名转换、默认值和错误信息整理

### 11.2 组件禁止直接调用 invoke

组件中禁止继续散落如下代码：

```typescript
invoke('get_tasks')
```

组件只能这样使用：

```typescript
const tasks = await taskService.getTasks()
```

这条规则很重要。不然你名义上有接口层，实际上没有。

## 15. 结论

P0 阶段 A 的接口设计，不应该是“先有一堆 Rust Command，再让 UI 去适配”，也不应该是“先让 UI 乱连，再回头补接口”。

正确做法是：

1. 用场景确定模块边界
2. 用模块定义 service interface
3. 用 service interface 约束 mock 和 tauri 实现
4. 用 Tauri Commands 承接真实跨边界调用

这样做的结果是：

- UI 稳定
- Mock 可跑
- 真实实现可替换
- 多个 AI 可以并行开发，不会各写各的世界观

这才像一个能持续推进的工程，而不是一堆刚好能跑的按钮。
