# P0-14 VSCode 风格 UI 重构方案 v3.0

## 文档信息

- 版本：v3.0（信息架构重构版）
- 创建日期：2026-04-18
- 适用阶段：TweetPilot V2 / P0-14 UI 重构
- 设计目标：纠正当前主界面的导航逻辑，让左侧栏承担“列表/树/入口”，让中间主显示区承担“详情/编辑”，并顺手统一混乱的卡片与面板视觉系统

---

## 1. 本次重构的核心结论

当前 UI 最大的问题，不是少几个页面，而是信息架构错位了。

现在左侧 Activity Bar 一点击，就把内容直接切到中间主显示区。这个逻辑不对，也不符合 VS Code、Finder、IDE、桌面工作台类产品的通用认知模型。

这次重构后的正确模型应该是：

- Activity Bar 只负责切换“当前工作域”
- Left Sidebar 负责展示这个工作域下的列表、树、入口和轻操作
- Center Panel 只在用户选中了左侧具体内容之后，才显示详情或编辑态
- Settings 不再作为普通页面占据主显示区，而是作为独立 dialog 打开
- 左侧列表项回归轻量扫描型设计，主区内容才使用更完整的卡片/详情块

一句话总结：

> 左侧是工作面板，中间是详情视图，右侧是辅助面板，设置是全局弹窗。

这才是这套壳层应该有的样子。

---

## 2. 当前问题复盘

### 2.1 导航层级混乱

当前交互的问题在于：

- 点击 Activity Bar 后，主显示区直接切页
- 左侧栏没有成为真正的“工作面板”
- 用户无法形成“先选列表，再看详情”的稳定操作节奏
- 主显示区同时承担“首页、导航页、详情页、编辑页”多种职责，边界不清楚

结果就是：

- 中间区域过载
- 左侧区域存在感不足
- 后面每增加一个功能，结构都会越来越乱

### 2.2 左侧栏没有承担真正职责

左侧栏本来应该承载的是：

- 文件树
- 账号列表
- 数据积木列表
- 任务列表
- 顶部轻操作，例如新建、刷新、过滤

但当前很多内容都直接跳到主显示区，导致左侧栏更像“装饰区域”，而不是工作流入口。

### 2.3 卡片样式失控

当前视觉上的问题也很明显：

- 左侧列表项和主区详情块混用了同一种重卡片语言
- 圆角、留白、边框、背景层级不统一
- 该轻的地方太重，该重的地方又没有足够层次
- 扫描型内容被做成“阅读型卡片”，导致视觉噪音增加

这会让 UI 看起来像是局部拼接，而不是一个有纪律的系统。

## 3. 本次重构目标

### 3.1 交互目标

建立统一的三层工作流：

1. Activity Bar 切换工作域
2. Left Sidebar 展示该工作域下的结构化列表/树/入口
3. Center Panel 展示当前选中项的详情、预览或编辑态

### 3.2 产品目标

让以下五类内容都进入同一种可预测工作流：

1. 工作区
2. 推特账号
3. 数据积木
4. 任务
5. 设置

### 3.3 视觉目标

顺手把当前混乱的样式收回来，建立稳定的 UI 纪律：

- 左侧列表轻量化
- 主区详情层次化
- 状态信息弱化但不丢失
- 卡片只用于真正需要阅读的内容
- 设置入口与主工作域入口在视觉上明确区隔

### 3.4 架构目标

为后续继续扩展打基础：

- 文件树能力
- 文件预览能力
- 推特账号详情
- 账号关联实例信息
- 数据积木详情与创建
- 任务详情与新建
- 设置弹窗体系

## 5. 五个菜单的重构方案

## 5.1 工作区 Workspace

### 左侧栏职责

工作区是典型的“树 + 详情”结构，左侧栏应直接承担文件浏览职责。

左侧应显示：

- 当前工作区名称
- 新建文件
- 新建文件夹
- 刷新目录
- 当前目录树

建议结构：

- 顶部工具栏
  - 当前目录名
  - 新建文件按钮
  - 新建文件夹按钮
  - 刷新按钮
- 内容区
  - 文件夹树
  - 文件列表
  - 展开/收起
  - 选中态

### 中间主显示区职责

点击左侧文件后，中间主区显示文件详情：

- `txt` 文本，显示文本内容预览
- 图片，显示图片预览
- 其它文件类型，先显示 TODO 占位页

如果没有选中文件，中间主区显示：

- 当前工作区空状态
- 或“请选择左侧文件查看内容”的引导提示

### 本阶段能力范围

本次先实现：

- 浏览当前目录
- 新建文件
- 新建文件夹
- 刷新
- `txt` 文本预览
- 简单图片预览
- 其它文件类型 TODO

### 设计理由

这是最符合桌面工具习惯的方式：

- 左边找文件
- 中间看内容
- 左边扫描快
- 中间阅读深

工作区绝不能继续被做成“点击图标后主区直接切首页”的结构。

## 5.2 推特账号 Accounts

### 说明

原方案中的“TweetClaw 实例与账号”在这一版中改为“推特账号”。

原因很简单：

- 账号和实例是一对一关系
- 不需要在左侧同时维护两个平行列表
- 左侧只展示推特账号即可
- 账号关联的实例信息，放到主显示区详情里展示

这样更简洁，也更符合用户认知。

### 左侧栏职责

左侧显示推特账号列表。

顶部可保留：

- 标题：Twitter Accounts / 推特账号
- 刷新按钮

列表项本次先保持简洁，只显示最基本信息：

- 头像
- 账号名称 / 昵称
- 用户名（可选）
- 在线状态或简单状态标记（可选）

除了头像和名字外，其它字段本轮可以为空或占位，后面再补。

### 中间主显示区职责

点击左侧某个推特账号时，主显示区展示：

- 账号头像
- 账号名称
- 用户名（可空）
- 账号状态（可空）
- 关联实例信息
  - 实例名称
  - 实例 ID
  - 在线/离线状态
  - 最近活跃时间

如果当前没有数据，允许用 mock 或空态占位。

### 设计理由

实例信息虽然重要，但它是账号的附属上下文，不需要在左侧占据与账号并列的一层导航结构。

先让用户看到“账号”，再在详情页里看到“这个账号绑定了哪个实例”，认知成本更低。

## 5.3 数据积木 Data Blocks

### 左侧栏职责

左侧显示数据积木列表，顶部有一个新增入口。

顶部工具区：

- `+` 新增积木

点击 `+` 后：

- 打开下拉菜单或 popover
- 列出可添加的积木类型
- 本轮使用 mock 数据即可

列表项信息保持轻量：

- 积木名称
- 类型
- 简单状态标签

### 中间主显示区职责

点击某个积木时，中间显示：

- 积木标题
- 类型
- 配置说明
- 输入输出说明
- 示例数据
- TODO 能力区块

### 设计理由

数据积木的核心也是“列表选择 + 详情理解/编辑”。

左侧越轻越好，中间再展开复杂信息。不要一上来把一张完整积木卡片塞到左侧。

## 5.4 任务 Tasks

### 左侧栏职责

左侧显示任务列表，顶部提供新增任务入口。

顶部工具区：

- 新增任务按钮

主体内容：

- 任务名称
- 状态
- 更新时间
- 标签摘要（可选）

### 中间主显示区职责

有两种主要状态：

1. 选中已有任务
   - 显示任务详情
   - 任务描述
   - 状态
   - 执行信息
   - 参数 / 时间 / 日志占位

2. 点击新增任务
   - 中间主显示区打开“新建任务”页面
   - 不要在左侧窄栏里做复杂表单

### 设计理由

左侧栏适合列出任务，不适合放完整创建流程。

创建新任务属于编辑行为，应该在中间主区完成。

## 5.5 设置 Settings

### 新位置

设置从普通菜单中移除，改为：

- Activity Bar 底部独立入口
- 与上方工作域图标分组显示
- 点击后弹出 Settings Dialog

### 新行为

设置不再作为主显示区页面出现。

打开设置后：

- 弹出对话框
- 保持用户当前工作区上下文不丢失
- 关闭 dialog 后回到原来的工作流状态

### 设计理由

设置属于全局配置，不属于“当前正在处理的内容”。

把设置做成 dialog，有两个直接收益：

- 不污染主工作流
- 与 VS Code / 桌面工具的使用习惯更一致

## 6. 新的状态模型

这是这次重构的关键。

当前最大的结构性问题，是把“工作域切换”和“详情内容切换”混在一起了。

本次建议拆成三层状态。

### 6.1 第一层：当前工作域

```ts
activeWorkbench: 'workspace' | 'accounts' | 'data-blocks' | 'tasks'
```

它只决定：

- 左侧栏当前显示什么

它不应该直接决定主显示区详情内容。

### 6.2 第二层：左侧选中项

```ts
leftSelection: {
  workspaceFileId?: string
  accountId?: string
  dataBlockId?: string
  taskId?: string
}
```

它决定：

- 当前左侧到底选中了什么内容

### 6.3 第三层：主区模式

```ts
centerMode: 'empty' | 'detail' | 'create-task' | 'file-preview'
```

它决定：

- 中间主显示区当前是空状态、详情态、创建态、预览态中的哪一种

### 6.4 设置弹窗状态

```ts
settingsDialogOpen: boolean
```

设置单独管理，不再挤进主工作流路由。

### 6.5 这套状态模型带来的好处

结果会非常清楚：

- 点 Activity Bar，只切换左侧工作域
- 点左侧内容，才切换中间详情
- 点新增任务，主区进入创建态
- 点设置，只打开 dialog

这才是一个长期可维护的桌面工作台结构。

## 7. UI / UX 重构原则

## 7.1 左侧轻，中间重

这条是本次最重要的视觉纪律。

- 左侧栏负责扫描和选择
- 中间主区负责阅读和编辑

所以：

- 左侧列表项要轻
- 中间详情块可以重

如果左右两边都做成大卡片，界面一定乱。

## 7.2 重新定义表面层级

建议统一为三层：

### Level 0，背景层

用于：

- 整体页面背景
- 大分区底色

### Level 1，面板层

用于：

- Left Sidebar 背景
- Center Panel 背景
- Right Panel 背景
- Dialog 背景

### Level 2，内容块层

用于：

- 主显示区详情卡片
- 重要说明块
- 空状态块
- 表单分组块

注意：

左侧列表项通常不应该升到 Level 2。

## 7.3 左侧列表项统一成 row，不做重卡片

左侧所有列表项统一采用紧凑 row 风格：

- 小图标 / 头像
- 标题
- 补充信息
- 状态点
- hover 背景
- selected 高亮态

不使用：

- 大面积厚重边框
- 过度圆角
- 过大留白
- 内容块式设计

这样左侧才能像一个真正的工作面板，而不是资讯流。

## 7.4 主区详情才允许更完整的内容组织

主区可以使用更明确的结构表达：

- 详情头部
- 信息卡片
- 分组区块
- 属性列表
- 表单
- 空状态
- TODO 区域

这才是卡片真正该出现的地方。

## 7.5 Settings 在视觉上必须与工作域入口分离

Settings 放到底部的意义不只是位置调整，而是语义分离：

- 上方图标是“我正在做什么”
- 底部设置是“这个应用怎么配置”

这两者不能混在同一组里。

## 9. 分阶段实施计划

这次建议不要一次把所有业务能力做满，而是按结构优先、能力递进的方式推进。

### Phase A：信息架构纠正

目标：先把交互模型改对。

任务：

1. Activity Bar 改为只切换工作域
2. Left Sidebar 根据工作域切换内容
3. Center Panel 改为依赖左侧选择结果
4. Settings 挪到 Activity Bar 底部，并改为 dialog
5. 清理当前把内容直接渲染到主区的错误逻辑

验收标准：

- 点击 Activity Bar，不再直接切主区内容页
- 左侧栏内容会跟随工作域切换
- 主区默认显示空状态，只有选中左侧项后才显示详情
- 设置以 dialog 打开，不再占据主区页面

### Phase B：工作区能力落地

目标：先完成最核心的树 + 预览工作流。

任务：

1. 显示当前目录树
2. 支持新建文件
3. 支持新建文件夹
4. 支持刷新目录
5. 支持 txt 文本预览
6. 支持图片预览
7. 其它文件显示 TODO 占位

验收标准：

- 左侧显示当前目录结构
- 可创建文件和文件夹
- 刷新后目录同步更新
- 选中 txt 可在中间查看内容
- 选中图片可在中间查看预览

### Phase C：推特账号工作流重构

目标：把当前左下角散落的数据收编成正式工作区。

任务：

1. 左侧实现推特账号列表
2. 将当前已有实例相关信息合并到账号详情页中展示
3. 保留账号和实例的一对一关系模型
4. 没有字段的数据先空着或 mock 占位

验收标准：

- 左侧只显示推特账号列表
- 点击账号后，中间显示账号详情
- 详情中可看到该账号关联实例信息

### Phase D：数据积木与任务工作流落地

目标：统一“列表 + 详情/创建”模式。

任务：

1. 数据积木左侧列表 + 顶部新增入口
2. 数据积木主区详情页
3. 任务左侧列表 + 顶部新增任务入口
4. 任务主区详情页
5. 新建任务在主区打开

验收标准：

- 数据积木左侧仅显示简洁摘要
- 点击积木后主区显示详情
- 任务左侧显示任务列表
- 点击新增任务后主区进入新建任务页面

### Phase E：视觉系统收口

目标：把现在混乱的视觉风格统一掉。

任务：

1. 统一左侧 row 样式
2. 统一主区详情卡片层级
3. 统一圆角、边框、背景、hover、selected
4. 收紧信息密度，移除多余重卡片表达
5. 统一空状态与占位态样式

验收标准：

- 左侧所有工作域列表风格一致
- 主区详情块层级一致
- 卡片风格不再混乱
- 整体 UI 更接近真正的桌面 IDE 工作台

## 10. 视觉规范建议

### 10.1 Activity Bar

- 宽度保持 48px
- 主工作域图标放顶部
- Settings 单独放底部
- 激活态仍使用左侧竖线 + 图标高亮
- 设置入口不要和上方功能入口挤成一组

### 10.2 Left Sidebar

- 宽度建议维持 240px - 280px 默认区间
- 顶部工具条高度固定
- 内容区统一滚动
- 左侧列表行高保持紧凑，优先扫描效率

### 10.3 Center Panel

- 详情页采用“头部 + 内容分组”的组织方式
- 允许使用更完整的区块感和内容块
- 空状态不要太花，保持桌面工具感

### 10.4 Right Panel

- 继续保留辅助面板角色
- 不参与本次核心信息架构冲突
- 只做轻度视觉收口，不作为本轮主战场

### 10.5 Dialog

Settings Dialog 建议：

- 居中弹出
- 宽度中等
- 层级清晰
- 不做过度花哨动画
- 保持桌面工具气质

## 12. 风险与注意事项

### 12.1 最大风险是继续混用“工作域”与“详情页”概念

如果这次不把状态模型拆开，后续继续追加功能时会再次回到：

- 左侧只是陪衬
- 中间越来越乱
- 组件不断堆 if/else

这次必须一次纠正。

### 12.2 不要把所有内容都做成卡片

这是当前最需要收住的地方。

IDE 类产品的高级感，往往不是因为“卡片多”，而是因为：

- 扫描型信息足够轻
- 阅读型信息足够清楚
- 层次被严格控制

### 12.3 推特账号页先不要追求信息完整

本轮目标是重构 UI 结构，不是一次性做完整业务系统。

所以推特账号列表和详情页：

- 头像、名称先做好
- 其它字段允许为空
- 实例信息先在详情里展示
- 后续再补真实字段和操作能力

这很合理，不会阻碍本次重构。

## 13. 最终建议

本次重构要坚持三条纪律：

1. Activity Bar 只切工作域，不切详情页
2. 左侧栏负责列表/树/入口，中间主区负责详情/编辑
3. 左侧轻量化，主区层次化，不要再乱用重卡片

如果这三条做对了，整个 TweetPilot 壳层会立刻稳定很多。

后面的功能，不管是文件、账号、任务还是数据积木，都能自然挂进来，而不会越改越乱。
