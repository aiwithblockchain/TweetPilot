# 决策卡：Task-AI Full Access 模式（版本 A）

## 1. 决策结论

本项目允许为 Task-AI 提供一个用户可见的高权限模式：

- **完全授权 AI**
- 或英文命名：**Full Access**

但本模式采用的是 **版本 A**，而不是“彻底无边界执行”的版本 B。

也就是说：

> **Full Access 是产品语义上的“完全授权”，不是工程实现上的 `BypassPermissions`。**

系统内部仍然保留：

- 统一权限框架
- 路径分类
- 命令风险分类
- 审计能力
- 极少数系统级底线限制

因此，Full Access 的真实含义是：

> **给当前 task 或特定高级 session 一个极宽松的预授权策略，让绝大多数操作自动执行，但不把系统退化成无控制裸执行。**

---

## 2. 为什么选择版本 A，而不是版本 B

### 2.1 版本 A
产品上允许用户看到：
- 完全授权 AI
- Full Access

但工程上仍然：
- 走统一权限框架
- 记录审计
- 保留极少数底线禁止项

### 2.2 版本 B
如果采用版本 B，则意味着：
- 真正任何目录
- 真正任何命令
- 完全不拦截
- 完全不保留边界

这本质上会退化成另一种形式的：
- `BypassPermissions`

### 2.3 为什么不选版本 B
原因如下：

1. **Task-AI 是后台无人值守执行**
   - 比前台 IDE 协作更危险
   - 错误可能被定时任务重复放大

2. **用户理解风险，不等于系统可以取消审计和底线**
   - 用户可以承担“高权限”风险
   - 但工程上仍应保留最小可观测性和可回收性

3. **保留统一权限框架有利于长期维护**
   - 普通 chat / task-ai / future automation 可以共用一套模型
   - 不会因为 Full Access 出现第二套完全脱轨的执行路径

因此最终结论是：

> **支持 Full Access，但 Full Access 不等于重新启用 `BypassPermissions`。**

---

## 3. Full Access 的产品语义

当用户为某个 Task-AI 开启 **完全授权 AI / Full Access** 时，表示：

- AI 可以自动执行绝大多数本地工具操作
- 不再因为大多数中高风险操作而逐次弹窗
- 更适合后台自动化任务、无人值守定时任务、高级用户场景

对用户来说，它应当足够接近：

- “我愿意让这个 task 基本自己做事”
- “不要频繁打断我”
- “我知道它会操作本地环境，并接受相关风险”

也就是说，**产品语义上可以称之为“完全授权”**。

---

## 4. Full Access 的工程语义

虽然对用户可称为“完全授权”，但系统内部不能把它实现为简单跳过权限系统。

Full Access 在工程上的真实语义应当是：

- `session_auth_policy = preauthorized` 的一种极宽松模式
- 或等价的“broad scope preauthorization”

仍然保留以下机制：

### 4.1 仍然保留路径分类
系统仍应知道一次文件操作是：
- workspace
- external
- system_sensitive
- unknown

即使 Full Access 打开，也不能因为放行就丢失路径分类信息。

### 4.2 仍然保留命令风险分类
系统仍应知道一次命令属于：
- readonly
- write
- destructive
- network
- unknown

即使最终允许执行，也必须先分类，再决定是否命中底线限制。

### 4.3 仍然保留审计能力
至少要记录：
- task id
- run id
- session id
- 使用的授权模式
- 实际执行了哪些工具请求
- 哪些请求被允许
- 哪些请求因底线规则被拦截

### 4.4 仍然保留极少数系统底线
Full Access 不等于“任何事都能自动执行”。

系统应当仍然保留少量不可自动放行的底线动作。

---

## 5. 建议保留的系统底线

以下动作即使在 Full Access 下，也不建议自动放行：

- `sudo`
- `curl ... | sh`
- 对系统敏感目录的大范围递归删除
- 对系统敏感目录的大范围递归改写
- 明显不可解释的大范围 destructive command
- 其他你们认定必须保留硬边界的动作

处理方式可以是：

- 直接拒绝
- 或标记为需要更高等级确认

但无论如何：

> **不能静默自动执行这些底线动作。**

---

## 6. Full Access 的作用域建议

Full Access 不应设计为：

- 全局永久生效
- 一次开启后对所有 task 生效
- 一次开启后对所有工程生效

建议设计为：

### 首选：任务级（task-level）
- 只对当前这个 task 生效
- 不影响其他 task

### 可选：特定高级 session 级
- 只对当前特定 session 生效
- session 结束后失效

### 不建议：全局级 Full Access
- 风险过大
- 容易误伤所有使用场景
- 不利于理解与撤销

因此建议明确原则：

> **Full Access 是 task 级或特定 session 级能力，不是全局级能力。**

---

## 7. 与普通授权模式的关系

Full Access 不是替代普通权限系统，而是建立在普通权限系统之上的一个高权限档位。

普通模式下：
- 系统做风险判断
- 命中 ask 的请求需要授权或被拒绝

Full Access 下：
- 系统仍然做风险判断
- 绝大多数请求自动通过
- 但命中系统底线的少数动作仍被拒绝或更高门槛处理

所以它与普通模式的关系应理解为：

- 普通模式：保守授权
- Full Access：宽松预授权
- 二者共用同一套权限与审计框架

而不是：

- 普通模式走权限系统
- Full Access 走完全另一条无控制旁路

---

## 8. Task-AI 推荐档位

Task-AI 最终建议提供如下几档：

### 模式 1：系统自定义
含义：
- 按系统默认授权策略运行
- 那些被全局允许的操作自动执行
- 那些被系统判定为需要授权的动作，在无人值守 task 场景下阻止本次执行
- 那些明确无需授权的低风险操作（如 `ls`）可以自动执行

适合：
- 保守自动化
- 测试期
- 低风险任务

### 模式 2：工作目录完全授权
含义：
- 当前工作区内的读写删改自动执行
- 工作区外路径不自动放开
- 仍然保留系统底线

适合：
- 项目内脚本生成、文件整理、代码操作等任务

### 模式 3：完全授权 AI（Full Access）
含义：
- 绝大多数目录和命令操作自动执行
- 不再局限于工作区
- 但仍保留系统底线与审计

适合：
- 高级自动化
- 高权限后台任务
- 用户明确接受高风险的场景

---

## 9. Full Access 不是 `BypassPermissions`

这一点必须在实现和文档中反复强调。

### 错误理解
Full Access = 关闭全部权限判断 = 恢复 `BypassPermissions`

### 正确理解
Full Access = 在统一权限系统内，给予极宽松的自动放行策略，但仍保留：
- 风险分类
- 审计
- 最小系统底线

因此，开发时必须避免以下实现：

- 看到 Full Access 就直接跳过权限框架
- 不再记录路径/命令分类
- 不再记录审计
- 不再区分 task 范围
- 一处开启，所有地方都吃到高权限

---

## 10. 当前阶段的实施建议

### 10.1 现在不要急着把 Full Access 立刻全面实装到 Task-AI
如果 Task-AI 刚开发完、正处于测试阶段，建议：

1. 先继续完成 task-ai 的功能测试
2. 先把 Full Access / 工作区授权 / 系统自定义 这套产品模型定下来
3. 普通 chat 先完成本地工具授权 Phase A 底座
4. 等 task-ai 测试稳定后，再把 task-ai 接入正式授权模式

### 10.2 但如果 task-ai 当前已经能对真实环境执行高风险本地操作
则不能无条件拖后：

- 至少要加临时闸门
- 或限制发布范围
- 或默认切到更保守模式

因为这已经是现实安全风险，而不只是未来优化问题。

---

## 11. 一句话定义

如果需要一句最准确的产品/工程统一定义，可使用：

> **Task-AI 的 Full Access 是一种用户显式开启的高权限模式：它允许绝大多数本地操作自动执行，但内部仍然走统一权限框架，并保留极少数系统底线与完整审计。**
