# Python 任务执行结果一致性与执行详情方案

## 1. 背景

当前任务详情页中，Python 脚本任务失败后，用户有时能看到错误信息，有时只能看到失败状态与退出码，体验不一致。

最初观察时，问题看起来像是：

- 前端没有展示 Python 失败原因
- 或前端没有及时刷新到最新错误结果

但进一步核对代码后，发现真实问题比 UI 展示更底层：

1. **前端其实已经具备错误信息展示能力**
2. **前端其实已经监听任务执行完成事件并会自动刷新详情**
3. **真正不一致的是后端在不同执行路径下，对失败结果的处理不统一**

这意味着，本问题不能仅靠“加一个执行详情按钮”解决，必须先补齐执行结果一致性。

---

## 2. 代码现状确认

### 2.1 前端已有错误信息展示

任务详情页组件 `src/components/TaskDetailContentPane.tsx` 已支持：

- 展示最后一次执行状态
- 展示退出码
- 展示输出
- 当 `task.lastExecution.error` 存在时，展示“错误信息”区块

因此，“页面完全没有错误展示能力”这一判断不成立。

---

### 2.2 前端已有事件监听与自动刷新

同一组件中已经监听以下事件：

- `task-executed`
- `task-updated`
- `task-paused`
- `task-resumed`
- `task-deleted`

当收到当前任务对应的事件后，会调用 `loadDetail()` 重新拉取任务详情。

因此，“前端完全没有刷新机制”这一判断也不成立。

---

### 2.3 真正问题：后端失败路径处理不一致

核对后端发现：

#### 手动执行路径：`src-tauri/src/task_commands.rs -> execute_task`

- 成功分支会：
  - 保存 execution record
  - 更新任务状态
  - 发布 `task-executed` 事件

- 失败分支当前只会：
  - 更新任务状态为 `failed`
  - 直接返回错误

也就是说，当前手动执行在失败时，**并没有保证统一完成以下动作**：

- 保存失败执行记录
- 广播执行完成事件

---

#### Python 脚本执行器：`src-tauri/src/task_executor.rs`

这里实际上已经能拿到：

- `stdout`
- `stderr`
- `exit_code`
- `status = success | failure`

换句话说，Python 脚本即使失败，底层执行器也已经拿到了完整错误结果。

问题不在“拿不到错误”，而在“上层是否把这次失败当成一次完整执行记录来处理”。

---

#### 定时执行路径：`src-tauri/src/unified_timer/executors/python_script.rs`

该路径会在失败时：

- 组装失败的 execution result
- 保存 execution record
- 写入 stderr / error_message

因此，**定时执行失败路径比手动执行失败路径更完整**。

这正是当前体验不一致的根源之一。

---

## 3. 问题定义

当前 Python 任务存在两个层面的不一致：

### 3.1 数据一致性不一致

不同执行入口下，失败结果的处理不统一：

- 手动执行成功：通常可见
- 手动执行失败：可能没有完整 execution record / 没有事件通知
- 定时执行成功：可见
- 定时执行失败：可见

这导致前端并不是始终拿不到数据，而是**有些路径有数据，有些路径没有完整闭环**。

---

### 3.2 界面交互不完整

即使前端拿到了错误数据，目前展示仍然偏“内嵌错误块”，缺少统一的详情入口。

因此用户体验仍有两个问题：

1. 某些情况下页面不能及时更新到失败原因
2. 即使更新到了，也缺少一个清晰统一的“执行详情”入口

---

## 4. 方案目标

本次方案目标拆为两个层次。

### 4.1 第一目标：执行结果一致性

无论任务是：

- 手动执行 / 定时执行
- 成功 / 失败

都应统一满足：

1. 生成 execution record
2. 持久化 execution record
3. 更新任务状态
4. 发布执行完成事件
5. 前端收到事件后刷新任务详情

这是本次方案的首要目标。

---

### 4.2 第二目标：执行详情展示

在执行结果一致性的前提下，为 Python 任务增加统一入口：

- 按钮名：**执行详情**

让用户可以稳定查看：

- 执行状态
- 时间
- 耗时
- 退出码
- 错误摘要
- stderr / traceback
- stdout
- 执行上下文

---

## 5. 非目标

本次不处理：

- 实时流式日志系统
- 全局日志搜索
- 日志下载中心
- 所有任务类型的统一日志平台
- 复杂错误分类引擎
- 跨设备远程执行日志同步

---

## 6. 设计原则

1. **先修数据闭环，再做交互增强**
2. **执行失败也必须被视为一次有效执行结果**
3. **前端刷新依赖统一事件，不依赖偶然重进页面**
4. **优先最小改动，不重构整套任务系统**
5. **展示摘要 + 原始错误并存**

---

## 7. 第一部分：执行结果一致性方案

## 7.1 核心要求

对于 Python 任务，无论执行结果是否成功，都必须形成一条完整 execution record。

建议统一约束：

- **脚本运行完成但 exit code 非 0** → 视为一次失败执行结果，不是“系统异常”
- **执行器启动失败 / 脚本文件不存在 / 超时 / 运行器异常** → 也应尽可能落为失败 execution record

换句话说，**只要用户发起了一次执行，就应该能在“最后执行”里看到这次执行的结果**。

---

## 7.2 手动执行链路应补齐的动作

当前手动执行路径在失败分支缺少完整闭环。

应调整为：

### 手动执行成功
保持现有行为：

- 保存 execution record
- 更新任务状态
- 发布 `task-executed`

### 手动执行失败
新增统一处理：

- 生成失败 execution record
- 保存 execution record
- 更新任务状态
- 发布 `task-executed`
- 将错误返回前端（如果仍需 toast 或按钮态反馈）

重点是：

> “返回错误给前端” 不应替代 “保存失败执行记录并广播执行完成事件”。

---

## 7.3 推荐处理策略

建议将 `execute_task` 视为一个“无论成功失败都要落执行记录”的编排层。

### 推荐行为

#### 情况 A：执行器返回 `Ok(ExecutionResult)`，其中状态为 `success`
- 保存记录
- 发事件
- 返回结果

#### 情况 B：执行器返回 `Ok(ExecutionResult)`，其中状态为 `failure`
- 保存记录
- 发事件
- 返回结果（而不是当作系统错误直接中断）

#### 情况 C：执行器返回 `Err(String)`
- 将该错误包装为一条失败 execution record
- 尽可能补齐：
  - startTime / endTime
  - status = failure
  - exitCode = -1 或约定值
  - stderr / errorMessage = err
- 保存记录
- 发事件
- 再决定是否把错误继续抛给前端

这样能最大化保证“用户发起一次执行，就一定有可见结果”。

---

## 7.4 事件机制要求

事件仍然使用现有 `task-executed` 机制即可，不必先新增全新消息类型。

关键要求是：

- 成功时发
- 失败时也发

前端已有监听能力，因此后端只要补齐广播，前端详情页就会自动 `loadDetail()`。

这意味着：

- 如果只是“失败后没有及时更新”这一类问题
- 现有事件机制已经足够承载修复

---

## 7.5 状态一致性建议

建议统一任务执行后的状态约定，避免任务长期停留在 `failed` 影响后续判断。

可以评审时确认以下二选一策略：

### 策略 A：执行结束后恢复 `idle`，失败信息放在 lastExecution
优点：
- 任务对象状态表示“当前是否在运行”
- 失败结果由 execution history 表达
- 与很多 job/task 系统更一致

### 策略 B：执行失败后任务主状态置为 `failed`
优点：
- 列表态更直观

风险：
- 容易把“最后一次执行失败”与“任务当前不可执行”混在一起

**建议优先考虑策略 A**：
- `running` 只表示运行中
- 执行结束后回到 `idle`
- 最近一次执行状态单独显示 success/failure

这一点需要与你们现有产品状态语义对齐后决定。

---

## 8. 第二部分：执行详情交互方案

在执行结果一致性补齐后，再做执行详情入口。

### 8.1 按钮命名

统一命名：

- **执行详情**

该命名既适用于成功，也适用于失败，比“查看日志”或“查看过程”更中性、更产品化。

---

### 8.2 按钮显示规则

建议：

- 有执行记录时显示并可点击
- 无执行记录时禁用，提示“暂无执行记录”

按钮位置建议仍放在：

- 任务详情页“任务操作”区域

---

### 8.3 展示形式

推荐使用 Drawer，其次是 Modal。

原因：

- stderr / traceback 往往较长
- 更适合纵向浏览
- 不会过度打断当前详情页上下文

---

## 9. 执行详情展示内容

建议按以下结构展示。

### 9.1 执行摘要

- 状态
- 开始时间
- 结束时间
- 耗时
- 退出码

---

### 9.2 失败原因摘要（失败时）

优先展示可读摘要，例如：

> 存在多个 tweetClaw 实例在线，脚本请求状态接口时未指定 `instanceId`，导致执行失败。

摘要来源优先级：

1. `errorMessage`
2. `stderr` 最后一条核心异常
3. 默认文案：`执行失败，请查看错误详情`

---

### 9.3 原始错误详情

展示 stderr / traceback 原文：

- 可滚动
- 可复制
- 保留换行格式

---

### 9.4 标准输出

stdout 不为空时显示。

---

### 9.5 执行上下文

建议展示：

- 脚本路径
- 工作目录
- 执行命令
- Python 解释器（若可得）

---

## 10. 数据需求

为了支撑一致性与执行详情，执行记录建议至少具备以下字段：

```ts
type TaskExecutionRecord = {
  status: "success" | "failure"
  startedAt: string
  finishedAt?: string | null
  durationMs?: number | null
  exitCode?: number | null
  stdout?: string | null
  stderr?: string | null
  finalOutput?: string | null
  errorMessage?: string | null
  scriptPath?: string | null
  workingDirectory?: string | null
  command?: string | null
  metadata?: Record<string, any> | null
}
```

当前代码中 `stdout` / `stderr` / `errorMessage` 已基本具备。需要重点确认或补齐的是：

- scriptPath
- workingDirectory
- command
- 更稳定的 error summary 来源

如果本次不想扩字段，可以先只用已有字段完成第一阶段。

---

## 11. 最小落地范围

### Phase 1：先修一致性

1. 手动执行失败也保存 execution record
2. 手动执行失败也发布 `task-executed`
3. 前端继续复用现有监听与 `loadDetail()` 刷新机制
4. 确认失败后页面无需手动刷新即可看到最新错误

这是必须先完成的部分。

---

### Phase 2：补“执行详情”入口

1. 在任务详情页增加“执行详情”按钮
2. 打开 Drawer / Modal
3. 展示执行摘要、错误摘要、stderr、stdout、上下文

---

### Phase 3：可选增强

仅在有必要时再评估：

- 错误摘要提炼优化
- 多次执行记录切换查看
- 一键复制完整执行上下文
- 列表页显示最近一次失败摘要

---

## 12. 验收标准

### 12.1 一致性验收

对于同一个 Python 任务：

1. 点击“立即测试”后，若脚本失败：
   - 必须生成失败 execution record
   - 任务详情页必须自动刷新
   - 页面必须看到最近一次失败信息

2. 定时执行失败时：
   - 仍能看到失败 execution record
   - 与手动执行表现一致

3. 无论成功或失败：
   - `task-executed` 事件都应触发详情页刷新

---

### 12.2 执行详情验收

当已有执行记录时：

1. 页面显示“执行详情”按钮
2. 点击后能查看：
   - 状态
   - 时间
   - 耗时
   - 退出码
3. 若失败：
   - 能看到失败摘要
   - 能看到 stderr / traceback
4. 用户无需查看 dev.log 即可理解问题方向

---

## 13. 风险与注意事项

### 13.1 失败不应只作为异常抛出

如果继续把“脚本失败”仅当作 Rust 命令错误抛出，而不是执行结果，就会继续造成 UI 与数据层断裂。

---

### 13.2 长错误文本的 UI 承载

stderr 可能很长，必须支持：

- 最大高度限制
- 内部滚动
- 复制
- 合理换行

---

### 13.3 状态语义要统一

需要在实现前明确：

- 任务对象的 `status` 是表示“当前运行态”
- 还是表示“最后一次执行结果”

建议不要混用。

---

## 14. 推荐结论

本次方案不再把问题定义为“单纯缺少失败信息展示”，而是定义为：

> **Python 任务在不同执行路径下，失败结果的持久化与事件广播不一致，进而导致界面反馈不稳定。**

因此推荐按以下顺序实施：

1. **先补齐执行结果一致性**
   - 手动执行失败也要落库
   - 手动执行失败也要广播 `task-executed`
   - 前端复用现有事件监听自动刷新

2. **再补执行详情交互**
   - 新增“执行详情”按钮
   - 展示完整错误与执行上下文

只有先修第一层，第二层的“执行详情”才有稳定可靠的数据基础。
